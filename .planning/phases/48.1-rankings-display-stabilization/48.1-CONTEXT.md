# Phase 48.1: Rankings Display Stabilization - Context

## Phase Boundary

This phase replaces the unstable default rankings display path on `/tech/rankings/laptops`.

It does not rebuild the whole rankings page. It keeps:

- the existing server route and data queries
- the existing local filter state and filter controls
- the existing tech shell and nav
- the existing GlitchTech visual language
- the existing review detail links and buy button behavior where feasible

It replaces:

- the default TanStack/full table runtime path in `src/components/tech/leaderboard-table.tsx`
- the duplicated hidden desktop/mobile result rendering where it contributes to route-transition instability
- any default reliance on `?simple=1` for safe navigation

<decisions>
## Implementation Decisions

### Diagnosis And Scope

- **D-01:** Diagnosis is locked. Real macOS testing on 2026-05-03 proved `?simple=1&diag=1` works after filter -> reset -> nav, `?simple=1` also works after filter -> reset -> nav, and the default page freezes after the same interaction. Therefore the filter UI and tech nav shell are not the primary suspects; the unstable surface is the full rankings display path.
- **D-02:** Do not full-scrap the page. Preserve the page tree, route, data fetch, filter UI, and visual direction. Replace the display layer.

### Display Architecture

- **D-03:** The default rankings route must not import or execute `@tanstack/react-table`, `useReactTable`, or `flexRender`. The stable custom display should be normal React rendering over `filteredRows`.
- **D-04:** The passing `?simple=1` proof is the architectural seed, not the final UI. Build a polished custom rankings display from that lightweight path.
- **D-05:** The custom display must expose rank, product name and review link, BPR tier and BPR score when present, GlitchMark score and partial marker when present, key benchmark columns or a visible not-tested placeholder, release year, and price.

### Filter And Responsive Behavior

- **D-06:** Existing `LeaderboardFilters` controls remain in scope and should not be rewritten unless a plan proves the display replacement still freezes. Current evidence says filters are safe.
- **D-07:** Interactive sort headers were previously disabled for stability. This phase should not re-enable every-column sorting unless it can be implemented in the custom display without reintroducing route-transition instability. Default ordering by GlitchMark/rank is acceptable for the stabilization fix.
- **D-08:** Desktop and mobile must both work, but the implementation should avoid mapping the full result set into separate hidden display trees on desktop and mobile at the same time. CSS-hidden paint is not the same as avoiding React reconciliation.

### Verification

- **D-09:** Headless Playwright is not sufficient. The phase cannot close until the user verifies on real macOS Safari and Firefox: load production `/tech/rankings/laptops`, click any filter, optionally reset filters, click a tech sidebar nav tile, and confirm the page and other `glitchtech.io` tabs do not freeze.

</decisions>
