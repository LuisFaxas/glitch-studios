# Phase 48.1: Rankings Display Stabilization - Research

## Research Question

What do we need to know to plan a safe replacement for the default rankings display layer?

## Findings

### F-01: `?simple=1` Is The Decisive Fork

The live diagnostic path `?simple=1` removes TanStack and the full table rendering path while keeping the same filter state and filter sidebar. The user tested:

- `?simple=1&diag=1`: passed
- `?simple=1`: passed
- default page: froze immediately when returning to regular page and running the same flow

This isolates the practical fix to the display layer.

### F-02: Default Path Still Has Multiple Heavy Surfaces

`src/components/tech/leaderboard-table.tsx` currently includes:

- `@tanstack/react-table`
- `useReactTable`
- `flexRender`
- a full table DOM path
- a hidden mobile result path that still maps rows into cards or another table path
- style containment on result wrappers
- `useSearchParams` dispatching between default and simple mode

Any one of these may contribute, but the phase does not need to identify which one. The stable strategy is to remove the full path from default runtime.

### F-03: Filter UI Is Likely Safe Enough

The passing `?simple=1` run preserved:

- desktop filter bar
- `CustomDropdown`
- filter state updates
- reset flow
- mobile filter sheet component existing in the tree
- tech nav route transition

So replacing filters would add risk without matching the evidence.

### F-04: Tests Are Stale Around Old URL/Sort Assumptions

`tests/29-leaderboard.spec.ts` still describes URL-backed sorting/filtering and old route expectations. The current implementation intentionally uses local filter state and has disabled sort interactions after previous stability work.

Phase 48.1 should update tests to verify the current contract, not resurrect old unstable requirements.

### F-05: The Custom Display Should Be Purpose-Built

The desired display is not a spreadsheet replacement. It should be a lightweight ranking surface optimized for:

- scanability
- score comparison
- safe filter mutation
- safe route unmount
- responsive behavior

Use regular React map rendering with stable keys and no table library.

## Recommended Architecture

Build a custom `LeaderboardTable` implementation that:

1. Keeps `applyFilters` and `deriveBounds`.
2. Removes TanStack imports and `useReactTable`.
3. Removes the `?simple=1` dispatcher.
4. Renders `filteredRows` directly into a single custom display component.
5. Uses CSS grid/flex rows instead of table row models.
6. Avoids mapping a second hidden full result tree.
7. Keeps `LeaderboardFilterSheet` for mobile unless real-device verification says otherwise.
8. Keeps `?diag=1` instrumentation because it is gated and useful for future production diagnostics.

## Verification Strategy

Automated verification can prove:

- no TanStack runtime remains in the default file
- filters still toggle and reset
- rows update after filters
- product links render
- mobile filter sheet opens
- no console/page errors in Playwright

Manual verification must prove:

- production real macOS Safari filter -> reset -> nav does not freeze
- production real macOS Firefox filter -> reset -> nav does not freeze

## Open Risks

- A custom display may lose some old full-table affordances. This is acceptable for stabilization, but the plan should make any deferrals explicit.
- Some existing tests encode old `nuqs` URL assumptions and need updating rather than blindly preserving.
- The mobile sheet still uses shared `Sheet` primitives. Because `?simple=1` passed with it mounted, it is lower priority.

## Validation Architecture

Validation is a mixed automated/human gate:

- Static source grep for removal of TanStack/default simple dispatcher.
- Playwright filter/display regression on desktop and mobile.
- Build/lint command pass.
- Human macOS Safari and Firefox production proof before phase close.
