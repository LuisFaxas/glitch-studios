    ```
    export function formatPriceChip(priceLabel: string): string {
      const trimmed = priceLabel.trim()
      if (trimmed.toLowerCase().startsWith("from ")) {
        return `FROM ${trimmed.slice(5).toUpperCase()}`
      }
      return trimmed.toUpperCase()
    }
    ```

    File 2: `src/lib/services/format-duration-chip.ts`
    ```
    export function formatDurationChip(minutes: number | null): string | null {
      if (minutes == null) return null
      if (minutes >= 60) {
        const hours = minutes / 60
        if (hours === 1) return "1 HR"
        if (Number.isInteger(hours)) return `${hours} HRS`
        return `${hours.toFixed(1).replace(/\.0$/, "")} HRS`
      }
      return `${minutes} MIN`
    }
    ```

    Why these implementations (per RESEARCH §Example 2):
    - `formatPriceChip` mirrors UI-SPEC §6 chip content rule: pass-through `/hr` and `/track`, special-case `from $` prefix, otherwise uppercase. NO regex needed — `startsWith("from ")` after `toLowerCase()` is the contract.
    - `formatDurationChip` returns `null` (not empty string, not `—`) when the input is null so the card can conditionally render the chip (UI-SPEC §6: "If null → omit the chip entirely").
    - `toFixed(1).replace(/\.0$/, "")` strips trailing `.0` so `120 / 60 = 2` (already integer-checked) and `150 / 60 = 2.5` (kept) both render correctly; the regex is a safety net for floating-point edge cases.

    Do NOT add JSDoc unless TypeScript strict mode requires it (it does not for these signatures). Do NOT add tests in this plan — Playwright in Plan 04 will exercise outputs end-to-end; vitest is overkill for 5-line helpers (RESEARCH §Example 2 explicitly defers tests).
