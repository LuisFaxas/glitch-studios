## 1. Goal

Make `/services` a cohesive design that **speaks differently per screen** instead of stretching the mobile accordion:

- **Mobile (< lg):** the existing vertical accordion, but the **open card is enriched** to match desktop — faded pixel-icon watermark + spec strip + full color bloom. (Closed rows unchanged.)
- **Desktop (≥ lg):** a brand-new **master–detail** layout — a compact mono list **rail** (all 7) + a rich **detail canvas** for the selected service.
- **Shared:** one rich "detail" interior (watermark, eyebrow, spec strip, what-you-get, CTA) + the same DNA (mono→accent bloom, vendored pixel icons, brutalist). Each size lays it out natively.

Fixes the two complaints: desktop is no longer a stretched stack; cards are no longer bland.

---

## 2. Architecture (the key decision)

Extract the shared interior into one component, render it in two host layouts driven by one shared selection state. **DRY = the cohesion, in code.**

| Component | Path | Role |
|---|---|---|
| `ServiceDetail` | `src/components/services/service-detail.tsx` (NEW, client) | The rich interior: eyebrow + (optional big icon+title) + lead + **spec strip** + "WHAT YOU GET" + CTA + **faded icon watermark**. Prop `variant: "inline" | "canvas"` (inline omits the big title — the accordion head already shows the name; canvas shows big icon+title). |
| `ServiceCard` | `service-card.tsx` (edit) | Mobile accordion row. Closed head unchanged; **open body now renders `<ServiceDetail variant="inline">`** instead of the ad-hoc interior. |
| `ServiceRail` | inline in `service-grid.tsx` or small component | Desktop rail rows (mono; selected = accent edge + accent icon/name/price). Custom Request = link row. |
| `ServiceGrid` | `service-grid.tsx` (edit) | Holds `selectedSlug`. Renders **both** layouts; CSS controls which shows. |

### Responsive switch — CSS, not JS media query
Render **both** the mobile accordion and the desktop master–detail in the DOM; toggle with Tailwind `lg:hidden` / `hidden lg:grid`. **Rationale:** avoids a `useMediaQuery` hydration mismatch (server can't know viewport) and the layout-shift flash. Cost is rendering 7 small items twice — negligible, and both share `<ServiceDetail>` so no logic duplication. (This is why the deleted overlay's `useMediaQuery` is NOT reintroduced.)

### Selection semantics
- One `selectedSlug` state shared by both layouts.
- **Mobile:** single-open accordion; `null` = all closed (default). Toggle on tap.
- **Desktop:** the canvas always shows a service; if `selectedSlug` is `null`, default to `services[0]`. Clicking a rail row sets it.
- Deep-link (`#slug`) sets `selectedSlug` on load (keep the load-bearing `setTimeout(0)` defer). Works for both layouts.
- Custom Request is NOT selectable in either layout — it's a link row to `/contact?service=custom`.

## 3. Spec strip — data source (derive, don't invent)

`deriveSpecs(service)` → up to 3 `{k,v}` cells from existing fields only:
- Cell 1 — **bookable:** `RATE` = price chip; **non-bookable:** `FROM` = price chip.
- Cell 2 — if `durationMinutes`: `SESSION` = formatted duration; else `SCOPE` = `BY BRIEF`.
- Cell 3 — if bookable + deposit present (`depositValue`/`depositType` from `serviceBookingConfig`): `DEPOSIT` = `{n}%` or `${n}`; else omit (2-cell strip).

No per-service hardcoding. Recording/Mixing → 3 cells (rate/session/deposit); Video/SFX/Photography/Graphic → 2 cells (from/scope). Custom Request has no detail (link row) so no strip.

**Open question for Codex:** is deriving DEPOSIT from `serviceBookingConfig.depositValue/Type` correct, and is it already passed through `enrichedServices` in `page.tsx`? (It is — `depositType`/`depositValue` are on the enriched object.) Confirm formatting (percentage vs flat).

## 10. Codex review (archived) — 2026-05-27, APPROVE-WITH-CHANGES

**Confirmed:** render-both/CSS-switch is the right SSR-safe pattern; `serviceBookingConfig.depositType/depositValue` semantics match (per `src/lib/booking/deposit.ts`: percentage = value like 50 → /100; flat = dollars); `page.tsx` already exposes `depositType` + numeric `depositValue`; one shared `selectedSlug` + desktop default `services[0]` is good.

**Required changes (govern the build):**
1. **No duplicate `id`s.** Layout-scoped: mobile `id="service-mobile-body-${slug}"`, desktop `id="service-desktop-panel-${slug}"`.
2. **Layout-scoped wrappers + test selectors.** Wrap each layout in `[data-services-layout="mobile"]` / `[data-services-layout="desktop"]`. All tests scope through these (the bare `[data-service-slug]` / `[data-testid="service-card"]` are now ambiguous across the two copies). Keep `data-service-slug` on rows but always query within a layout wrapper.
3. **Hide only via `display:none`** (`lg:hidden` / `hidden lg:grid`) so hidden controls leave the focus order + a11y tree. Never opacity/visibility/offscreen.
4. **Desktop rail a11y:** rail rows are `<button>` with `aria-current="true"` on the selected one; the canvas is a labelled region (`aria-labelledby` → the canvas H2). Custom Request stays a normal `<Link>`.
5. **Deposit formatting:** `percentage` → `${depositValue}%`; `flat` → currency `$${depositValue}` (not a raw number).

**Resolutions to §8 open questions:** (1) render-both via CSS — adopted; (2) deposit semantics confirmed, format per #5; (3) desktop default `services[0]` — adopted; (4) `ServiceDetail` in its own file — adopted, and `deriveSpecs` in its own pure helper (`src/lib/services/derive-specs.ts`) so it's unit-testable.

**Nice-to-haves adopted:** layout-scoped tests (mobile-hidden-vs-desktop-visible at lg, rail-select swaps canvas, deep-link selects desktop canvas, duplicate-selector safety).
