# Cross-AI Plan Review — Phase 1.4

## Codex Review

Overall, the three plans are directionally solid and match real defects in the current codebase, but Plans 01 and 03 need tighter scoping around shared primitives, and Plan 02 needs stronger failure-state handling to fully meet the phase goal.

### Plan 01 — Sidebar Overhaul

**Summary:** Strong plan aimed at real defects in navigation and hover behavior. The main risk is changing the wrong abstraction or overloading the shared tile primitive.

**Strengths:**
- Targets real root issues: the current glitch hover animates an empty overlay in tile.tsx
- Compact mode plus scroll affordance is the right response to the 900px viewport constraint
- Desktop nav, mobile overlay, and bottom tab belong in the same wave because they share the same content model

**Concerns:**
- `HIGH`: There are two sidebar implementations. The live one is tile-nav.tsx, while side-nav.tsx appears unused. If the plan is not explicit, work could land in dead code.
- `HIGH`: The logo problem is structural, not just color. logo-tile.tsx has no intrinsic height because the image is absolutely positioned, so `invert(1)` alone will not make it visible.
- `HIGH`: Horizontal icon+label layout and compact mode should not be implemented by changing the shared defaults in tile.tsx, or widgets/auth tiles will regress.
- `MEDIUM`: Duplicating content for glitch is sound for simple nav tiles, but risky as a universal pattern for arbitrary children, media, or interactive content.
- `MEDIUM`: The mobile overlay is visually modal but still lacks focus trapping/restored focus.

**Suggestions:**
- Scope the plan explicitly to TileNav, LogoTile, MobileNavOverlay, and BottomTabBar
- Add a nav-specific tile variant instead of mutating the base Tile
- Fix logo sizing first with explicit aspect ratio or min-height, then handle inversion/glow
- Trigger glitch on focus-visible as well as hover, and keep reduced-motion behavior intact
- Use actual overflow detection for the scroll hint rather than assuming the layout will fit

**Risk Assessment:** MEDIUM-HIGH — direction is right but shared-component regressions are likely if scoping is loose.

### Plan 02 — Home Page & Routes

**Summary:** Necessary plan and close to the phase goals, but it needs real data-failure handling, not just empty-state polish.

**Strengths:**
- Fixes the most user-visible breakage: unreadable home hero and missing /beats route
- Grouping home-page fixes with route fixes is the right dependency choice
- Scanline background plus inverted CTA tiles fits the intended design language better than the current black void

**Concerns:**
- `HIGH`: The home page currently hard-fails on DB errors via raw `Promise.all` in page.tsx. "Fallbacks" need to include query failure handling, not only empty arrays.
- `HIGH`: "About tile fixed" is underspecified. Current nav says About but links to /artists. Decide whether that is a rename or a real new route.
- `MEDIUM`: services-overview.tsx has no empty state, so the current plan should explicitly cover that case.
- `MEDIUM`: Removing the hero scrim entirely may work now, but it weakens future readability once a real video background is added.

**Suggestions:**
- Add /beats with metadata and sitemap updates
- Use guarded queries or Promise.allSettled so the hero can still render during DB issues
- Define exact rules per home section: fallback, hide, or placeholder
- Treat service work as polish-only unless there is a specific functional defect

**Risk Assessment:** MEDIUM — scope is reasonable but incomplete unless route semantics and DB failure states are tightened.

### Plan 03 — Design Consistency

**Summary:** Good cleanup wave, but this is the plan most likely to sprawl if it becomes a broad search-and-replace instead of a token/primitives pass.

**Strengths:**
- Correctly waits until nav and home defects are fixed
- Footer, contact form, blog cards, 404, and spacing are the right surfaces to unify next
- Replacing obviously rounded/default-looking surfaces will materially improve cohesion

**Concerns:**
- `HIGH`: Blanket replacement of all gray-* classes with hardcoded hex values is high-churn and low-discipline; much of that palette already exists in globals.css
- `HIGH`: Replacing all shadcn/Base UI wrappers with plain HTML is scope creep and risks losing accessibility/interaction behavior
- `MEDIUM`: #555 on #111 is too low-contrast for normal text. Current metadata in post-card.tsx is already close to that problem.
- `MEDIUM`: If visible glitch is part of consistency, blog cards alone are not enough; the same invisible-overlay pattern exists in artist-card.tsx and video-card.tsx
- `LOW`: Services UI is already closer to the target style, so deep rewrite there may be wasted motion

**Suggestions:**
- Start with shared semantic primitives or utilities (page-shell, tile-surface, tile-button, form-field)
- Retheme existing wrappers before deleting them
- Keep this wave scoped to public marketing surfaces; avoid auth-form churn
- Reserve #555 for decorative chrome only and use #888 or lighter for readable text

**Risk Assessment:** MEDIUM-HIGH — high visual payoff but highest scope-creep and accessibility risk.

### Cross-Plan Notes
- The glitch duplication approach is valid for simple visual content but should stay aria-hidden, be limited to safe surfaces, and support focus-visible plus reduced motion
- The biggest missing items across the plans are keyboard accessibility, modal focus handling, homepage DB-error resilience, and explicit scoping of shared primitives versus page-level polish

## Consensus Summary

### Agreed Strengths
- Plans target real, observable defects grounded in actual codebase state
- Wave structure is well-designed — sidebar/home parallel, then consistency sweep
- Cyberpunk Metro design language direction is correct and coherent

### Agreed Concerns
1. **HIGH: Logo tile has no intrinsic height** — `invert(1)` alone won't fix visibility; needs explicit sizing (aspect-ratio or min-height) since the image is absolutely positioned
2. **HIGH: Shared Tile primitive mutation risk** — Changing flex-col to flex-row in the base Tile component will break widgets, auth tiles, and other non-nav consumers. Need a nav-specific variant or prop guard.
3. **HIGH: Home page DB error handling** — `Promise.all` will crash the entire page on any single query failure. Need `Promise.allSettled` or try/catch per query.
4. **HIGH: Two sidebar implementations exist** — `side-nav.tsx` is dead code alongside `tile-nav.tsx`. Plans should explicitly note which is active and consider cleanup.
5. **MEDIUM: Accessibility regressions** — Replacing shadcn wrappers with plain HTML risks losing built-in ARIA behavior. Mobile overlay lacks focus trapping. #555 text on #111 bg fails WCAG contrast.

### Divergent Views
- None (single reviewer) — consider adding Gemini CLI review for adversarial coverage.
