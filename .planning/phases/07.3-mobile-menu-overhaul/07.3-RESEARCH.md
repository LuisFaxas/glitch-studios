# Phase 07.3: Mobile Menu Overhaul - Research

**Researched:** 2026-04-03
**Domain:** Mobile navigation overlay — layout, animation, gesture interaction
**Confidence:** HIGH

## Summary

This phase is a focused layout rework of an existing mobile nav overlay, not a greenfield build. The current overlay has the right architecture (Framer Motion AnimatePresence, tile grid, focus trap, route-change auto-close) but cramming too many items into viewport. The UI-SPEC already defines the target layout, the three research-driven decisions (D-01, D-09, D-10) have been answered in the spec, and the implementation is primarily CSS grid restructuring + adding two features: staggered entrance animation and swipe-down-to-dismiss gesture.

The project already has `motion@12.38.0` installed, which includes `drag="y"`, `dragConstraints`, `dragSnapToOrigin`, `dragElastic`, and `onDragEnd` — everything needed for the swipe-down dismiss without any new dependencies. The staggered entrance is a straightforward Framer Motion pattern using `variants` with `staggerChildren` or individual tile `transition.delay` values.

**Primary recommendation:** Restructure the overlay grid to 4 rows (logo, nav, widgets, auth+close), remove CompactNowPlaying and the 4 tab-bar-duplicated nav items, add `drag="y"` with `dragSnapToOrigin` for swipe dismiss, and add row-by-row stagger delays matching the UI-SPEC timing.

### Pattern 1: Staggered Row Entrance with Framer Motion
**What:** Each grid row animates in with increasing delay, creating a top-to-bottom cascade.
**When to use:** Menu open animation.
**Implementation approach:**

```typescript
// Wrap each row in a motion.div with individual delay
// Per UI-SPEC timing:
const rowVariants = {
  hidden: { opacity: 0, y: 12 },
  visible: (delay: number) => ({
    opacity: 1,
    y: 0,
    transition: {
      duration: 0.18,
      delay,
      ease: [0.215, 0.61, 0.355, 1], // ease-out-cubic
    },
  }),
}

// Logo: delay 0, y: -8
// Nav row: delay 0.08
// Widget row: delay 0.16
// Auth row: delay 0.24
// Exit: all simultaneous, 120ms fade
```

Source: UI-SPEC D-09 parameters, Motion docs for animation variants.

### Pattern 2: Swipe-Down to Dismiss with drag="y"
**What:** Overlay container responds to vertical drag; if dragged down past threshold, dismisses.
**When to use:** Touch gesture to close overlay.
**Implementation approach:**

```typescript
// On the main overlay motion.div:
<motion.div
  drag="y"
  dragConstraints={{ top: 0, bottom: 0 }}
  dragElastic={{ top: 0, bottom: 0.4 }}
  dragSnapToOrigin
  onDragEnd={(_, info) => {
    // Dismiss if dragged 80px+ with velocity > 0.3px/ms (300px/s)
    if (info.offset.y > 80 && info.velocity.y > 300) {
      onClose()
    }
  }}
  style={{ touchAction: "none" }}  // CRITICAL for touch devices
>
```

Key details:
- `dragConstraints={{ top: 0, bottom: 0 }}` prevents upward drag and defines snap-back range
- `dragElastic={{ top: 0, bottom: 0.4 }}` allows downward rubber-banding but blocks upward
- `dragSnapToOrigin` ensures overlay returns to position if swipe doesn't meet threshold
- `onDragEnd` checks both distance (80px) AND velocity (300px/s) per UI-SPEC
- `touchAction: "none"` on the drag container is REQUIRED for touch devices to prevent browser scroll interference
- Progressive opacity: use `onDrag` to update opacity based on `info.offset.y / 300`

Source: Motion docs (motion.dev/docs/react-drag), `motion-dom@12.38.0` types confirming `dragSnapToOrigin` availability.

### Anti-Patterns to Avoid
- **Duplicating tab bar items in the menu:** Industry standard (iOS "More" tab pattern, Spotify, Apple Music) is to NOT repeat items already accessible via the tab bar. The menu is for overflow/secondary destinations only.
- **Custom touch handler for swipe dismiss:** Motion's built-in `drag="y"` with `dragSnapToOrigin` handles edge cases (momentum, elastic snap-back, cancellation) that a manual `touchstart/touchmove/touchend` handler would need to reimplement.
- **Scroll container inside the overlay:** D-06 explicitly forbids scrolling. The grid must fit in 100dvh. Do not add `overflow-y: auto` as a fallback.
- **Using `staggerChildren` on a parent variant:** While this works for uniform grids, the UI-SPEC specifies different delays per row type (logo gets different y-offset than nav tiles). Use individual `transition.delay` per row instead.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Swipe-to-dismiss gesture | Custom touchstart/touchmove/touchend handler with velocity calculation | Motion `drag="y"` + `dragSnapToOrigin` + `onDragEnd` | Motion handles momentum physics, elastic snap-back, pointer vs touch normalization, and cancellation. Manual handler would need ~60 lines to match. |
| Staggered animation | Manual setTimeout chains or CSS animation-delay | Motion `transition.delay` per row element | Integrates with AnimatePresence exit animations. CSS-only approach can't coordinate enter/exit. |
| Focus trap | Custom keyboard event handler | Keep existing implementation | Already built and working. No change needed. |

## Common Pitfalls

### Pitfall 1: Touch Scroll Interference with Drag
**What goes wrong:** On mobile, browser tries to scroll the page when user swipes down on the overlay, interfering with the drag gesture.
**Why it happens:** Default browser touch-action allows scrolling. Motion's drag needs exclusive pointer control.
**How to avoid:** Set `style={{ touchAction: "none" }}` on the drag-enabled motion.div. The overlay already has `overflow: hidden` on body, but `touch-action` is the CSS-level fix.
**Warning signs:** Drag gesture feels "sticky" or doesn't register on iOS Safari.

### Pitfall 2: Drag Gesture Conflicting with Tile Tap
**What goes wrong:** User taps a nav tile but the overlay interprets it as a tiny drag, causing snap-back animation.
**Why it happens:** Any pointer movement during a tap is treated as drag start.
**How to avoid:** Motion handles this internally — small movements below drag threshold are treated as taps, not drags. The default drag distance threshold is typically 3px. If issues arise, add `dragListener={false}` to child tiles and only enable drag on the overlay container itself.
**Warning signs:** Tiles feel unresponsive or have a slight delay before navigation.

### Pitfall 3: 100dvh Not Including Safe Areas on iOS
**What goes wrong:** Menu content overflows or gets clipped by notch/home indicator.
**Why it happens:** `100dvh` accounts for browser chrome but safe area insets are additional.
**How to avoid:** The current implementation already uses `pt-[calc(0.75rem+env(safe-area-inset-top))]` and `pb-[calc(0.75rem+env(safe-area-inset-bottom))]`. The UI-SPEC updates these to `px-4` (16px) edge padding. Ensure safe area padding is applied INSIDE the 100dvh container, not added to it.
**Warning signs:** Content hidden behind notch on iPhone 15 Pro, or close button behind home indicator.

### Pitfall 4: AnimatePresence Exit vs Drag Dismiss Conflict
**What goes wrong:** When swipe dismiss triggers `onClose()`, AnimatePresence starts exit animation, but the overlay is already translated down from the drag. Exit animation looks janky — it snaps back to center then fades.
**How to avoid:** On drag dismiss, use `animate` controls to set `y` to window height and `opacity` to 0 as the exit animation, bypassing the standard fade-out. Or simply set exit to `{ opacity: 0, y: 100 }` so it continues downward during exit.
**Warning signs:** Overlay "jumps" when dismissed via swipe.

### Pitfall 5: Prefers-Reduced-Motion Not Respected
**What goes wrong:** Users with motion sensitivity see staggered tile animations.
**Why it happens:** Framer Motion doesn't automatically disable all custom animations for reduced motion users.
**How to avoid:** Use Motion's `useReducedMotion()` hook. If true, set all delays to 0 and durations to 0 (instant show/hide). The UI-SPEC accessibility checklist requires this.
**Warning signs:** Accessibility audit flags motion that can't be disabled.

## Open Questions

1. **Player bar visibility during overlay**
   - What we know: D-10 removes Now Playing from the menu. The player bar is a "persistent element" at the bottom.
   - What's unclear: Is the player bar visible BEHIND/BELOW the overlay (z-index layering), or is it hidden by the full-screen overlay? If hidden, removing Now Playing means no audio control while menu is open.
   - Recommendation: This is acceptable — the menu is a transient overlay (under 5 seconds of use typically). Users close the menu before interacting with the player. No action needed, but worth noting.

2. **Progressive opacity during drag**
   - What we know: UI-SPEC says `opacity: 1 - (deltaY / 300)` during swipe.
   - What's unclear: Whether to implement this via `onDrag` callback updating a `useMotionValue`, or via Motion's built-in `useTransform` mapping y position to opacity.
   - Recommendation: Use `useMotionValue` + `useTransform` for GPU-accelerated opacity during drag. This avoids React re-renders during the gesture.
