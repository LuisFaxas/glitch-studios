## Auth State: Sign In Placement Contract

### Desktop Sidebar (expanded, 280px)

- **Logged out:** "Sign In" button below CartIcon, full sidebar width, border border-[#222222] bg-[#111111], LogIn icon + "SIGN IN" 11px mono bold uppercase — pre-existing, matches current tile-nav.tsx exactly. No change needed.
- **Logged in:** Replace "Sign In" with a two-element row: user initials avatar (24px circle, bg #222222, text #f5f5f0, font-mono 10px uppercase) + "MY ACCOUNT" text link to /dashboard — left side; "Sign Out" icon button — right side. Both share the same full-width row with the existing border pattern.
- **Logged in admin:** Same row but "MY ACCOUNT" links to /admin instead of /dashboard.

### Desktop Sidebar (collapsed, 64px)

- **Logged out:** LogIn icon only, aria-label "Sign In" — pre-existing, no change.
- **Logged in:** Replace LogIn with user initials monogram (24px, circular) centered in the same 48px tap target. Clicking navigates to /dashboard (or /admin for admin role). No sign-out in collapsed mode — user must expand.

### Mobile Bottom Tab Bar

- Tab bar does NOT get a Sign In slot. Tab bar items are locked to: Beats, Services, Portfolio, Blog, Menu (5 slots). Sign In lives exclusively in the Mobile Nav Overlay (hamburger menu). This is pre-existing — do not add a 6th tab.

### Mobile Nav Overlay (hamburger/bottom sheet)

- **Logged out:** "Sign In" Tile in the bottom row, left slot — pre-existing in mobile-nav-overlay.tsx. No change to structure.
- **Logged in:** "Sign In" Tile replaced by "My Account" Tile (User icon + "MY ACCOUNT" label) linking to /dashboard. Sign Out moves to same slot if session exists — pre-existing behavior. No change to structure.

## Role-Based Redirect UX Contract

### After Successful Login (NAV-01)

- **Client role:** Hard redirect via `router.push("/dashboard")` — no toast, no loading spinner beyond the existing "Signing in..." button label state. Redirect is immediate on success.
- **Admin role:** Hard redirect via `router.push("/admin")` — same treatment.
- **Redirect detection:** Read session role from Better Auth session data after `signIn.email()` resolves. Role field: `session.user.role`. Value `"admin"` → /admin; all others → /dashboard.
- **Current bug (NAV-01):** login/page.tsx always redirects to /admin regardless of role. Fix: replace `router.push("/admin")` with role-conditional push.

### After Successful Registration

- Current behavior: redirects to / (homepage). Change to: redirect to /dashboard after registration so client immediately lands in their account. Show `toast.success("Account created. Welcome to Glitch Studios.")` — copy already exists, keep it.

### Unauthenticated access to /dashboard

- Pre-existing guard: dashboard/page.tsx already redirects to /login if no session. No change needed.

## Auth Pages: Visual Separation Contract (NAV-03)

### Client Login (/login) — Extend, do not redesign

- Pre-existing layout: centered card, max-w-md, GlitchLogo size="md", h1 "SIGN IN" (font-mono 24px bold uppercase), 2 fields (email, password), "Sign In" CTA, "Create Account" link.
- Phase 8 change: add role-conditional redirect logic only. No visual changes to the page itself.
- Page does NOT indicate it is for clients vs admins — it is the universal login. Context is provided by placement (public nav → /login) and the absence of any admin-specific framing.

### Client Register (/register) — Extend, do not redesign

- Pre-existing layout: centered card, max-w-md, GlitchLogo size="md", h1 "CREATE ACCOUNT" (font-mono 24px bold uppercase), 4 fields (name, email, password, confirmPassword), "Create Account" CTA, "Sign In" link.
- Phase 8 change: redirect to /dashboard after success instead of /. No visual changes.
- Page remains visually identical to login page in terms of card size, logo, and field density — this is intentional for brand consistency.

### Admin Login — Separation Strategy

- There is NO separate admin login page currently (confirmed: no admin/login/ route found in filesystem scan).
- Admin separation (NAV-03) is achieved through UX flow, not separate pages:
  - The public "Sign In" nav entry at /login is the client-facing entry point.
  - Admin users use the same /login page but are redirected to /admin after authentication via role check.
  - Admin login is NOT linked from the public nav, mobile tab bar, or mobile overlay — admin accesses /login directly (URL) or via /admin redirect.
  - Visual separation is reinforced by: (a) the /admin route group having its own layout with no public nav, and (b) the /login page having no "admin" framing or admin-specific copy.
- No new admin login page is created in this phase. The role-redirect logic is the separation mechanism.

## Redirect Flow Diagram

```
User clicks "Sign In" in nav
  → /login page

  Enter credentials → signIn.email()
    ├── Error → toast.error, stay on /login
    └── Success → check session.user.role
          ├── role === "admin" → router.push("/admin")
          └── role !== "admin" → router.push("/dashboard")

User clicks "Create Account" on login page
  → /register page

  Fill form → signUp.email()
    ├── Error → toast.error, stay on /register
    └── Success → toast.success → router.push("/dashboard")

Unauthenticated user visits /dashboard
  → redirect("/login")  [pre-existing, no change]
```
