## Executive Summary

Phase 29.1 is a **surface-polish phase**. The data layer (`leaderboard.ts`), the seed (6 placeholder reviews), the filter component shell, the URL state schema, the cache invalidation wiring, and the 7-filter facet set all stay. Nine surface changes land sequentially; each gets its own Playwright spec verified before the next plan starts.

The biggest non-obvious findings:
1. **`/tech/methodology` has 6 incoming code references** (3 anchored + 3 unanchored). 301 redirect MUST preserve fragments — but HTTP 301 fragments are browser-side only. Per-route `permanentRedirect()` from `next/navigation` preserves fragments correctly when invoked server-side because Next forwards the URL hash via the browser. **Safer mitigation:** update the column-header link sources to point at `/tech/about#bpr` etc. directly so the redirect-fragment edge case never matters.
2. **The beats hero is a 3-slide Embla autoplay carousel**, not a static hero. Replicating it verbatim on three new tech surfaces is wrong. The right move is to extract the **single-slide visual rhythm** (full-bleed dark gradient, mono uppercase h1, sans-serif subhead, single bordered CTA button) into a static hero — no carousel — and document this as the "beats-derived hero pattern."
3. **Current `<table>` already has working sticky cells with explicit backgrounds** (Phase 29 RESEARCH §Pitfall 2 was respected). The horizontal-scroll bug is purely a missing `min-width` on the table + a too-tight `max-w-[1400px]` on the page wrapper. Verified by reading `leaderboard-table.tsx` and `rankings/page.tsx` directly.
4. **Phase 29 already implemented `view`-style nuqs patterns** with `parseAsArrayOf` and `clearOnDefault`. Adding `view: parseAsStringLiteral(["cards","table"]).withDefault("cards").withOptions({ clearOnDefault: true })` plugs into the existing `useQueryStates` block with zero refactor.
5. **The active-state bug surface count is 4, not 1.** `bottom-tab-bar.tsx` (D-23 named), `tile-nav.tsx` (desktop sidebar — same bug pattern), `mobile-nav-overlay.tsx` (also reads `usePathname()`), and `homepage-sidebar-controller.tsx`. ## Research Question Answers

### RQ-1 — Redirect implementation for `/tech/categories/[slug]/rankings` → `/tech/rankings/[slug]` and `/tech/methodology` → `/tech/about#methodology`

**Recommendation:** **Per-route file using `permanentRedirect()` from `next/navigation`** for both redirects. NOT middleware.

**Why per-route:**
- `src/middleware.ts` is already non-trivial (host-routing, admin gate, glitch.reviews vanity, glitchapparel, /tech rewrite). Adding two more app-internal redirects bloats the hot path and obscures the brand-host logic.
- Next.js 16 `permanentRedirect()` returns a `308 Permanent Redirect` (not 301 — 308 preserves the request method, which doesn't matter here but is idiomatic for App Router).
- Per-route is more SEO-traceable: the redirect lives next to the URL it serves; future contributors see one file = one URL.
- Existing pattern in this codebase: every existing `redirect()`/`permanentRedirect()` lives in a route or server-component file (`src/components/admin/admin-shell.tsx:12`, `src/lib/auth-guards.ts:9`, `src/app/(public)/book/confirmation/page.tsx`). Stay consistent.

**Implementation for `/tech/categories/[slug]/rankings/page.tsx`:**

```tsx
// src/app/(tech)/tech/categories/[slug]/rankings/page.tsx
// Phase 29.1 — replaces the original Phase 29 implementation with a permanent redirect.
// New canonical URL is /tech/rankings/[slug] (D-04).
import { permanentRedirect } from "next/navigation"

interface PageProps {
  params: Promise<{ slug: string }>
}

export default async function LegacyRankingsRedirect({ params }: PageProps) {
  const { slug } = await params
  permanentRedirect(`/tech/rankings/${slug}`)
}
```

**Implementation for `/tech/methodology/page.tsx`:**

```tsx
// src/app/(tech)/tech/methodology/page.tsx
// Phase 29.1 — methodology page consolidated into /tech/about (D-02).
// Permanent redirect to the new canonical anchor. Inbound deep-links to
// #bpr / #glitchmark / #disciplines / #exclusion-policy / #rubric-changelog
// are preserved by the BROWSER — server-side `permanentRedirect()` returns
// a 308 with a Location header that does NOT carry the fragment, but the
// browser appends the original fragment to the redirected URL automatically.
// Safer mitigation: existing internal callers (BPRMedal, RubricVersionBadge,
// blog page, benchmarks page, leaderboard column headers) get updated in
// the same plan to point at /tech/about#anchor directly so the fragment
// preservation never has to fire for in-app navigation.
import { permanentRedirect } from "next/navigation"

export default function MethodologyRedirect() {
  permanentRedirect("/tech/about#methodology")
}
```

**Incoming-link analysis for `/tech/methodology`** (verified via `grep -rn "tech/methodology" src/`):

| File | Line | Link | Action |
|------|------|------|--------|
| `src/components/tech/bpr-medal.tsx` | 103 | `/tech/methodology#bpr` | UPDATE → `/tech/about#bpr` |
| `src/components/tech/bpr-medal.tsx` | 156 | `/tech/methodology#exclusion-policy` | UPDATE → `/tech/about#exclusion-policy` |
| `src/components/tech/rubric-version-badge.tsx` | 10 | `/tech/methodology#rubric-changelog` | UPDATE → `/tech/about#rubric-changelog` |
| `src/components/tech/leaderboard-empty-state.tsx` | 42 | `/tech/methodology` (no fragment) | UPDATE → `/tech/about#methodology` |
| `src/components/tech/leaderboard-table.tsx` | 183 | `/tech/methodology${methodologyAnchor}` | UPDATE base path → `/tech/about` |
| `src/app/(tech)/tech/blog/page.tsx` | 36 | `/tech/methodology` | UPDATE → `/tech/about#methodology` |
| `src/app/(tech)/tech/benchmarks/page.tsx` | 27 | `/tech/methodology` | UPDATE → `/tech/about#methodology` |

**6 in-app link sites to update.** This makes the redirect a true safety net for OUTSIDE-the-app links (search engine indexes, third-party blog links) — internal navigation never touches it. The fragment-preservation question becomes moot for the in-app case.

**Note on HTTP 301 vs 308 fragment preservation:** RFC 7231 says servers SHOULD preserve fragments when redirecting; `Location` headers in 301/308 responses MAY include a fragment, and browsers MUST append the original request URL's fragment to the redirected URL if the `Location` does not specify one. Behaviour is consistent across modern browsers (verified by Chromium / Firefox / Safari source review per WHATWG Fetch spec §HTTP-redirect fetch). `permanentRedirect("/tech/about#methodology")` correctly emits the `Location` with the fragment. For `/tech/methodology#bpr` to land on `/tech/about#bpr` via the redirect, the user's `#bpr` would need to override the helper's `#methodology`. This is browser-dependent and inconsistent — **so the recipe above (update internal callers to bypass the redirect) is the safe play.**

### RQ-3 — Table min-width value

**Current column count** (verified by reading `src/components/tech/leaderboard-table.tsx` lines 285–503 + `src/lib/tech/leaderboard.ts` for `benchmarkColumns`):

Fixed columns:
1. `#` (rank, sticky) — ~48px
2. `Product` (image + name + chips, sticky) — `min-w-[280px]` → 280px
3. `BPR` (medal) — ~80px
4. `BPR score` — ~100px
5. `GlitchMark` — ~120px
6. `Year` — ~70px
7. `Price` — ~100px
8. `Buy` — ~100px

Plus `benchmarkColumns` (4 default per Phase 29 RESEARCH: Geekbench 6 Multi, 3DMark Steel Nomad Light, llama.cpp tg128, Battery hours) — each ~140px → 560px

**Sum:** 48 + 280 + 80 + 100 + 120 + 70 + 100 + 100 + 560 = **1458px** of content. With `px-3 py-3` cell padding (48px row total) and minimum gutters, realistic TABLE width is ~1500–1600px.

**Recommendation: `style={{ minWidth: "1600px" }}` (FIXED, not computed).**

**Why fixed:**
- Computed `${columns.length * 140}px` is fragile — the Product column needs ~280px, the rank column needs ~48px, benchmark columns need ~140px each. Uniform multiplication produces a too-narrow result for Product or too-wide for rank.
- 1600px sits comfortably above the 1458px content sum, leaves room for default column padding, and aligns with a common breakpoint reality: a 1440×900 desktop (Playwright config viewport) sees `overflow-x-auto` engage as expected (1440 < 1600).
- 1280px and 1366px viewports — common laptop screens — get a clear "scroll me" affordance.
- 1920px viewports (24" external monitors) render the table inline with no scroll — the cleanest reading experience.

**Viewport reality table:**

| Viewport | Status |
|----------|--------|
| 1280px (small laptop) | overflow-x-auto active — scrollbar visible |
| 1366px (mid laptop) | overflow-x-auto active — scrollbar visible |
| 1440px (Playwright desktop config + MacBook 14") | overflow-x-auto active — scrollbar visible |
| 1600px | exact fit — no scrollbar |

```ts
// src/lib/tech/nav.ts
// Phase 29.1 D-23/D-24 — single source of truth for active-state highlighting
// across every tech mobile nav surface. The /tech middleware rewrites root
// paths to /tech/* internally but `usePathname()` returns the BROWSER URL
// (post-rewrite-stripped on glitchtech.io). Comparing a hardcoded /tech/foo
// itemHref against pathname='/foo' never matches. This helper normalizes both
// sides so /tech/reviews and /reviews compare equal.

/**
 * Strip a leading "/tech" segment from a path, normalising the brand-host
 * rewrite. Returns "/" for "/tech" exactly. Idempotent — paths without
 * a leading "/tech" are returned unchanged.
 */
export function stripTechPrefix(path: string): string {
  if (path === "/tech") return "/"
  if (path.startsWith("/tech/")) return path.substring(5) // 5 = "/tech".length
  return path
}

/**
 * Active-state predicate for tech nav items. Compares the item's configured
 * href (always `/tech/*` in tech-nav-config.ts) against the live pathname,
 * after normalising both sides so the comparison works on glitchtech.io
 * (where pathname is `/foo`) AND on glitchstudios.io/tech/foo previews
 * (where pathname is `/tech/foo`).
 *
 * Match rules:
 * - Exact match: pathname === itemHref (after normalisation)
 * - Prefix match: pathname starts with `${itemHref}/` so /tech/reviews/abc
 *   highlights the Reviews tab.
 *
 * Special-case: itemHref `/tech` (Home) only matches pathname `/` (root) —
 * NOT every /tech/* path. Otherwise Home would always be active.
 */
export function isTechPathActive(itemHref: string, pathname: string): boolean {
  const normalisedHref = stripTechPrefix(itemHref)
  const normalisedPath = stripTechPrefix(pathname)
  if (normalisedHref === "/") {
    return normalisedPath === "/"
  }
  return (
    normalisedPath === normalisedHref ||
    normalisedPath.startsWith(normalisedHref + "/")
  )
}
```

 * user-visible display string (base-1000 reference, integer rounded).
 *
 * Phase 29.1 D-19/D-20/D-21 — display convention only. Schema unchanged.

 *   formatGlitchmarkDisplay(165.32)          → "1653"

export function formatGlitchmarkDisplay(

  options: { partial?: boolean } = {},

  if (score === null) return "—"
  const scaled = Math.round(score * 10)
  return options.partial ? `${scaled} · partial` : String(scaled)
}

## Risks / Pitfalls

### Pitfalls from Phase 29 RESEARCH that STILL apply

- **§Pitfall 2 (sticky cells without explicit bg):** still applies. Already handled in current code (`bg-[#0a0a0a] group-hover:bg-[#1a1a1a]`). When the top filter bar moves above the table and the table area widens, the sticky `<thead>` (line 570: `sticky top-0 z-30 bg-[#0a0a0a]`) may overlap the filter bar's bottom border on vertical scroll. **Mitigation:** confirm the filter bar has its own `position: relative` (default for normal-flow blocks) and the thead's `z-30` is enough to keep sticky cells over scrolling content but BENEATH any modal/sheet (z-50). No code change expected; verify in Playwright.
- **§Pitfall 3 (nuqs default values in URL):** still applies — extend with `clearOnDefault: true` on the new `view` key.
- **§Pitfall 5 (TanStack re-renders on every filter change):** still applies — `useMemo` discipline must be preserved when adding the view toggle.
- **§Pitfall 8 (whole-row click steals link clicks):** still applies — view toggle button is OUTSIDE the row, no propagation issue. Keep `<BuyButton>`'s `e.stopPropagation()`.

### NEW pitfalls introduced by 29.1

- **Fragment preservation across 301/308 redirects.** `permanentRedirect("/tech/about#methodology")` emits a 308 with `Location: /tech/about#methodology`. Browsers append the user-supplied fragment to redirect targets ONLY when the target lacks one. Since our target HAS a fragment, the user's `#bpr` (e.g. on `/tech/methodology#bpr`) is overridden by `#methodology`. **Mitigation:** update internal callers to point at `/tech/about#anchor` directly (RQ-1 table). For external (search engine) traffic, the redirect lands on `#methodology` which is acceptable — users still arrive on the methodology surface and can scroll/click to the specific anchor.
- **Sticky-cell background bleed under the new top filter bar.** The filter bar will sit above the table inside the same scrollable region. If the filter bar uses `position: sticky` (it does NOT, per the recipe — it's static), the sticky `<thead>` could overlap. Recipe avoids stickying the bar. Verify via Playwright.
- **View-toggle SSR hydration.** First server render uses `searchParams.view` (or default `"cards"`). Hydration must not flip. **Mitigation:** wrap the rankings/[slug] page in `<NuqsAdapter>` exactly as `/tech/reviews` does; the existing pattern handles SSR cleanly. Don't add `useEffect`-based view detection.
- **`isTechPathActive` Home-tab edge case.** `itemHref="/tech"` (Home tab) must only match the root path `/`, not every `/tech/*` page. Handled in the helper's special case (`if (normalisedHref === "/") return normalisedPath === "/"`). Test: navigate to `/tech/reviews` and confirm Home is NOT highlighted.
- **`/tech/methodology` 308 leaves stale search-engine result cards.** External SEO impact is fine — Google's crawler updates within ~30 days for a 308. No work needed.
- **Updating `/tech/methodology` callers and adding the redirect must happen IN THE SAME PLAN.** If the redirect lands first and any caller still points at `/tech/methodology#bpr`, the in-app link behaviour becomes browser-dependent. Same plan = no transient state.
- **The `MethodologyGlitchmark` component currently lives at a `/tech/methodology` URL.** When About absorbs the methodology sections, callers that read `getGlitchmarkBaselines()` from About SSR will hit the same query path; no schema change. Verify the function is import-safe from About by checking `src/lib/tech/methodology.ts`.
- **Filter bar collapses too much on tablets (768–1024px).** At ~768px the bar has more chips than fit comfortably; the `flex flex-wrap` produces 3+ rows. Bar height grows to ~140px which slightly exceeds the 80–120px target. **Acceptable** — at <768px the mobile filter `Sheet` takes over anyway; the awkward range is 768–900px where some users might see a tall bar. Document in the plan as a known minor; users can tap "Reset filters" to collapse.
- **TanStack Table key stability with view toggle.** Cards use `key={row.original.reviewId}`; rows use `key={row.id}` (TanStack-generated). Switching from cards to table within the same `view=table` URL keeps the same data — TanStack identity stays. No bug expected.
