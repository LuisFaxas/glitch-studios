# Cross-AI Plan Review — Phase 07.2

## Codex Review (gpt-5.4, reasoning: xhigh)

Across the set, the sequencing is mostly sensible: audit first, fix structural mobile foundations, adjust the hero, then run a final verification pass. The main weakness is proof quality: too much of the phase can "pass" by generating screenshots, without actually proving no overflow, correct player/tab stacking, or full page/state coverage.

### Plan 01

**Summary**
Good audit-first intent, but the plan is internally inconsistent on coverage and does not truly emulate mobile behavior. It will produce useful artifacts, but not enough evidence by itself to validate the phase goals.

**Strengths**
- Starts with a baseline audit before fixes, which is the right order.
- Uses explicit viewport targets and a stable screenshot directory.
- Reuses an existing Playwright pattern, which reduces implementation risk.
- Intentionally avoids failing on a 404, which is pragmatic for discovery.

**Concerns**
- HIGH: The page inventory is inconsistent. The plan says 11 public pages and ~33 screenshots, but only lists 10 pages and 30 tests.
- HIGH: Creating a fresh `browser.newContext()` under `--project=desktop` with only a viewport set does not fully emulate mobile. You lose `isMobile`, touch behavior, mobile UA, and other device semantics.
- MEDIUM: Screenshot generation is not validation. A 404 page or broken layout can still make the test pass.
- MEDIUM: The screenshot directory creation is not specified; this can fail at runtime depending on environment behavior.
- LOW: It only captures default page state, not overlay-open or player-active states, which are central to this phase.

**Suggestions**
- Resolve the public-page list first and explicitly document any intentional exclusions.
- Use true mobile emulation for each width, or at minimum set `isMobile` and `hasTouch` in the created contexts.
- Add assertions for horizontal overflow and basic page validity in addition to screenshots.
- Create the screenshot directory explicitly before capturing files.
- Add separate audit cases for overlay-open and player-active states if those are phase-critical surfaces.

**Risk Assessment**
MEDIUM. It is a useful baseline plan, but its current form is too weak and inconsistent to serve as a reliable audit contract.

### Plan 02

**Summary**
This plan attacks the right structural issues, especially tab order and content padding, but D-07 is only partially covered. The bottom padding work is valuable, yet it does not guarantee the player bar actually stacks above the tab bar.

**Strengths**
- Directly addresses the tab bar contract from the phase goal.
- Uses a thin client wrapper instead of converting the whole layout to client-side.
- Reuses existing CSS variables and safe-area handling instead of hardcoding pixel math.
- Keeps the overlay fix narrow and low-risk.

**Concerns**
- HIGH: The plan changes content padding, but does not explicitly change or verify player bar positioning. Padding alone does not prove the player stacks above the tab bar.
- MEDIUM: It introduces `mobile-content-wrapper.tsx` but does not list that file in `files_modified` or artifacts.
- MEDIUM: `playerActive = !!currentBeat && !isMinimized` may not match actual on-screen player visibility. Paused/minimized states could still need clearance.
- MEDIUM: Moving `Footer` into the wrapper could change semantics or layout behavior if the current structure expects it outside `<main>`.

**Suggestions**
- Add an explicit step to inspect and, if needed, update the player component's fixed bottom offset and z-index.
- Verify padding against actual player visibility, not just `currentBeat` plus `isMinimized`.
- Include the new wrapper file in plan metadata.
- Preserve current footer semantics unless there is a deliberate reason to change them.
- Add one verification check that proves the stacked player/tab layout in the DOM or via Playwright.

**Risk Assessment**
MEDIUM-HIGH. The navigation changes are solid, but the plan does not yet fully prove or implement the stacking behavior required by the phase.

### Plan 03

**Summary**
The plan is focused and directionally correct, but the hero positioning details are shaky and the splash verification is not real verification. It risks solving the headline issue while missing the actual mobile rendering outcome.

**Strengths**
- Targets the known mobile hero problems without broad scope creep.
- Uses responsive breakpoint guards to keep desktop behavior intact.
- Keeps CTA stacking behavior unchanged, which is probably correct.

**Concerns**
- HIGH: Reducing the hero to `70vh` while leaving internal positioning tied to `50vh` and `12vh` can still misplace content. Those values are viewport-relative, not hero-relative.
- HIGH: The splash task mostly inspects code, but Plan 01 explicitly skips the splash. That means the plan never truly verifies splash behavior on mobile.
- MEDIUM: It says to review homepage audit findings, but only allocates `hero-section.tsx` and `splash-overlay.tsx`. Any other homepage issue becomes awkwardly out of scope.

**Suggestions**
- Replace viewport-based positioning with container-relative positioning such as `top-1/2`, `bottom-*`, or section-relative percentages.
- Add a dedicated mobile test run with splash enabled at 375px.
- Define how other homepage issues found in the audit should be handled if they are outside these two files.
- Add a concrete assertion that the primary CTAs are visible at 375px without scrolling.

**Risk Assessment**
MEDIUM. The fix direction is good, but the current positioning approach and splash verification leave too much uncertainty.

### Plan 04

**Summary**
This is the most important plan and also the weakest as a quality gate. It has the right intent, but the proposed verification is too manual and too screenshot-dependent to prove the phase is actually complete.

**Strengths**
- Uses a final audit plus human approval, which is appropriate for visual UX work.
- Treats the post-fix pass as the phase gate rather than an afterthought.
- Keeps follow-up fixes targeted.

**Concerns**
- HIGH: Screenshot review is not enough to prove no horizontal overflow, correct touch target sizes, or non-404 rendering.
- HIGH: It proposes reading 375px screenshots only, but the phase success criteria explicitly include 375px, 390px, and 430px.
- HIGH: The rule to defer issues that require more than 3 files conflicts with the phase goal. If the mobile experience is still broken, the phase is not done.
- MEDIUM: `depends_on` omits Plan 01 even though this plan requires its test spec and audit artifacts.
- MEDIUM: The human verification URL does not match the earlier `baseURL`, which creates handoff risk.

**Suggestions**
- Turn the audit into an assertive test: check `scrollWidth <= innerWidth`, confirm key elements are visible, and fail on unexpected 404s.
- Verify all three target widths, not just 375px.
- Add explicit post-fix checks for overlay-open and player-active states.
- Make Plan 01 an explicit dependency.
- Standardize the verification environment and URL.
- Remove the "future phase if >3 files" escape hatch for anything that violates current success criteria.

**Risk Assessment**
HIGH. This is the final gate, and in its current form it can approve a phase without actually proving the mobile overhaul works.

## Consensus Summary

### Top Concerns (by severity)

1. **Verification is too weak** — Plans 01 and 04 rely on screenshots without automated assertions for horizontal overflow, element visibility, or correct stacking. The phase can "pass" without proving it works.
2. **Player bar stacking not fully addressed** — Plan 02 handles content padding but never explicitly verifies or adjusts the player bar's fixed position relative to the tab bar.
3. **Mobile emulation incomplete** — Playwright contexts use viewport-only, missing `isMobile`, `hasTouch`, and mobile UA string. This may miss touch-specific layout behavior.
4. **Page count inconsistency** — Plan 01 says 11 pages / ~33 screenshots but only lists 10 pages / 30 tests.
5. **Overlay and player-active states untested** — The audit only captures default page state, not the overlay-open or player-playing states that are central to this phase.
6. **Escape hatch weakens the gate** — Plan 04's "defer if >3 files" rule conflicts with the phase goal of zero layout breakage.

### Strengths Noted

- Correct sequencing: audit → fix → verify
- Structural changes are narrow and well-scoped
- Existing patterns reused (Playwright, CSS variables, component wrapping)
- Tab bar restructuring directly maps to success criteria

### Recommendations

The plans should be re-planned via `/gsd:plan-phase 07.2 --reviews` to address the HIGH-severity concerns, particularly adding automated overflow assertions and true mobile emulation to the Playwright specs.
