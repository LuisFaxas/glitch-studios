# Phase 07.2: Mobile Experience Overhaul - Research

**Researched:** 2026-04-01
**Domain:** Mobile responsive UX, mobile navigation patterns, player/nav coexistence
**Confidence:** HIGH

## Summary

This phase is an audit-and-fix pass to make the existing mobile experience match the quality of the desktop site. The codebase already has a functional bottom tab bar, mobile nav overlay, and mobile player bar -- all need restructuring, not building from scratch. The UI-SPEC (07.2-UI-SPEC.md) already locks down visual contracts for spacing, typography, color, and layout. Research focuses on the deferred decisions: industry-leading mobile nav patterns (D-02, D-03), player+tab bar coexistence (D-07), hero section mobile best practices (D-05), viewport breakpoints (D-10), and Playwright audit methodology (D-11).

The primary risk is the player bar + tab bar stacking adding 136px of fixed UI at the bottom of the viewport, which on a 375px-wide / 667px-tall phone leaves only ~531px of scrollable content. This is manageable with the stacked-bars pattern (Spotify, Apple Music, SoundCloud all use it), but content padding must be dynamic based on player state.

**Primary recommendation:** Execute as a Playwright-first audit (screenshot every page at 3 viewports), then fix issues systematically page-by-page, with tab bar restructuring and player coexistence as the foundational structural work done first.

### Dynamic Bottom Padding Pattern

The public layout currently uses static `pb-16` for tab bar clearance. This needs to become dynamic based on player state:

```typescript
// Current (static):
<main className="pb-16 md:pb-0">

// Target (dynamic, via CSS custom properties):
// When player is inactive: pb = tab-bar-height + safe-area
// When player is active: pb = tab-bar-height + player-height + safe-area
```

**Approach:** Use a CSS class toggle or CSS custom property set by the AudioPlayerProvider context. The player provider already tracks `currentBeat` and `isMinimized` -- expose a `playerVisible` boolean that the layout can consume.

```typescript
// In layout or a wrapper component:
const { currentBeat, isMinimized } = useAudioPlayer()
const playerActive = !!currentBeat && !isMinimized

// Dynamic padding via Tailwind arbitrary values:
className={clsx(
  "md:pb-0",
  playerActive
    ? "pb-[calc(var(--tab-bar-height)+var(--spacing-player)+env(safe-area-inset-bottom))]"
    : "pb-[calc(var(--tab-bar-height)+env(safe-area-inset-bottom))]"
)}
```

## Research Findings

### D-02: Cart Placement and Tab Bar Layout (MEDIUM confidence)

**Industry pattern analysis:**

| App/Site | Tab Count | Cart Position | Player Position |
|----------|-----------|---------------|-----------------|
| Spotify | 5 | N/A (no cart) | Mini-player above tab bar |
| Apple Music | 5 | N/A | Mini-player above tab bar |
| SoundCloud | 5 | N/A | Mini-player above tab bar |
| Beatstars | 5 | In tab bar (rightmost) | Mini-player above tab bar |
| Bandcamp | 4 | Separate page (no persistent cart) | Mini-player above tab bar |
| Shopify stores | 5 | In tab bar (rightmost) or floating badge | N/A |

**Recommendation: Keep cart as rightmost tab bar item.** This matches e-commerce convention (Beatstars, Shopify mobile patterns) and the UI-SPEC already specifies this layout. The cart icon with badge count is immediately discoverable. Moving it to a floating button or hiding it in the overlay would reduce conversion -- users browsing beats need one-tap cart access.

The 6-item tab bar (4 nav + menu + cart) is at the upper limit of the 3-5 item guideline, but acceptable because all items are icon-only (no labels to truncate) and each slot at `flex-1` on a 375px screen yields ~62px per item -- well above the 48px minimum touch target.

### D-03: Mobile Nav Overlay Improvements (MEDIUM confidence)

**Industry patterns for full-screen mobile overlays on creative/music sites:**

1. **Visual hierarchy through size variation** -- The existing tile-grid approach (wide + small tiles) is already industry-leading. Most creative sites use flat link lists. The Metro-inspired tile grid is distinctive and on-brand.

2. **Logo as anchor** -- The UI-SPEC correctly places the logo tile as the primary visual element. Creative sites (resn.co.nz, activetheory.net, locomotive.ca) use their logo prominently in mobile nav overlays.

3. **Quick dismiss patterns** -- The overlay already supports Escape, X button, and route change dismiss. This covers all standard patterns. Swipe-to-dismiss is intentionally omitted (correct -- it conflicts with internal scrolling).

4. **Scroll handling** -- For screens where the overlay content exceeds viewport (small phones with many widgets), `overflow-y-auto` with body scroll lock is the correct approach. Already implemented.

**Recommendations for enhancement:**
- The existing overlay is already well-structured. Focus fixes on:
  - Ensuring the LogoTile renders at `wide` size spanning full 2-column grid (per UI-SPEC)
  - Verifying all tiles have 48px minimum touch targets
  - Confirming focus trap works correctly (already implemented)
  - Testing overlay scroll behavior on small phones (375px)

### D-05: Mobile Hero Best Practices for Creative/Music Sites (MEDIUM confidence)

**Key findings from industry research:**

1. **Mobile hero height:** 50-70vh recommended for mobile (vs 90-100vh on desktop). The current hero is `h-[90vh]` with no mobile breakpoint. At 90vh on a 667px-tall phone, the hero consumes 600px -- leaving almost no visible content below, especially with the tab bar. **Reduce to 70vh on mobile** (`h-[70vh] md:h-[90vh]`).

2. **Content simplification:** Mobile heroes should show headline + one CTA, not three buttons stacked vertically. The current hero has 3 CTA buttons in a `flex-col sm:flex-row` layout -- on mobile they stack and push below the fold. **Reduce to primary CTA + secondary CTA only on mobile**, or arrange as 2 buttons side-by-side.

3. **Typography scaling:** The subtitle currently uses `text-2xl md:text-4xl` (28px to 40px). The UI-SPEC specifies Display (28px) for mobile, which matches. The logo glitch image uses `w-[80vw] max-w-[600px]` which scales appropriately.

4. **Sidebar offset issue:** The hero uses `translateX(-32px)` when sidebar is collapsed to center content. On mobile, the sidebar doesn't exist, so `collapsed` should have no effect. Need to verify `useSidebar` returns a sensible value on mobile.

5. **Performance:** Parallax (`useScroll` + `useTransform`) should be verified on mobile. The scroll indicator animation is lightweight. The scanline background pattern is CSS-only and performant.

### D-07: Player Bar + Tab Bar Coexistence (HIGH confidence)

**The industry-standard pattern is stacked bars.** Every major music app uses this approach:

| App | Tab Bar | Mini Player | Stacking |
|-----|---------|-------------|----------|
| Spotify | Bottom, fixed | Above tab bar, fixed | Tab bar bottom, mini-player directly above |
| Apple Music | Bottom, fixed | Above tab bar, fixed | Same as Spotify |
| SoundCloud | Bottom, fixed | Above tab bar, fixed | Same as Spotify |
| Tidal | Bottom, fixed | Above tab bar, fixed | Same as Spotify |

**The stacking order in the UI-SPEC is correct:**
```
[Page Content -- scrollable]
[Player Bar -- 72px, z-40, bottom: tab-bar-height]
[Tab Bar -- 64px, z-50, bottom: 0]
```

The existing player bar already positions itself at `bottom-[var(--tab-bar-height,0px)]` on mobile. The z-index layering (tab bar z-50, player z-40) is correct -- the tab bar should visually "anchor" the bottom.

**Key implementation detail:** The main content padding must be dynamic. When the player is active, total fixed bottom UI is 136px (64px tab + 72px player) plus safe area. When inactive, it's 64px plus safe area. The current static `pb-16` (64px) only accounts for the tab bar.

**Mini-player on expand:** The existing player bar shows cover art + title + play/pause + waveform on mobile. This is equivalent to what Spotify/Apple Music show in their mini-player. No need for an "expanded" player view in this phase -- the current mobile layout is sufficient.

## Common Pitfalls

### Pitfall 1: Static Bottom Padding with Dynamic Player
**What goes wrong:** Content gets hidden behind the player bar when it appears, or there's a jarring layout shift when the player opens/closes.
**Why it happens:** The current `pb-16` in layout.tsx is static and only accounts for the tab bar.
**How to avoid:** Make bottom padding reactive to player state. Use the AudioPlayerProvider context to toggle padding classes. Add a smooth transition on the padding change to avoid jarring shifts.
**Warning signs:** Content cut off at the bottom of any page when audio is playing.

### Pitfall 2: Hero Section Overflows on Small Phones
**What goes wrong:** Hero at 90vh on a small phone pushes CTAs below the fold or causes the subtitle text to overlap with the logo.
**Why it happens:** 90vh is designed for desktop viewports. On a 667px phone, 90vh = 600px, and with the tab bar (64px), only 603px of viewport is usable.
**How to avoid:** Use `h-[70vh] md:h-[90vh]` and verify CTA visibility at 375px viewport.
**Warning signs:** CTAs not visible without scrolling on mobile hero.

### Pitfall 3: Sidebar Context Undefined on Mobile
**What goes wrong:** The hero section uses `useSidebar()` for `collapsed` state to calculate content offset. On mobile, the sidebar doesn't render, but the context provider may still return values that affect hero positioning.
**Why it happens:** `HomepageSidebarController` wraps the entire layout including mobile. The `translateX(-32px)` offset for collapsed sidebar should not apply on mobile.
**How to avoid:** Gate the sidebar offset with a mobile check: `const sidebarOffset = collapsed && !isMobile ? -32 : 0` or use CSS: `md:translate-x-[-32px]` only.
**Warning signs:** Hero content appears off-center on mobile.

### Pitfall 4: Tab Bar Icon Confusion Without Labels
**What goes wrong:** Users can't identify which icon corresponds to which section, especially Blog (BookOpen) vs Beats (Music).
**Why it happens:** Icon-only tab bars rely on universally recognized icons, but `BookOpen` and `Music` are not as universally distinct as Home/Search/Profile.
**How to avoid:** The UI-SPEC mandates icon-only. Ensure proper `aria-label` on every tab. Test with real users if possible. The existing tab bar already uses icon-only successfully -- the risk is low since the site audience is tech-savvy.
**Warning signs:** User confusion in testing about which tab does what.

### Pitfall 5: iOS Safari Safe Area Issues
**What goes wrong:** On iPhones with home indicator (notch phones), the tab bar renders behind the home indicator or the safe area padding doubles up.
**Why it happens:** `env(safe-area-inset-bottom)` is only non-zero on devices with home indicators. But it can interact unexpectedly with `position: fixed` and `bottom: 0`.
**How to avoid:** The tab bar already uses `pb-[env(safe-area-inset-bottom)]`. Verify in Playwright at iPhone viewport sizes. Also check that the player bar's `bottom: calc(var(--tab-bar-height) + env(safe-area-inset-bottom))` is correct.
**Warning signs:** Extra gap at bottom of tab bar, or content clipped behind home indicator.

### Pitfall 6: Playwright Screenshots Don't Match Real Devices
**What goes wrong:** Playwright uses viewport emulation, not real device rendering. CSS features like `env(safe-area-inset-bottom)` return 0 in Playwright.
**Why it happens:** Playwright doesn't emulate device-specific CSS environment variables.
**How to avoid:** Accept this limitation. Playwright catches layout/overflow issues. Safe area testing requires real device or Safari Responsive Design Mode. Note this in the audit output.
**Warning signs:** Safe area padding appears correct in Playwright but broken on real iPhone.

```typescript

import path from "path"

const screenshotDir = path.resolve(
  __dirname,
  "../.planning/phases/07.2-mobile-experience-overhaul/screenshots"

const viewports = [
  { name: "375", width: 375, height: 812 },
  { name: "390", width: 390, height: 844 },
  { name: "430", width: 430, height: 932 },
]

  { name: "home", path: "/" },
  { name: "beats", path: "/beats" },

  { name: "services", path: "/services" },
  { name: "portfolio", path: "/portfolio" },
  { name: "artists", path: "/artists" },
  { name: "blog", path: "/blog" },
  { name: "contact", path: "/contact" },
  { name: "login", path: "/login" },
  { name: "signup", path: "/signup" },
]

for (const vp of viewports) {
  for (const pg of pages) {
    test(`${pg.name} at ${vp.name}px`, async ({ browser }) => {
      const ctx = await browser.newContext({
        viewport: { width: vp.width, height: vp.height },
      })
      const page = await ctx.newPage()
      await page.addInitScript(() => {
        sessionStorage.setItem("glitch-splash-seen", "true")
      })
      await page.goto(pg.path, { waitUntil: "networkidle" })
      await page.waitForTimeout(500)
      await page.screenshot({
        path: path.join(screenshotDir, `${pg.name}-${vp.name}-default.png`),
        fullPage: true,
      })
      await ctx.close()
    })
  }
}
```
