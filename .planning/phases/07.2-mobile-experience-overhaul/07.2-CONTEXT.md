# Phase 07.2: Mobile Experience Overhaul - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Make mobile a first-class citizen across the entire site. Fix navigation layout (move menu trigger to the right), ensure the hero and every page render correctly on mobile viewports, resolve player bar + tab bar coexistence, and eliminate all layout breakage. This is an audit-and-fix phase, not a redesign — the desktop experience is the reference, and mobile needs to match its quality.

</domain>

<decisions>
## Implementation Decisions

### Navigation & Menu Model
- **D-01:** Keep the current bottom tab bar items (Beats, Services, Portfolio, Blog) but **move the menu trigger to the right side** instead of center position. The center position feels wrong.
- **D-02:** Cart placement and overall tab bar layout should be **researched for industry-leading patterns**. Don't assume current cart icon position is optimal — could be a floating button, badge on menu, inside the overlay, or removed from the tab bar entirely. Researcher must investigate best-in-class mobile nav from music/creative industry apps and sites.
- **D-03:** The mobile nav overlay (full-screen menu with tile grid, widgets, auth) also needs **research-backed improvements**. Researcher should investigate mobile overlay/drawer patterns and recommend enhancements to the current tile-based approach.

### Hero & Landing Mobile
- **D-04:** The hero section already works on desktop. The task is to **audit the current mobile rendering via Playwright**, identify what's broken or suboptimal, and fix it — not redesign from scratch.
- **D-05:** Researcher should also investigate mobile hero best practices for creative/music industry sites to inform fixes.
- **D-06:** No specific known issues yet — **full Playwright audit required** to capture the current state and identify all problems.

### Player Bar + Tab Bar Coexistence
- **D-07:** How the player bar and bottom tab bar coexist on mobile is a **research-driven decision**. Researcher must investigate how music apps (Spotify, SoundCloud, Tidal, Apple Music) handle player + navigation coexistence and recommend the best pattern for this site.
- **D-08:** The Phase 07.1 mobile waveform work (replacing 2px progress bar with real waveform) must be preserved — this phase builds on top of that, not replaces it.

### Page-by-Page Audit
- **D-09:** **Audit every public page** on mobile — Home, Beats (catalog + detail), Services, Portfolio, Artists, Blog, Contact, Login, and all detail/sub-pages. Fix everything found, not just critical issues.
- **D-10:** Mobile viewport breakpoints should be **researched based on current mobile market data** for a music/creative studio audience. Researcher determines the standard test set.
- **D-11:** Use **Playwright for all mobile auditing** — screenshot every page at determined viewports so the AI can verify visual output (per user's standing preference for Playwright verification during dev).

### Claude's Discretion
- Specific CSS/layout fix approaches for each page's breakage
- Animation behavior adjustments for mobile (reduce motion, simplify transitions if needed)
- Safe area inset handling details (already partially implemented with `env(safe-area-inset-bottom)`)
- Exact pixel dimensions for mobile-specific adjustments

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `BottomTabBar` + `MobileNavOverlay`: Already functional mobile nav — needs restructuring, not building from scratch
- `Tile` component: Used in both desktop and mobile nav — consistent tile-based pattern
- `CartIcon` component: Standalone cart icon that can be repositioned
- `Waveform` component: Canvas-based waveform from Phase 07.1 — already handles mobile sizing

### Established Patterns
- Mobile-first breakpoint: `md:hidden` for mobile-only, `hidden md:flex` for desktop-only
- Bottom spacing: `pb-16 md:pb-0` on main content for tab bar clearance
- Safe area: `pb-[env(safe-area-inset-bottom)]` on tab bar for notched phones
- Animation: Framer Motion throughout — `AnimatePresence` for overlay entrance/exit
- Body scroll lock: Manual `document.body.style.overflow` toggle in overlay

### Integration Points
- `PublicLayout` renders `TileNav`, `main`, `BottomTabBar` — any structural changes happen here
- Player bar is rendered at app level (above layout) — independent of page routing
- `useSidebar` context controls desktop sidebar state — not used on mobile

</code_context>
