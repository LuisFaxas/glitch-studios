## Summary

This phase makes every existing feature functional end-to-end. The codebase has 37 plans worth of features built, but several integration issues prevent the site from working in a browser. The primary technical issues are: (1) potential postgres driver leakage into client bundles (no `server-only` guard on `db.ts`), (2) auth 500 errors on admin routes, (3) missing environment variables (RESEND_API_KEY, ADMIN_EMAIL, CRON_SECRET), and (4) the seed script uses the old `@neondatabase/serverless` driver instead of the current `postgres` driver.

Investigation of the running dev server shows all routes return 200 server-side. The postgres client bundle leak issue needs careful diagnosis -- the server-component-as-slot pattern used for `WidgetLatestPost` is architecturally correct (server component renders on server, passed as ReactNode to client `TileNav`). The issue may manifest as hydration errors in the browser rather than server errors, or may have been partially resolved in a previous fix. Regardless, adding `server-only` to `db.ts` and `auth.ts` is the correct hardening measure.

**Primary recommendation:** Fix in dependency order: (1) add `server-only` to `db.ts`/`auth.ts` and diagnose any resulting build errors to find actual client-bundle leaks, (2) debug auth 500, (3) add missing env vars, (4) rewrite seed script with postgres-js driver, (5) smoke-test all routes.

## Architecture Patterns

### Pattern 1: Server-Only Module Guard
**What:** Add `import "server-only"` at the top of any module that uses Node.js APIs (postgres, crypto, fs) or contains secrets.
**When to use:** Any file that should never be included in client bundles.
**Why it matters:** Without this guard, accidental import chains silently bundle server code into the client. The build succeeds but the browser crashes at runtime with "module not found" errors for Node.js builtins.

**Files that need `server-only`:**
```
src/lib/db.ts              ← postgres driver
src/lib/auth.ts            ← imports db.ts, uses crypto
src/lib/r2.ts              ← AWS S3 client
src/lib/permissions.ts     ← imports auth.ts and db.ts
```

**Example:**
```typescript
// src/lib/db.ts
import "server-only"
import postgres from "postgres"
import { drizzle } from "drizzle-orm/postgres-js"
import * as schema from "@/db/schema"

const databaseUrl = process.env.DATABASE_URL || "postgresql://build:build@localhost/build"
const client = postgres(databaseUrl)
export const db = drizzle(client, { schema })
```

### Pattern 2: Seed Script with postgres-js Driver
**What:** The existing `src/db/seed.ts` uses `@neondatabase/serverless` which was replaced with `postgres` in Phase 1.3. Must rewrite to use the same driver as `db.ts`.
**Approach:** Import `db` from `@/lib/db` directly (same as all other server code) instead of creating a separate connection.

**Example:**
```typescript
// src/db/seed.ts
import { db } from "@/lib/db"
import * as schema from "./schema"

async function seed() {
  console.log("Seeding database...")
  // ... insert operations using db instance
}
```

**Important:** The seed script runs via `tsx src/db/seed.ts`. The `server-only` import in `db.ts` will cause issues when running outside Next.js. Solution: either (a) conditionally import `server-only` (check for Next.js runtime), or (b) create the postgres connection inline in the seed script. Option (b) is simpler and more reliable.

### Pattern 3: Admin Seed via Direct DB Insert
**What:** The existing `src/db/seed-admin.ts` tries to call the Better Auth sign-up API endpoint, which requires the dev server to be running. For a standalone seed script, use direct database insertion instead.
**Approach:** Use Better Auth's `hashPassword` utility to hash the password, then insert directly into the `user` table with the correct role.

### Pattern 4: Smoke Testing with curl/fetch Scripts
**What:** Verify all routes render without errors by hitting each URL and checking for 200 status + absence of error indicators.
**Why not Playwright:** Playwright requires browser binaries (~400MB), adds complexity for what is essentially "does the page load." For this stabilization phase, a simple script is sufficient.
**Approach:** A TypeScript script that fetches each route, checks status code, and optionally checks for hydration error markers in the HTML response.

```typescript
// scripts/smoke-test.ts
const routes = [
  "/", "/beats", "/services", "/book", "/portfolio",
  "/artists", "/blog", "/contact", "/login", "/checkout",
  "/dashboard", "/dashboard/purchases", "/dashboard/bookings",
  "/admin", "/admin/beats", "/admin/blog", "/admin/bookings",
  "/admin/services", "/admin/clients", "/admin/inbox",
  "/admin/newsletter", "/admin/media", "/admin/team",
  "/admin/testimonials", "/admin/settings", "/admin/roles",
]

for (const route of routes) {
  const res = await fetch(`http://localhost:3000${route}`)
  const ok = res.status === 200 || res.status === 302 // 302 for auth redirects
  console.log(`${ok ? "PASS" : "FAIL"} ${res.status} ${route}`)
}
```

### Anti-Patterns to Avoid
- **Do not use `"use server"` to fix bundle leaks:** The `"use server"` directive marks functions as Server Actions (callable from client). It does NOT prevent the module from being bundled. Use `import "server-only"` instead.
- **Do not conditionally import postgres:** Dynamic `import()` for the database client adds unnecessary complexity. The `server-only` guard catches the problem at build time.
- **Do not create separate DB connections in each seed script:** Use the shared `db` instance from `@/lib/db` when running within Next.js context, or create a standalone connection only for CLI scripts.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Server/client boundary enforcement | Manual import tracking | `server-only` package | One-line addition, catches violations at build time with clear error messages |
| Password hashing for seed user | Custom bcrypt setup | `better-auth` internal utilities or direct `bcryptjs` | Better Auth stores hashed passwords; must match their format |
| Email testing | Custom email mock | Resend test domain `onboarding@resend.dev` | Works with real Resend API, just sends from test domain |
| CRON_SECRET generation | Manual string | `openssl rand -base64 32` | Cryptographically secure, standard practice |

## Common Pitfalls

### Pitfall 1: server-only Breaks Standalone Scripts
**What goes wrong:** Adding `import "server-only"` to `db.ts` causes `tsx src/db/seed.ts` to fail because `server-only` throws outside Next.js runtime.
**Why it happens:** The `server-only` package checks for React server component context. Standalone scripts (tsx, node) don't have this context.
**How to avoid:** Seed scripts should create their own postgres connection instead of importing from `@/lib/db`. This also avoids the dummy URL fallback in db.ts.
**Warning signs:** "server-only can only be used in Server Components" error when running scripts.

### Pitfall 2: Better Auth Admin Seed Requires Password Hashing
**What goes wrong:** Inserting a user directly into the database without properly hashing the password means login fails.
**Why it happens:** Better Auth uses its own password hashing (bcrypt-based). Raw passwords in the DB won't match.
**How to avoid:** Either (a) use Better Auth's API to create the user (requires running server), or (b) use the same hashing function Better Auth uses internally. Better Auth uses `bcryptjs` -- hash with `bcrypt.hash(password, 10)`.
**Warning signs:** Login returns "Invalid credentials" despite correct email/password.

### Pitfall 3: Resend Test Domain Limitations
**What goes wrong:** Emails sent from `onboarding@resend.dev` can only be delivered to the verified account email, not to arbitrary addresses.
**Why it happens:** Resend's test domain has delivery restrictions to prevent abuse.
**How to avoid:** For testing, ensure ADMIN_EMAIL matches the Resend account's verified email address. All test emails (contact notifications, booking confirmations) will route to this address.
**Warning signs:** Emails silently fail to deliver to non-verified addresses.

### Pitfall 4: Stripe Webhook Testing in Dev
**What goes wrong:** Stripe webhooks don't fire in local development because Stripe can't reach localhost.
**Why it happens:** Webhooks are HTTP callbacks to a public URL.
**How to avoid:** Use `stripe listen --forward-to localhost:3000/api/webhooks/stripe` (Stripe CLI). This forwards webhook events to local dev server. Alternatively, test the webhook handler directly with mock payloads.
**Warning signs:** Beat purchase completes in Stripe but no receipt email is sent, no download links appear.

### Pitfall 5: Empty Database Crashes
**What goes wrong:** Pages that query the database crash or show blank when no data exists.
**Why it happens:** Missing null checks, .map() on undefined, or accessing properties of empty query results.
**How to avoid:** Test with empty database FIRST (D-08), then with seeded data. The home page already uses Promise.allSettled for resilience, but other pages may not.
**Warning signs:** Blank pages or "Cannot read properties of undefined" errors on routes with no data.

## Open Questions

1. **What exactly causes the "client-bundle crash"?**
   - What we know: Server-side rendering works (all routes return 200). The `server-only` package is not installed. The server-component-as-slot pattern for WidgetLatestPost is architecturally correct.
   - What's unclear: Whether the issue is hydration errors in the browser, a different import chain pulling postgres into client code, or if the issue was partially resolved. Need to test in a browser.
   - Recommendation: Add `server-only` to db.ts/auth.ts first. If the build breaks, the error messages will pinpoint exactly which client component is importing server code. If the build succeeds, test in browser to verify hydration works.

2. **Auth 500 root cause**
   - What we know: Better Auth is configured with Drizzle adapter, admin plugin, owner/admin roles. The API route exists at `/api/auth/[...all]/route.ts`. Middleware redirects unauthenticated users to /login.
   - What's unclear: The specific 500 error. Could be missing database tables (Better Auth needs its own tables), schema mismatch, or a configuration issue.
   - Recommendation: Check Better Auth tables exist in DB. Run `drizzle-kit push` to ensure schema is synced. Check server logs when hitting `/api/auth/sign-in/email`.

3. **Better Auth password storage format**
   - What we know: Better Auth uses its own internal password hashing for the credential provider.
   - What's unclear: The exact hashing algorithm and table structure (does it store passwords in the `user` table or `account` table?).
   - Recommendation: Check the Better Auth docs or inspect the database schema to determine where passwords are stored and how to hash them for direct seeding.
