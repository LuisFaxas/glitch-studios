# Phase 17 — UI Design Contract
## BPR Medal UI + Methodology Page

> Visual and interaction contract for the `<BPRMedal>` component (surfacing on every GlitchTech review card + detail scorecard) and the public `/tech/methodology` page. Inherits the GlitchTech monochrome Cyberpunk Metro language from Phase 07.4 and the review detail architecture from Phase 07.6. Generated by gsd-ui-researcher under AUTO mode.

> **Brand note:** The tech sub-brand is **GlitchTech** (matches `glitchtech.io`). Earlier roadmap/STATE references spelled it "GlitchTek" — this is stale. Phase 17 copy uses **GlitchTech**. Do not do a bulk rename of historical docs, but do not repeat the old spelling in new surfaces.

## Spacing Scale

All multiples of 4, inherited from 07.4 / 07.6.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | Gap between medal score number and "%" unit; icon-to-label gap inside medal |
| sm | 8px | Vertical gap between medal tier label and score; gap between medal and Rubric v1.1 badge |
| md | 16px | Padding inside the medal (`px-4 py-3`); gap between methodology subsections within a section |
| lg | 24px | Page gutter on methodology (`px-6`); gap between methodology sections |
| xl | 32px | Vertical rhythm between `<h2>` section headers on methodology |
| 2xl | 48px | Methodology page top padding under nav; gap between hero section and first methodology section |
| 3xl | 64px | Bottom padding before newsletter/footer on methodology page |

**Exceptions:**
- **Medal in review card (compact):** reduces internal padding to `px-3 py-2` to fit the card chip layout alongside rating + category

- **Methodology formula code block:** `p-6` inner padding (24px), slightly more generous than default to give the geomean expression breathing room

## Color

**Strict monochrome. Locked by MEDAL-01 requirement.** No accent colors introduced in Phase 17. The four medal tiers are distinguished **entirely by fill and border treatment** — Platinum is the brightest (filled, inverted); each subsequent tier is darker/lighter-weight, culminating in Bronze as a dashed outline.

### Base palette (inherited)

| Role | Value | Usage in this phase |
|------|-------|---------------------|
| Dominant (60%) | `#000000` | Methodology page background, scorecard outer background |
| Secondary (30%) | `#111111` | Methodology section cards, scorecard card surface, BPR formula code block, medal Silver/Bronze inner fill |
| Tertiary surface | `#1a1a1a` | Hover state on methodology anchor links, accordion row hover (if used) |
| Border (default) | `#222222` | Methodology section card borders, formula code block border |
| Border (emphasis) | `#444444` | Methodology section card border on hover; Bronze medal dashed border |
| Text (primary) | `#f5f5f0` | Methodology headings, body copy, tooltip text, Silver medal text |
| Text (label/muted) | `#888888` | Methodology metadata, Gold medal tier label (when on `#888` fill for visibility), Bronze medal score number, tooltip secondary line |
| Text (tertiary) | `#555555` | Table cell non-emphasis text, methodology breadcrumb separators, Silver medal border |
| Text (inverted) | `#000000` | Text on Platinum and Gold medal fills |

### Medal tier palette (MEDAL-01, locked)

| Tier | Threshold | Background | Border | Text Color | Tier Label Color | Visual Hierarchy Intent |
|------|-----------|------------|--------|------------|------------------|--------------------------|
| **Platinum** | `score ≥ 90` | `bg-[#f5f5f0]` (off-white) | `border border-[#f5f5f0]` | `text-[#000]` | `text-[#000]` | Brightest; inverted fill; reads as a "trophy chip" |
| **Gold** | `score ≥ 80` | `bg-[#888]` (mid gray) | `border border-[#888]` | `text-[#000]` | `text-[#000]` | Mid-tone filled; still solid but visibly dimmer than Platinum |
| **Silver** | `score ≥ 70` | `bg-transparent` | `border border-[#555]` (outlined) | `text-[#f5f5f0]` | `text-[#888]` | Outlined only; no fill — reads as "qualifier" tier |
| **Bronze** | `score ≥ 60` | `bg-transparent` | `border border-dashed border-[#444]` (dashed outline) | `text-[#888]` | `text-[#555]` | Dashed outline, mutest; reads as "barely earned" |
| _(null)_ | `score < 60` or fewer than 5 of 7 eligible disciplines scored | — | — | — | — | **No medal rendered.** Render the "Not enough data" placeholder instead (§MEDAL-03) |

**Accent reserved for (10% rule):**

The inverted `#f5f5f0` surface in this phase is reserved strictly for:
1. The **Platinum medal fill** (the brightest tier is this phase's visual "reward" accent)
2. The **primary CTA button** on the methodology page ("Back to Reviews" bottom-of-page link) — inherits the bordered-ghost → filled-on-hover pattern from 07.6
3. The **Rubric v1.1 badge** bordered outline (`border border-[#f5f5f0]` with transparent background)

No other element uses the inverted surface. Specifically: Gold is **not** an accent — it's a mid-gray `#888` fill. Silver and Bronze do **not** use the inverted surface at all.

**Destructive:** Not applicable. This phase has no destructive actions.

## The `<BPRMedal>` Component — Visual Contract

### Full variant (detail scorecard)

```
┌──────────────────────────┐
│   88%                    │   ← score number, font-mono 24px bold
│   ──────                 │
│   GOLD                   │   ← tier label, font-mono 11px bold uppercase
└──────────────────────────┘
   ↑ background = tier fill
   ↑ border = tier border
   ↑ min-width 120px, px-4 py-3
```

- **Platinum** `bg-[#f5f5f0] border-[#f5f5f0] text-[#000]`
- **Gold** `bg-[#888] border-[#888] text-[#000]`
- **Silver** `bg-transparent border-[#555] text-[#f5f5f0]` with tier label `text-[#888]`
- **Bronze** `bg-transparent border-dashed border-[#444] text-[#888]` with tier label `text-[#555]`

Score number always shows `Math.round(score)` (no decimals). The `%` unit is a sibling span at opacity 0.8 so it tonally recedes. Whole-number rounding is intentional — a medal is a category summary, not a precise metric; the exact value is available on the methodology page's explanation of how BPR is computed and in the console/admin surfaces.

### Compact variant (review card)

```
┌─────────────┐
│  GOLD  88%  │   ← single row, tier label + score
└─────────────┘
   ↑ px-2 py-1, h-6 min
```

Tier label and score inline; no separator line. Same fill/border rules as full variant.

### Tooltip (hover, desktop)

**Trigger:** Hovering anywhere on the medal.
**Content (single paragraph):**
> Based on **X of 7** eligible disciplines. Click for methodology.

Where `X` = `disciplineCount` prop. If `X === 7`, the tooltip reads "Based on all 7 eligible disciplines." (not "7 of 7" — reads better).

**Tooltip styling:** `bg-[#111] border border-[#222] text-[#f5f5f0] font-sans text-[13px] px-3 py-2` — matches Phase 07.6 benchmark tooltip pattern.

### Mobile interaction (no hover)

**Decision (AUTO):** On mobile, the tooltip is **not shown on tap**. Instead, the medal is always wrapped in a `<Link href="/tech/methodology#bpr">` (when `asLink` prop is `true`, the default), and the click/tap navigates directly to the methodology page at the BPR section. This is simpler than a two-tap-to-open-tooltip pattern and better for SEO (the methodology link is discoverable in the DOM).

**Accessibility:** The medal's `aria-label` always reads `"{tier} medal: {score}%. Based on {X} of 7 eligible disciplines. See methodology."` — screen readers get the full disclosure regardless of hover state.

### Not-enough-data placeholder (MEDAL-03)

Rendered in place of `<BPRMedal>` **only on the review detail scorecard** when `bprScore === null` AND `bprDisciplineCount < 5`. Review cards simply omit the medal entirely (fall back to star rating — see Modified components above).

```
┌──────────────────────────┐
│   NOT ENOUGH DATA        │   ← font-mono 11px bold uppercase
│   2 of 7 disciplines     │   ← font-sans 13px, text-[#888]
└──────────────────────────┘
   ↑ border border-dashed border-[#333]
   ↑ bg-transparent
   ↑ text-[#888] for primary line
```

Clicking the placeholder navigates to `/tech/methodology#exclusion-policy` (not `#bpr`, because the user needs to understand why there's no medal). The entire chip is a `<Link>`.

### Motion & interaction

- **No hover animation on the medal itself** beyond the tooltip fade-in (tooltip uses shadcn default 150ms fade + 4px translate from Radix).
- The medal does **not** glitch on hover. Reason: the medal communicates earned rank — glitching it would undercut the "stable verdict" semantics. This is an explicit exception to the "every interactive element gets glitch" default and is documented here.
- Headings on the methodology page **do** follow the site-wide hover-only RGB-split glitch rule via `<GlitchHeading>` (MEMORY: hover-glitch on all headers). No auto-running animations on headings.

## Review Card Integration

Modify `ReviewCard` to add the compact medal chip **in place of** the existing star + rating row **when `review.bprScore !== null`**.

Existing row:
```
⭐ 8.4 / 10  ·  LAPTOPS
```

New row (with BPR score):
```
[GOLD 88%]  LAPTOPS
```

New row (no BPR score — fallback):
```
⭐ 8.4 / 10  ·  LAPTOPS
```

Rationale: A review card is a discovery surface — we want a single clear signal (the medal) rather than two competing scores. The overall 1–10 rating is reached by clicking into the review. When no BPR score is available we fall back to the star rating so the card isn't visually broken.

**Compact medal `showTooltip={false}`** on cards — the card itself is already a link, and a nested tooltip/link pair creates awkward hover states. The tooltip is reserved for the detail-page scorecard medal where it's the primary interactive element.

## Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| Medal contrast (Platinum: `#000` on `#f5f5f0`) | ~19:1 — WCAG AAA for all text sizes |
| Medal contrast (Gold: `#000` on `#888`) | ~5.7:1 — WCAG AA for normal text; AAA for large text (tier label is small at 11px but bold which bumps effective ratio requirement per WCAG) |
| Medal contrast (Silver: `#f5f5f0` on transparent over `#111` page bg) | ~15:1 — WCAG AAA |
| Medal contrast (Bronze: `#888` on transparent over `#111` page bg) | ~4.6:1 — WCAG AA for large text. The Bronze tier label `#555` on `#111` is **~3.1:1 — below AA**. **Accepted trade-off** because Bronze is intentionally the "barely earned" tier and the visual hierarchy demands the muted label. `aria-label` provides full accessible disclosure regardless. This exception is logged here rather than silently. |
| Colorblind safety | All four tiers distinguished by **fill pattern + border style**, never by hue. Platinum = solid bright fill; Gold = solid mid fill; Silver = solid border; Bronze = dashed border. Works in grayscale, deuteranopia, and tritanopia. |
| Keyboard navigation | Medal is a `<Link>`; receives focus via Tab; `focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-[#f5f5f0]` (white outline matches site convention). Tooltip opens on focus (Radix default). |
| Reduced motion | Medal has no motion. Tooltip fade respects `prefers-reduced-motion` via Radix defaults. `<GlitchHeading>` on methodology page respects reduced-motion (existing behavior). |
| Screen reader | Medal `aria-label` declares tier + score + discipline count + CTA. Rubric badge declares version + CTA. Jump-to anchor chips have discoverable `href` targets. |

### `getMethodologyData()` return shape (new query)

```ts
export interface MethodologyData {
  disciplines: Array<{
    slug: BenchmarkDiscipline
    name: string                     // human-readable: "CPU", "GPU", "LLM", …
    description: string              // static constant (copywriting contract §Disciplines)
    bprEligible: boolean
  }>
  currentRubric: {
    version: string                  // "1.1"
    publishedAt: Date
    testCount: number                // 43
    disciplineCount: number          // 13
    bprEligibleCount: number         // 7
  }
  rubricChangelog: Array<{
    version: string
    publishedAt: Date
    highlights: string[]             // bullets
  }>
  medalThresholds: Array<{
    tier: BprTier
    minScore: number                 // 60 / 70 / 80 / 90
    maxScore: number | null          // 69 / 79 / 89 / null (for platinum which is unbounded)
    description: string              // copywriting contract §Thresholds
  }>
}
```

Discipline names/descriptions are **static constants** on the server (`src/lib/tech/methodology-copy.ts`) — not DB-driven. Only version/publishedAt/counts come from the DB. This is a deliberate simplification: the methodology copy rarely changes, and when it does it's a code PR (content-owned by the project, not the CMS), matching the STATE.md guidance that "Methodology content is DB-driven via Tiptap" applies to the **narrative** (prose) portion only. If Josh wants long-form methodology editing later, that becomes a v1.2 scope item, not Phase 17.
