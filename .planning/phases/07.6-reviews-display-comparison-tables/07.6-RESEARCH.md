### Locked Decisions

**Related Reviews:**
- **D-01** — Carousel shows **3 most recently published reviews in the same category** as the current review, excluding self. Sort by `publishedAt DESC`. Status filter: `published` only.
- **D-02** — If fewer than 3 published reviews exist in the category, **widen to the parent category node** and refill until 3 are found. If still fewer than 3, render with however many we have (do not hide).

**Comparison Benchmark Charts:**
- **D-03** — On `/tech/compare`, render the first 3 benchmark charts eagerly; subsequent charts mount via `IntersectionObserver` when scrolled into view.
- **D-04** — Recharts is dynamically imported with SSR disabled on all compare-tool components: `const BarChart = dynamic(() => import("recharts").then(m => m.BarChart), { ssr: false })`. Applied to `BarChart`, `Bar`, `XAxis`, `YAxis`, `CartesianGrid`, `ResponsiveContainer`, `LabelList`. Review detail and reviews list pages never import Recharts — comparison-only dependency.

**HTML Sanitization:**
- **D-05** — Sanitize `review.body_html` server-side with `isomorphic-dompurify` before rendering in the RSC. Allowlist: headings (`h1`–`h4`), text (`p`, `strong`, `em`, `u`, `br`), lists (`ul`, `ol`, `li`), links (`a` — force `rel="noopener noreferrer"` + `target="_blank"` for external), media (`img`), blocks (`blockquote`, `code`, `pre`, `hr`). Forbidden: `script`, `style`, `iframe`, `object`, `embed`, event handlers, inline `style` attrs.

**Homepage Wire-Up:**
- **D-06** — Migrate `tech-featured-reviews-carousel.tsx` (latest 3 published reviews), `tech-category-tiles.tsx` (top-level categories with product + review counts), and `tech-benchmark-spotlight.tsx` (highest Geekbench 6 Multi score across published products; hide section if no published reviews) from hardcoded placeholders to real server-fetched data.
- **D-06a** — Migrate card rating display from `/5` to `/10` to match 07.5 schema (4 dimensions × 1–10 plus overall).

**Category URL Structure:**
- **D-07** — Category browse URL = `/tech/categories/[slug]` with `[slug]` being the leaf category slug only, globally unique across `tech_categories`. Breadcrumb walks `parent_id` chain at render time.
- **D-07a** — `/tech/categories` (index) lists only level 1 top-level categories with aggregate counts.

## Summary

Phase 07.6 is a **public-facing RSC rendering phase** with four new pages, one new dynamic route, three homepage component migrations, and one new npm dependency (Recharts). The `tech_*` schema from 07.5 is complete and unchanged; all work is read-side. Database access patterns are already demonstrated in the admin action files — public pages reuse the same Drizzle select-with-leftJoin style, just with `status='published'` filters and without `requirePermission()` gates.

The **only net-new stack piece** is Recharts for benchmark bar charts on `/tech/compare` (dynamic-imported, SSR-disabled per D-04). The second add, `isomorphic-dompurify`, is for defense-in-depth sanitization of the Tiptap-produced `body_html` (per D-05). Both packages support React 19 and Next.js 16 App Router cleanly.

The largest architectural concern is **single-sourcing the `ReviewCard`**: it appears in 4 surfaces (reviews list, related-reviews carousel, category browse grid, homepage featured carousel after D-06 migration). Building it first and migrating the existing hardcoded carousel to consume it — rather than duplicating the JSX — is critical to keep the monochrome visual grammar tight.

**Primary recommendation:** Install Recharts 3.8.1 (not 2.x as CLAUDE.md suggests) and isomorphic-dompurify 3.9.0; co-locate data fetching in `src/lib/tech/queries.ts` to share between pages and the homepage component migration; build `ReviewCard` first; wire review detail page second; parallelize reviews-list + category browse; finish with compare tool and homepage migration.

### CLAUDE.md divergence

CLAUDE.md says "Recharts 2.x". This research recommends **3.8.1** because:

1. 3.x is the actively maintained line; 2.15.x gets only security patches.
2. React 19 is first-class in peerDependencies (Issue #4558 resolution in 2.15.0; remains in 3.x).
3. The UI-SPEC's chart needs — `BarChart`, `Bar`, `XAxis`, `YAxis`, `CartesianGrid`, `ResponsiveContainer`, `LabelList`, `isAnimationActive` — are all present and API-compatible between 2.x and 3.x for these components. No migration risk.

**If the planner wants strict CLAUDE.md adherence:** Use `recharts@2.15.x` — API identical for the subset we use. Update this research note and CLAUDE.md together.

### Pattern 1: Public Drizzle Query Module

**What:** Centralize all public `tech_*` reads in `src/lib/tech/queries.ts`. RSC pages and the homepage migration all import from one file — one change, many surfaces update. Matches the `src/lib/booking/*` pattern already in the codebase.

**When to use:** Shared queries (same data accessed from multiple pages — e.g. `getLatestPublishedReviews(limit)` used by `/tech/reviews`, `/tech/`, and the homepage `tech-featured-reviews-carousel`).

```ts
// src/lib/tech/queries.ts
import "server-only"
import { db } from "@/lib/db"
import { techReviews, techProducts, techCategories, mediaAssets, user } from "@/db/schema"
import { eq, and, desc, inArray, ilike, or, lt, sql } from "drizzle-orm"

export interface PublicReviewCard {
  id: string
  slug: string
  title: string
  overallRating: number   // computed: override OR average
  heroImageUrl: string
  heroImageAlt: string | null
  productName: string
  categoryName: string | null
  publishedAt: Date
  excerpt: string         // first 140 chars of verdict
}

export async function getLatestPublishedReviews(limit: number): Promise<PublicReviewCard[]> {
  const rows = await db
    .select({
      id: techReviews.id,
      slug: techReviews.slug,
      title: techReviews.title,
      verdict: techReviews.verdict,
      ratingPerformance: techReviews.ratingPerformance,
      ratingBuild: techReviews.ratingBuild,
      ratingValue: techReviews.ratingValue,
      ratingDesign: techReviews.ratingDesign,
      overallOverride: techReviews.overallOverride,
      heroImageUrl: mediaAssets.url,
      heroImageAlt: mediaAssets.alt,
      productName: techProducts.name,
      categoryName: techCategories.name,
      publishedAt: techReviews.publishedAt,
    })
    .from(techReviews)
    .innerJoin(techProducts, eq(techReviews.productId, techProducts.id))
    .leftJoin(techCategories, eq(techProducts.categoryId, techCategories.id))
    .innerJoin(mediaAssets, eq(techReviews.heroImageId, mediaAssets.id))
    .where(eq(techReviews.status, "published"))
    .orderBy(desc(techReviews.publishedAt))
    .limit(limit)

  return rows.map((r) => ({
    id: r.id,
    slug: r.slug,
    title: r.title,
    overallRating: r.overallOverride !== null
      ? Number(r.overallOverride)
      : (r.ratingPerformance + r.ratingBuild + r.ratingValue + r.ratingDesign) / 4,
    heroImageUrl: r.heroImageUrl,
    heroImageAlt: r.heroImageAlt,
    productName: r.productName,
    categoryName: r.categoryName,
    publishedAt: r.publishedAt!,  // guaranteed non-null because status=published
    excerpt: r.verdict.slice(0, 140),
  }))
}
```

### Pattern 2: Cursor Pagination on Reviews List

**What:** Cursor-based "Load more" pagination keyed on `publishedAt` (ties broken by `id`). Avoids the `OFFSET N` slowdown + duplicate-row problem when new reviews publish mid-scroll.

**When to use:** The `/tech/reviews` list when filters or sorts are stable. For sort modes other than `latest` (e.g. `rating` or `product-name`), compound cursor keys are needed — acceptable complexity for sort='rating' since values collide more often.

```ts
// Cursor = encoded { publishedAt, id }. Decode, then:
const where = and(
  eq(techReviews.status, "published"),
  cursor
    ? or(
        lt(techReviews.publishedAt, cursor.publishedAt),
        and(eq(techReviews.publishedAt, cursor.publishedAt), lt(techReviews.id, cursor.id)),
      )
    : undefined,
  filters.categoryId ? eq(techProducts.categoryId, filters.categoryId) : undefined,
  filters.q
    ? or(
        ilike(techReviews.title, `%${filters.q}%`),
        ilike(techProducts.name, `%${filters.q}%`),
      )
    : undefined,
)
```

**Anti-pattern:** `OFFSET page * perPage` — every subsequent page re-reads all prior rows. Fine for admin lists (page 1 is nearly always used), poor for public "Load more" UX.

### Pattern 6: Server-Side HTML Sanitization

```ts
// src/lib/tech/sanitize.ts
import "server-only"
import DOMPurify from "isomorphic-dompurify"

const REVIEW_BODY_CONFIG: DOMPurify.Config = {
  ALLOWED_TAGS: [
    "h1", "h2", "h3", "h4",
    "p", "strong", "em", "u", "br",
    "ul", "ol", "li",
    "a", "img",
    "blockquote", "code", "pre", "hr",
  ],
  ALLOWED_ATTR: ["href", "src", "alt", "title", "width", "height"],
  FORBID_TAGS: ["script", "style", "iframe", "object", "embed"],
  FORBID_ATTR: ["style", "onerror", "onclick", "onload", "onmouseover"],
}

// Add target + rel to external links after sanitize (hook)
DOMPurify.addHook("afterSanitizeAttributes", (node) => {
  if (node.tagName === "A" && node.getAttribute("href")?.startsWith("http")) {
    node.setAttribute("target", "_blank")
    node.setAttribute("rel", "noopener noreferrer")
  }
})

export function sanitizeReviewBody(html: string): string {
  return DOMPurify.sanitize(html, REVIEW_BODY_CONFIG)
}
```

Called once per RSC request in the review detail page — output fed into `<div className="review-body prose prose-invert ..." dangerouslySetInnerHTML={{ __html: clean }} />`.

**Memory note:** `isomorphic-dompurify` internally holds a jsdom Window that can accumulate DOM state across calls. For short-lived RSC requests this is not an issue. If a long-running background job ever reuses this util, add `DOMPurify.clearWindow()` between calls.

```tsx
  const [visible, setVisible] = useState(false)

  useEffect(() => {
    if (!ref.current) return
    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting) {
          setVisible(true)
          observer.disconnect()
        }
      },
      { rootMargin: "200px" },  // start loading 200px before viewport
    )
    observer.observe(ref.current)
    return () => observer.disconnect()
  }, [])

  return (
    <div ref={ref} style={{ minHeight: 320 }}>
      {visible ? children : null}
    </div>
  )
}
```

Usage: first 3 charts render directly; charts 4+ wrap in `<LazyChart>`.

### Anti-Patterns to Avoid

- **Hydrating Tiptap on public pages.** Even read-only, it's ~50kB JS for a static HTML render. Use `dangerouslySetInnerHTML` with DOMPurify.
- **Importing Recharts statically in any component.** Even the `ComparisonBenchmarkChart` module itself must avoid top-level `import { BarChart } from "recharts"` — use the dynamic pattern. Static import would leak Recharts into any bundle that reaches this module.
- **Putting `generateMetadata` queries inside the page component's `data` call.** Next.js already dedupes identical fetches, but raw `db.select()` calls inside Drizzle aren't deduped the same way. Either use Drizzle's `.findFirst()` with stable inputs (Next uses `React.cache` dedup for identical async function calls within a render) or extract the query and wrap it with `React.cache` explicitly.
- **`OFFSET`-based pagination for public lists.** See Pattern 2.
- **Mutations in RSC data-fetching functions.** All 07.6 pages are read-only. Do not `revalidatePath()` from a GET-only page.
- **Re-defining the card JSX in the homepage carousel after D-06.** The whole point of D-06 is single-source. Migrate `tech-featured-reviews-carousel.tsx` to consume `<ReviewCard />`, do not duplicate markup.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|

| Horizontal bar chart | Manual SVG `<rect>` + `<text>` | Recharts `BarChart` + `<Bar layout="vertical">` | Manual SVG means re-implementing responsiveness, axis ticks, label positioning, and reduced-motion toggles. Recharts handles all of it. |

## Common Pitfalls

### Pitfall 1: Recharts in the `/tech/reviews` bundle

**What goes wrong:** A developer adds a small Recharts component "just for one chart" on the review detail page. Static import leaks Recharts into the review route bundle → LCP degrades from 2.0s to 3.5s.
**Why it happens:** Habit. `import { BarChart } from "recharts"` is one line; dynamic import is 7 lines.
**How to avoid:** UI-SPEC and D-04 both say "comparison-only dependency". Keep the dynamic pattern factored into `ComparisonBenchmarkChart` so no one copies a static import.
**Warning signs:** After `pnpm build`, `.next/analyze` or Next.js build output shows Recharts in any route other than `/tech/compare`.

### Pitfall 2: Forgetting to sanitize, or sanitizing client-side

**What goes wrong:** `<div dangerouslySetInnerHTML={{ __html: review.bodyHtml }} />` without DOMPurify → future Tiptap vulnerability or admin XSS attack reaches users.
**Why it happens:** D-05 is "defense-in-depth" so it feels optional. It is not.
**How to avoid:** Every `dangerouslySetInnerHTML` involving review content must go through `sanitizeReviewBody()`. Add a `// eslint-disable-next-line` with a comment explaining why if ESLint flags it.
**Warning signs:** `git grep 'dangerouslySetInnerHTML.*bodyHtml'` returns matches not preceded by `sanitize` in the same function.

### Pitfall 3: Cursor pagination races with admin publish

**What goes wrong:** User scrolls "Load more" at t=0; admin publishes a new review at t=1; user clicks "Load more" — the new review shifts page boundaries and the user sees a duplicate of what they already saw.
**Why it happens:** Offset pagination uses position; new rows shift positions. Cursor pagination on `publishedAt` mostly avoids this, but ties on `publishedAt` (unlikely but possible for batch publishes) still need tie-breaking on `id`.
**How to avoid:** Cursor = `{publishedAt, id}` compound. `WHERE (publishedAt < cursor.publishedAt) OR (publishedAt = cursor.publishedAt AND id < cursor.id)`. Pattern 2 above.
**Warning signs:** Users report seeing "the same review twice" in dev testing.

### Pitfall 4: `generateStaticParams` fails at build time

**What goes wrong:** Vercel preview builds run with no DATABASE_URL connectivity; `generateStaticParams` throws; the entire route group fails to build.
**Why it happens:** Edge-case deploy environments (preview, CI).
**How to avoid:** Wrap in try/catch, return `[]` — matches blog page pattern. Next.js falls back to on-demand rendering, which is correct.
**Warning signs:** Red build logs referencing `[slug]` / `getStaticProps`-style errors.

### Pitfall 5: `useEmblaCarousel` imported inside RSC

**What goes wrong:** A Server Component imports `embla-carousel-react` (which is a client-only hook) → build fails with "You're importing a component that needs `useState`. It only works in a Client Component."
**Why it happens:** Mixing RSC page data fetching with Embla carousel markup.
**How to avoid:** Carousel components are **client components** (`"use client"`). RSC page fetches data, passes plain serializable data to `<RelatedReviewsCarousel reviews={...} />` — the client component uses Embla internally.
**Warning signs:** Build errors about `useCallback`/`useRef` in a non-client boundary.

### Pitfall 6: Missing Category Products in Homepage Tiles

**What goes wrong:** `tech-category-tiles.tsx` hardcodes 3 entries today; real data has 1 or 4+ top-level categories. Grid breaks.
**Why it happens:** Designer-first layouts with assumed counts.
**How to avoid:** Use `grid-cols-2 md:grid-cols-3` (current) but drop the `aspect-square` constraint if only 1 or 2 items, OR show a "more categories coming" placeholder. UI-SPEC says tiles should match existing pattern — keep the grid but degrade gracefully.

### Pitfall 7: Rating Conversion Drift (D-06a)

**What goes wrong:** The existing carousel shows `{rating} / 5`. Schema stores 1–10 integers (four dimensions) + optional 1.0–10.0 override. If migration misses a spot, a value of `8.4` displays as "8.4 / 5" — nonsense.
**Why it happens:** Partial migration. Card rating lives in 4 surfaces after 07.6.
**How to avoid:** Single-source `ReviewCard` — migrate the existing hardcoded carousel to consume it. Grep for `/ 5` and replace.
**Warning signs:** Any display of rating with denominator `5`. All denominators in 07.6 must be `10`.

### Pitfall 8: `isomorphic-dompurify` jsdom memory in serverless

**What goes wrong:** Under high load on Vercel (many concurrent requests), the internal jsdom Window accumulates DOM state across sanitize calls — progressive slowdown, eventual memory pressure.
**Why it happens:** Default behavior; documented by the library maintainer.
**How to avoid:** For RSC pages rendered per-request, the serverless lambda instance is typically fresh per invocation and GC'd. Not a real concern for this use case. If the project ever adds a long-running worker that sanitizes many reviews, call `DOMPurify.clearWindow()` periodically.
**Warning signs:** Vercel function memory approaching limit in `/tech/reviews/[slug]` logs.

### Pitfall 9: YouTube embed URL validation bypass

**What goes wrong:** Admin pastes `https://evil.com/steal` into the `videoUrl` field. The iframe renders that URL. Same-origin policy mitigates some attack surface, but not all.
**Why it happens:** No validation on save (07.5 schema has `videoUrl: text("video_url")` with no regex check).
**How to avoid:** Validate on **read** in 07.6. Accept only `youtube.com/watch?v=ID`, `youtu.be/ID`, `youtube.com/embed/ID`. Extract the video ID, render `https://www.youtube.com/embed/{id}`. Hide section if URL doesn't match allowlist.
**Warning signs:** Any iframe `src` that doesn't start with `https://www.youtube.com/embed/`.

### Pitfall 10: Winner-detection returns wrong direction

**What goes wrong:** A seed spec template has a field named "Weight (g)" — `spec-direction.ts` map keyed on lowercased `"weight"` catches grams correctly, but not `"weight (kg)"` if executor adds the unit to the field name.
**Why it happens:** Flexible admin field naming vs. rigid code-side map.
**How to avoid:** Winner-detection map should match by **field type hint** where possible (prefer matching the `unit` field from `techSpecFields.unit` — `kg`, `g`, `lb` → lower-is-better; `score`, `fps`, `pts` → higher-is-better). Fall back to field-name keyword match. Plan a task to audit the admin spec templates against the map before launch.
**Warning signs:** Comparison winner badges disagree with obvious truth (MacBook Pro Max shown as losing on "performance" against a cheaper laptop).

### Example 2: Reviews List Data Function (cursor pagination)

```ts
// src/lib/tech/queries.ts
export async function listPublishedReviews(filters: {
  categorySlug?: string
  sort: "latest" | "rating" | "name"
  q?: string
  cursor?: { publishedAt: Date; id: string }
  limit: number
}): Promise<{ reviews: PublicReviewCard[]; nextCursor: { publishedAt: Date; id: string } | null }> {
  const conditions = [eq(techReviews.status, "published")]

  if (filters.categorySlug) {
    // walk down category tree — find this slug and all descendants
    const categoryIds = await getCategoryDescendantIds(filters.categorySlug)
    conditions.push(inArray(techProducts.categoryId, categoryIds))
  }
  if (filters.q) {
    conditions.push(or(
      ilike(techReviews.title, `%${filters.q}%`),
      ilike(techProducts.name, `%${filters.q}%`),
    )!)
  }
  if (filters.cursor) {
    conditions.push(or(
      lt(techReviews.publishedAt, filters.cursor.publishedAt),
      and(
        eq(techReviews.publishedAt, filters.cursor.publishedAt),
        lt(techReviews.id, filters.cursor.id),
      ),
    )!)
  }

  const orderBy = filters.sort === "latest"
    ? [desc(techReviews.publishedAt), desc(techReviews.id)]
    : filters.sort === "rating"
      ? [desc(sql`COALESCE(${techReviews.overallOverride}, (${techReviews.ratingPerformance} + ${techReviews.ratingBuild} + ${techReviews.ratingValue} + ${techReviews.ratingDesign}) / 4.0)`), desc(techReviews.id)]
      : [asc(techProducts.name), asc(techReviews.id)]

  const rows = await db
    .select(/* ... card columns ... */)
    .from(techReviews)
    .innerJoin(techProducts, eq(techReviews.productId, techProducts.id))
    // ...joins
    .where(and(...conditions))
    .orderBy(...orderBy)
    .limit(filters.limit + 1)

  const hasMore = rows.length > filters.limit
  const trimmed = rows.slice(0, filters.limit)
  const nextCursor = hasMore
    ? { publishedAt: trimmed[trimmed.length - 1].publishedAt!, id: trimmed[trimmed.length - 1].id }
    : null

  return { reviews: trimmed.map(mapToCard), nextCursor }
}
```

### Example 3: Related Reviews with Parent-Category Widening (D-01, D-02)

```ts
export async function getRelatedReviews(
  currentReviewId: string,
  categoryId: string,
  parentCategoryId: string | null,
): Promise<PublicReviewCard[]> {
  // First try: same category
  let results = await db
    .select(/* card columns */)
    .from(techReviews)
    .innerJoin(techProducts, eq(techReviews.productId, techProducts.id))
    // ...joins
    .where(and(
      eq(techReviews.status, "published"),
      eq(techProducts.categoryId, categoryId),
      ne(techReviews.id, currentReviewId),
    ))
    .orderBy(desc(techReviews.publishedAt))
    .limit(3)

  if (results.length < 3 && parentCategoryId) {
    // Widen to parent: fetch siblings + self + widen to all children of parent
    const parentCategoryDescendants = await getCategoryDescendantIds(parentCategoryId, /*bySlug*/ false)
    const additional = await db
      .select(/* card columns */)
      .from(techReviews)
      .innerJoin(techProducts, eq(techReviews.productId, techProducts.id))
      .where(and(
        eq(techReviews.status, "published"),
        inArray(techProducts.categoryId, parentCategoryDescendants),
        ne(techReviews.id, currentReviewId),
        notInArray(techReviews.id, results.map((r) => r.id)),  // exclude already-shown
      ))
      .orderBy(desc(techReviews.publishedAt))
      .limit(3 - results.length)
    results = [...results, ...additional]
  }
  return results.map(mapToCard)
}
```

## Open Questions

1. **Where does the query layer live — `src/lib/tech/queries.ts` or co-located?**
   - What we know: CLAUDE.md and CONTEXT.md both defer this to planner. Pattern matches existing `src/lib/booking/*` for centralization, or blog page for co-location.

   - What we know: Admin publish is the user-facing trigger for content changes. `revalidateTag`/`revalidatePath` from the admin publish action would make 60s feel instant.

   - Recommendation: **Yes — add those `revalidatePath` calls inside the publish/unpublish actions** as a one-task chore early in this phase. Makes the 60s revalidate a safety net rather than the primary path.

3. **Suspense boundary on review body?**
   - What we know: UI-SPEC suggests page-level Suspense for the body. Server-side DOMPurify sanitization is fast (<5ms for typical reviews) — may not be worth streaming.
   - Recommendation: **No Suspense on body** — it's not a slow operation. Do Suspense the related-reviews carousel (lowest priority) so the main review renders before related query completes.

4. **Compare tool with URL-pasted invalid product slugs — silent drop vs. explicit error?**
   - UI-SPEC: silent drop + sonner toast "One or more products could not be loaded". Confirmed.

5. **Should `ReviewCard` live in `components/tech/` or `components/cards/`?**
   - Existing pattern puts feature-specific cards alongside feature components (`components/beats/beat-card.tsx`). Put it in `components/tech/review-card.tsx`.

6. **Cursor encoding format?**
   - Options: base64-encoded JSON `{publishedAt, id}`, or two separate URL params `?cursor_at=...&cursor_id=...`.
   - Recommendation: **Single `?cursor=` param, base64(JSON).** Cleaner URL; nuqs handles base64 via `parseAsString`. Server decodes once per request.
