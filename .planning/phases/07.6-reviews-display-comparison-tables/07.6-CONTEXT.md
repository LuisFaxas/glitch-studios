# Phase 07.6: Reviews display & comparison tables - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver the **public-facing consumption surfaces** for Glitch Tech's review vertical. This phase is the consumption counterpart to 07.5's admin input.

**In scope:**

1. `/tech/reviews/[slug]` — full review detail page (hero + verdict + body + rating card + pros/cons + gallery + video + specs accordion + benchmarks + audience + related + CTA + newsletter)
2. `/tech/reviews` — reviews list (filter bar + card grid + sort + search, cursor pagination)
3. `/tech/categories` — categories index (tile grid with counts)
4. `/tech/categories/[slug]` — single-category browse (sub-category pills + reviewed products + un-reviewed products)
5. `/tech/compare` — comparison tool (product picker up to 4 + summary + specs/benchmarks/price tabs + winner detection)
6. **Homepage wire-up** — migrate `tech-featured-reviews-carousel`, `tech-category-tiles`, `tech-benchmark-spotlight` from hardcoded placeholders to real server-fetched data (D-06)
7. Server query layer (Drizzle) for all of the above
8. SEO / metadata / JSON-LD for review detail and category pages

**Explicitly out of scope:**
- Admin CRUD surfaces (delivered in 07.5)
- Schema changes (reuse 07.5 schema as-is)
- Affiliate click tracking (schema has `affiliate_url`; rendered as plain link this phase)
- Review comments, voting, version history
- Full-text search (reviews list uses simple `ILIKE` on title/product name)
- Tech accent color (still deferred; winner signaling uses inverted-surface pattern)
- OG image generation for `/tech/compare`

</domain>

<decisions>
## Implementation Decisions

### Related Reviews (carousel at bottom of review detail)

- **D-01:** The carousel shows **3 most recently published reviews in the same category** as the current review, excluding the review itself. Sort by `publishedAt DESC`. Status filter: `published` only.
- **D-02:** If fewer than 3 published reviews exist in the category, **widen to the parent category node** and refill until 3 are found. If still fewer than 3 (reviews index is new, widened set is also <3), render the section with however many we have (do not hide).

### Comparison Benchmark Charts

- **D-03:** On `/tech/compare`, **render the first 3 benchmark charts eagerly**; subsequent charts mount via `IntersectionObserver` when scrolled into view. Initial DOM stays small on page load; full-page scroll reveals the rest without a manual "Show more" click.
- **D-04:** **Recharts is dynamically imported with SSR disabled** on all compare-tool components: `const BarChart = dynamic(() => import("recharts").then(m => m.BarChart), { ssr: false })`. Applied to `BarChart`, `Bar`, `XAxis`, `YAxis`, `CartesianGrid`, `ResponsiveContainer`, `LabelList`. Review detail and reviews list pages never import Recharts — comparison-only dependency.

### HTML Sanitization (review body rendering)

- **D-05:** Sanitize `review.body_html` **server-side with `isomorphic-dompurify`** before rendering in the RSC. Allowlist:
  - Headings: `h1`, `h2`, `h3`, `h4`
  - Text: `p`, `strong`, `em`, `u`, `br`
  - Lists: `ul`, `ol`, `li`
  - Links: `a` — force `rel="noopener noreferrer"` and `target="_blank"` for external URLs
  - Media: `img` (admin-authored, from media library)
  - Blocks: `blockquote`, `code`, `pre`, `hr`
  - Forbidden globally: `script`, `style`, `iframe`, `object`, `embed`, event handlers (`onclick` etc.), inline `style` attributes
  Defense-in-depth against a compromised admin account. Executed once per request (no client cost). `pnpm add isomorphic-dompurify`.

### Homepage Wire-Up (moved from placeholder to live data)

- **D-06:** Phase 07.6 includes a **final plan that migrates the homepage Tech section** from hardcoded data to server-fetched data:
  - `tech-featured-reviews-carousel.tsx` → fetch latest 3 `tech_reviews` where `status='published'`, joined with `tech_products` and hero image (`tech_reviews.heroImageId`)
  - `tech-category-tiles.tsx` → fetch top-level `tech_categories` (`level = 1`) with product counts (left join + count group by) and review counts
  - `tech-benchmark-spotlight.tsx` → fetch the product with the highest benchmark-run score for a configured "featured test" (initially: highest Geekbench 6 Multi score from any published review's product). If no published reviews exist, hide the spotlight section entirely.
  This makes the homepage go live with real editorial content in the same phase as the review detail page launches.
- **D-06a:** Rating format correction: migrate the card rating display in `tech-featured-reviews-carousel.tsx` from `/5` to `/10` to match 07.5 schema (4 dimensions × 1–10 plus overall).

### Category URL Structure

- **D-07:** Category browse URL = `/tech/categories/[slug]` with `[slug]` being the **leaf category slug only**, **globally unique** across the `tech_categories` table. Schema already enforces this (`slug: text("slug").notNull().unique()` in `src/db/schema.ts:642`). Breadcrumb (Category > Subcategory > Leaf) is resolved at render time by walking the `parent_id` chain.
- **D-07a:** `/tech/categories` (index) lists only **level 1** (top-level) categories with aggregate counts. Clicking a top-level tile navigates to `/tech/categories/[level1-slug]`, whose page shows its child categories as sub-category pills.

### Claude's Discretion

The following were intentionally left to planner/researcher/executor judgment:

- **Caching strategy** — Use Next.js default fetch caching with reasonable `revalidate` intervals per page. Suggested starting points (planner may adjust):
  - Review detail pages: `revalidate = 60` (1 min; admin edits feel near-live)
  - Reviews list / category pages: `revalidate = 60`
  - Homepage tech components: `revalidate = 300` (5 min)
  - Compare tool: no revalidate (on-demand with URL params)
- **Data-fetching architecture** — Server Components with Drizzle queries co-located near the page or extracted to `src/lib/tech/queries.ts` (planner picks based on reuse surface).
- **Image optimization sizing** — Follow existing `<Image>` usage in `tech-featured-reviews-carousel.tsx` for card thumbs; pass `priority sizes="100vw"` on review detail hero; all other uses inherit Next.js defaults.
- **Suspense boundaries** — Planner decides; suggest page-level Suspense for review detail (stream the body), and per-section Suspense for the related-reviews carousel (it's lowest priority).
- **Winner detection library** — Encode defaults in `src/lib/tech/spec-direction.ts` (new file) as a keyed map. Executor writes it.
- **404 routing** — Use Next.js `notFound()` helper on invalid review/category slug; copy matches UI-SPEC empty state text.
- **Test coverage** — Follow project convention (Playwright suite lives in `tests/`); executor decides whether to add E2E specs for the new routes.

</decisions>

<specifics>
## Specific Ideas

- **Review page reading rhythm.** UI-SPEC locks section order to mirror the admin editor field order. This is deliberate — editors should be able to scan a published review and see it structured exactly how they authored it. Don't reorder during implementation.
- **Comparison page is a tool, not a story.** UI-SPEC explicitly says "no animations on picker state toggle, filter pill toggle, sort dropdown, tab active state." The compare page should feel like an instrument — instant feedback, no theatrics.
- **Review body defense-in-depth.** D-05 sanitization is NOT because admin is untrusted. It's a belt-and-suspenders hedge against future admin account compromise or a Tiptap XSS bug. The cost is tiny (server-side, one call per request) and the protection surface is large.
- **Card single-sourcing matters.** The `ReviewCard` component is used in 4 surfaces (reviews list, related reviews carousel, category browse grid, homepage featured carousel post-migration). If the visual drifts across surfaces, the monochrome tightness of the brand cracks. Planner should build `ReviewCard` first and migrate existing hardcoded cards to use it (part of D-06).
- **Winner-detection map is a small but load-bearing file.** `src/lib/tech/spec-direction.ts` encodes "higher-is-better" vs "lower-is-better" defaults for numeric spec fields (RAM, SSD, weight, price, battery, score). A single misconfiguration here turns the comparison table misleading. Planner should include a task to verify the map against the current admin spec templates in the DB.

</specifics>
