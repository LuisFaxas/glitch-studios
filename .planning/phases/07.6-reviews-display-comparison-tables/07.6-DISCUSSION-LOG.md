## Related Reviews Selection

### Q1: How should 'Related reviews' be selected on a review detail page?

| Option | Description | Selected |
|--------|-------------|----------|
| Latest in same category | Pick 3 most recently published reviews in the same category (excluding self). Simple, predictable, freshness-forward — matches most review-site conventions. | ✓ |
| Highest-rated in category | Pick 3 highest-overall-rated reviews in the same category. Rewards quality but can feel stale (top reviews never rotate). | |
| Mix: 2 latest + 1 top-rated | Blend freshness + quality. More editorial feel. | |
| Latest in same subcategory, fallback to category | Narrowest first: 3 latest in leaf subcategory; if fewer than 3, widen to parent category. Tighter relevance for deep trees. | |

**User's choice:** Latest in same category (recommended)

### Q2: If the category has fewer than 3 published reviews, what's the fallback?

| Option | Description | Selected |
|--------|-------------|----------|
| Widen to parent category | Pull from the parent node of the category tree until we have 3. Preserves 'related' feel. | ✓ |
| Hide the section | If fewer than 3 exist, hide Related Reviews entirely. Cleanest visual but loses a conversion surface. | |
| Widen to any published review, latest first | Fall back to global latest published. Always surfaces something. | |

**User's choice:** Widen to parent category (recommended)

## Comparison Chart Rendering

### Q1: How should comparison benchmark charts render when many exist?

| Option | Description | Selected |
|--------|-------------|----------|
| Render 3 eagerly, rest on scroll-into-view | First 3 render immediately; remaining use IntersectionObserver to render as user scrolls. Fast initial paint. | ✓ |
| Render all eagerly | Simplest. All charts mount on load. Risky at max scale (10 charts). | |
| Paginate with 'Show more benchmarks' | First 3 charts + button to reveal remainder. One more interaction. | |
| Tab per benchmark test | Each benchmark under its own sub-tab. Only 1 chart rendered at a time. More clicks. | |

**User's choice:** Render 3 eagerly, rest on scroll-into-view (recommended)

### Q2: Should Recharts be dynamically imported?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — dynamic import with SSR disabled | `const BarChart = dynamic(() => import('recharts').then(m => m.BarChart), { ssr: false })`. Keeps /tech/reviews bundles lean; Recharts only loads on /tech/compare. | ✓ |
| Yes — dynamic but with server pre-render | Dynamic import but ssr: true. No flash but Recharts ends up in the SSR bundle. | |
| No — static import | Simplest. Recharts loads wherever it's referenced. Accept the ~60kB bundle cost project-wide. | |

**User's choice:** Dynamic import with SSR disabled (recommended)

## HTML Sanitization Approach

### Q1: How should review body HTML be sanitized when rendered?

| Option | Description | Selected |
|--------|-------------|----------|
| Sanitize server-side with isomorphic-dompurify | Run body_html through DOMPurify in the RSC. Allowlist headings, paragraphs, lists, links, images, blockquote, code. Defense-in-depth against XSS. ~20kB server-side, zero client cost. | ✓ |
| Trust admin, no sanitization | Render body_html raw via dangerouslySetInnerHTML. Smallest footprint. Only safe while Tiptap editor enforces allowlist AND no admin accounts compromised. | |
| Sanitize with rehype-sanitize at build/render time | Use rehype-sanitize. Similar protection but more verbose setup. Better if adding other rehype plugins later. | |

**User's choice:** Sanitize server-side with isomorphic-dompurify (recommended)

### Q2: What slug format should /tech/categories/[slug] use for the 3-level tree?

| Option | Description | Selected |
|--------|-------------|----------|
| Leaf slug only, globally unique | URL: /tech/categories/gaming-laptops. Slugs enforced unique across entire tech_categories table. Simplest routing, best for sharing. (Schema already has .unique() on slug.) | ✓ |
| Full hierarchical path | URL: /tech/categories/computers/laptops/gaming-laptops. Catch-all [...slug] route. Better SEO signal but complex routing + redirect on category moves. | |
| Leaf slug + numeric id suffix | URL: /tech/categories/gaming-laptops-a3f. Handles duplicate slug risk but uglier. Rarely worth it. | |

**User's choice:** Leaf slug only, globally unique (recommended)
