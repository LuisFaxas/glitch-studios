# Phase 48.2: Admin Team Access Stabilization - Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Make internal team access easy, reliable, and safe. This phase replaces the current incomplete "Invite Member" behavior, which requires a user to already exist, with a real admin invite flow: the owner/admin enters a team member's details, sends an invite link, the invitee sets their own password, then lands in the correct dashboard with only the permissions they should have.

This phase also verifies that a producer can actually use the admin after accepting an invite. If the producer can sign in but gets confused, blocked, or trapped by the sidebar/navigation, the phase is not done.

Out of scope: transactional email preview governance, newsletter design/approval workflow, broad dashboard IA redesign, public signup redesign, MFA/passkeys, and account-linking settings. Those belong in later phases, especially Phase 48.3 for email preview governance.

</domain>

<decisions>
## Implementation Decisions

### Invite Model
- **D-01:** Replace the current "user must register first" flow with a real invite flow. Admins should be able to add a person directly and send them a link.
- **D-02:** Use a pending-invite model as the default product behavior. The invitee should not need to find a register page first. They click the invite link, set their own password, and then the account becomes usable.
- **D-03:** The invite should collect at minimum name, email, and role. The planner may add a short optional note if it fits the existing UI without slowing the flow down.
- **D-04:** Invites should have clear status: pending, accepted, expired, and revoked/cancelled. The admin UI should support resend and revoke.
- **D-05:** Default invite expiry should be short and boring, approximately 7 days. Expired invites can be resent from the admin member UI.
- **D-06:** If the email already belongs to an existing user, do not create a duplicate account. The flow should safely upgrade that user to the selected role, then send the appropriate access/password email if needed.
- **D-07:** If the email belongs to an existing admin/team member, show that clearly and let an authorized admin change the role instead of sending a confusing duplicate invite.

### Producer Role And Permissions
- **D-08:** Create or standardize a "producer" style internal role/preset so the owner can invite a producer without designing permissions from scratch.
- **D-09:** Producer access should be useful but limited. Recommended starting permissions: manage content, manage media, view clients, manage bookings, and reply messages. Do not grant manage roles, owner access, or broad site settings by default.
- **D-10:** Existing custom roles remain supported. The producer preset is for ease, not a replacement for the role system.
- **D-11:** The admin member list should show both active members and pending invites so the owner understands who has access and who still needs to accept.

### Brand-Aware Access
- **D-12:** Glitch Studios and GlitchTech remain visibly separate brands, even though they share one app, one auth system, and one admin backend.
- **D-13:** Invite emails and invite acceptance screens should be brand-aware. A Studios producer invite should feel like Glitch Studios. A Tech invite should feel like GlitchTech.
- **D-14:** The accepted invite should land the user in the right place. Studios-oriented roles land in the Studios admin context; Tech-oriented roles land in the Tech admin context. If a role can see both, the existing context switcher remains the bridge.
- **D-15:** The planner should reuse the existing brand utilities and email sender helpers rather than inventing a separate brand system.

### Dashboard Simplicity
- **D-16:** Non-owner users should only see the admin sections they can actually use. Empty sections and dead-end links should not show.
- **D-17:** The first producer experience should be simple: accept invite, set password, sign in, arrive in admin, and use the relevant sections without needing technical help.
- **D-18:** Fold the pending todo `sidebar-cannot-expand-after-navigation.md` into this phase as a dashboard-access reliability check. If a sidebar/navigation bug blocks a producer after sign-in, fix it here.
- **D-19:** Keep the dashboard fast and direct. This phase should not add a heavy onboarding wizard or extra admin layers.

### Safety Rules
- **D-20:** Only users with role-management authority can invite team members, resend invites, revoke invites, or change roles.
- **D-21:** Owner access should not be assignable casually through the normal invite dialog. Preserve last-owner and self-demotion protections.
- **D-22:** Never expose secrets, temporary passwords, or raw tokens in the UI, logs, planning docs, or commit history.
- **D-23:** The invite link must be single-use or otherwise safely invalidated after acceptance.

### Proof Required
- **D-24:** Verification must include a real manual smoke path: owner sends invite, invitee receives/clicks link, sets password, signs in, lands in dashboard, sees only correct sections, and can navigate away/back without the sidebar getting stuck.
- **D-25:** Verification must cover at least one existing-user case: inviting/upgrading an email that already exists should not create duplicates or break login.
- **D-26:** Build/typecheck/lint coverage should run as appropriate for the implementation surface.

### Agent Discretion
- The user explicitly asked not to continue answering decision questions and asked the agent to choose the best path for the task.
- The planner/researcher may choose the exact storage shape: a new invite table, Better Auth token primitives, or a hybrid, as long as the product behavior above is preserved.
- The planner may decide whether invite acceptance reuses `/reset-password?invite=1` or gets a dedicated `/accept-invite` route. The user-facing result must be clearer than the current broken flow.
- The planner may tune the exact producer permission preset after checking existing admin routes and role conventions.

### Folded Todos
- `sidebar-cannot-expand-after-navigation.md` - Folded only as a producer/admin access reliability check. The bug should be fixed if it blocks the invited producer from using the admin after navigation.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `AdminMemberTable` already provides a Team Members surface and invite dialog, but the behavior is wrong. This is the natural UI to repair rather than building a new page.
- `admin-roles.ts` already has role assignment and removal logic with owner safeguards.
- `permissions.ts` and `permissions-shared.ts` already define the permission vocabulary. Keep using these unless research finds a hard blocker.
- `admin-sidebar.tsx` already filters visible navigation items by permission.
- `admin-artist-applications.ts` already demonstrates the "create user, mark trusted, send password-reset-style invite" pattern.
- `auth.ts` already has a branded password-reset invite branch via `invite=1`.

### Established Patterns
- Admin actions are server actions under `src/actions/`.
- Admin visibility is permission-driven, not hardcoded only by role names.
- Owner and legacy admin bypass permissions; custom roles are resolved from `admin_roles`.
- Brand detection and email sender logic already exist. Do not duplicate brand behavior in a separate invite-only system.
- React Email/Resend is the existing transactional email path.

### Integration Points
- Admin member UI: replace fetch-based lookup/incomplete invite behavior with a server action.
- Database/schema: add invite persistence if needed, or reuse Better Auth verification/reset token primitives if they satisfy expiry/status/revoke requirements.
- Auth flow: invite acceptance must set a password and result in a usable session or a clear login path.
- Admin navigation: confirm producer/non-owner roles can move through allowed admin sections and the sidebar remains usable after navigation.

</code_context>

<specifics>
## Specific Ideas

- User wants this to feel easy, not technical: "I want this registration to be easy... I want to be able to add people myself and send them a link so they can make their own password. Everything just needs to flow."
- Treat Glitch Studios and GlitchTech as separate visible brands while keeping the shared app/admin architecture.
- The best product direction is not "make the producer register first." The best direction is owner/admin sends invite, producer accepts and sets password.

</specifics>
