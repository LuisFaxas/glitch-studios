# Phase 07.1: Listening Experience & Waveform Overhaul - Research

**Researched:** 2026-03-31
**Domain:** Audio waveform visualization, canvas rendering, server-side peak extraction
**Confidence:** HIGH

## Summary

This phase replaces decorative Math.sin() waveform bars with real audio-driven waveforms across four surfaces: beat cards, list rows, sidebar widget, and player bar (desktop + mobile). The approach is: (1) pre-compute 256 peaks per beat server-side, (2) store in a `waveform_peaks` JSONB column, (3) render via a single reusable canvas component, and (4) keep WaveSurfer.js only in the desktop player bar where it decodes from the live audio element.

The technical risk is low. The project already has WaveSurfer.js working in the player bar and detail panel. The new work is a custom lightweight canvas renderer for pre-computed peaks (avoiding WaveSurfer overhead in cards/rows/widgets), a server-side peak extraction utility using `audio-decode` (pure JS/WASM, no native bindings, Vercel-compatible), and a Drizzle schema migration to add the JSONB column.

**Primary recommendation:** Build a single `<Waveform>` canvas component with `peaks`, `progress`, `interactive`, and `onSeek` props. Use `audio-decode` for server-side MP3-to-PCM decoding and peak extraction. Keep WaveSurfer.js only in the desktop player bar.

### Pattern 1: Canvas Waveform Renderer
**What:** A single reusable `<Waveform>` component using HTML5 Canvas that renders pre-computed peaks with progress fill and optional scrubbing.
**When to use:** Every waveform surface except the desktop player bar (which keeps WaveSurfer.js).
**Key design:**
```typescript
// src/components/player/waveform.tsx
interface WaveformProps {
  peaks: number[] | null    // 0-1 normalized amplitudes from DB
  progress: number          // 0-1 elapsed ratio
  height: number            // canvas height px
  barWidth?: number         // default 2
  barGap?: number           // default 1
  waveColor?: string        // default "#555555"
  progressColor?: string    // default "#f5f5f0"
  interactive?: boolean     // enables click/drag scrubbing
  onSeek?: (progress: number) => void  // 0-1 callback
  className?: string
}
```

**Rendering logic:**
1. Use ResizeObserver to track container width
2. Calculate bar count: `Math.floor(width / (barWidth + barGap))`
3. Downsample peaks array to bar count using max-in-bucket sampling
4. Draw bars bottom-aligned on canvas
5. Split bars at progress point: left side in progressColor, right in waveColor
6. Fallback when peaks is null: flat line at 30% height

**Interaction logic (when interactive=true):**
- `onMouseDown` / `onTouchStart`: begin scrubbing, calculate position
- `onMouseMove` / `onTouchMove`: continuous scrub while held
- `onMouseUp` / `onTouchEnd`: end scrub
- Position calculation: `clientX relative to canvas left / canvas width`
- Call `onSeek(position)` on each interaction

### Pattern 2: Server-Side Peak Extraction
**What:** A utility function that downloads an MP3 from R2, decodes to PCM, and extracts normalized peak amplitudes.
**When to use:** Called during beat upload (admin flow) and seed script.
```typescript
// src/lib/peaks.ts
import decode from "audio-decode"
import { r2Client } from "@/lib/r2"
import { GetObjectCommand } from "@aws-sdk/client-s3"

export async function extractPeaks(
  audioKey: string,
  peakCount: number = 256
): Promise<number[]> {
  // 1. Download audio buffer from R2
  const command = new GetObjectCommand({
    Bucket: process.env.R2_BUCKET_NAME!,
    Key: audioKey,
  })
  const response = await r2Client.send(command)
  const arrayBuffer = await response.Body!.transformToByteArray()

  // 2. Decode to PCM
  const { channelData } = await decode(arrayBuffer.buffer)
  const samples = channelData[0] // mono or left channel

  // 3. Extract peaks by bucketing
  const bucketSize = Math.floor(samples.length / peakCount)
  const peaks: number[] = []
  for (let i = 0; i < peakCount; i++) {
    let max = 0
    const start = i * bucketSize
    const end = Math.min(start + bucketSize, samples.length)
    for (let j = start; j < end; j++) {
      const abs = Math.abs(samples[j])
      if (abs > max) max = abs
    }
    peaks.push(max)
  }

  // 4. Normalize to 0-1 range
  const maxPeak = Math.max(...peaks)
  if (maxPeak > 0) {
    for (let i = 0; i < peaks.length; i++) {
      peaks[i] = peaks[i] / maxPeak
    }
  }

  return peaks
}
```

### Anti-Patterns to Avoid
- **Multiple WaveSurfer instances per page:** WaveSurfer decodes the entire audio file client-side. Creating one per beat card would decode the same file multiple times and be extremely slow. Use WaveSurfer ONLY in the desktop player bar.
- **Animating individual DOM divs as waveform bars:** The current approach creates 48-80 `<div>` elements per waveform. This is slow with many beats. A single `<canvas>` per waveform is far more performant.
- **Re-computing peaks client-side from audio:** Violates D-01 and D-15. Peaks must come pre-computed from the database.
- **Using requestAnimationFrame for progress updates:** The `timeupdate` event fires ~4 times/second which is sufficient for waveform progress. rAF would cause unnecessary repaints.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| MP3 decoding to PCM | Custom WASM decoder or ffmpeg wrapper | `audio-decode` | Handles format detection, WASM loading, buffer management. Pure JS/WASM, no native deps. |
| Desktop player bar waveform | Custom canvas renderer for live audio | WaveSurfer.js (already working) | WaveSurfer syncs to the `<audio>` element directly, handles live decoding + rendering. Already proven in codebase. |
| Drizzle JSONB column | Raw SQL queries | `jsonb()` from `drizzle-orm/pg-core` | Type-safe, integrates with schema, generates proper migration SQL. |

**Key insight:** The phase is split between two rendering strategies for good reason. WaveSurfer.js excels at live audio + waveform sync for a single player. A custom canvas renderer excels at rendering many static peak-based waveforms efficiently. Do not force one tool to do both jobs.

## Common Pitfalls

### Pitfall 1: Canvas Not Resizing on Container Change
**What goes wrong:** Canvas renders at initial size but does not update when the container (card, row, player bar) resizes due to responsive layout changes.
**Why it happens:** Canvas dimensions are set once at mount and forgotten.
**How to avoid:** Use ResizeObserver on the container element. When width changes, update canvas width/height attributes AND re-render bars. Set CSS width to 100% but canvas pixel dimensions to match device pixel ratio for crisp rendering.
**Warning signs:** Waveform looks blurry or has wrong bar count after window resize.

### Pitfall 2: Touch Event Conflicts on Mobile
**What goes wrong:** Scrubbing on the mobile player bar waveform causes the page to scroll, or the card waveform scrub conflicts with the card's click-to-expand behavior.
**Why it happens:** Touch events bubble and default behavior (scroll) is not prevented during scrub.
**How to avoid:** Call `e.preventDefault()` on `touchstart`/`touchmove` within the waveform scrub handler. For beat cards, use `e.stopPropagation()` on the waveform touch/click to prevent the card click handler from firing.
**Warning signs:** Page scrolls while trying to scrub, or card expands when user meant to scrub.

### Pitfall 3: Stale Progress After Beat Change
**What goes wrong:** Switching from beat A to beat B shows beat B's waveform with beat A's progress position briefly.
**Why it happens:** The `currentTime` from AudioPlayerProvider updates asynchronously after the new audio loads.
**How to avoid:** Reset progress to 0 when `currentBeat.id` changes. In the Waveform component, derive progress from `currentTime / duration` only when the beat ID matches. Non-active beats always show progress=0.
**Warning signs:** Brief flash of filled waveform on a beat that just started playing.

### Pitfall 4: JSONB Column Type Mismatch
**What goes wrong:** Drizzle reads the JSONB column but TypeScript types it as `unknown` or the wrong shape.
**Why it happens:** Drizzle's `jsonb()` returns `unknown` by default.
**How to avoid:** Use Drizzle's `.$type<number[]>()` method: `jsonb("waveform_peaks").$type<number[]>()` to get proper typing.
**Warning signs:** TypeScript errors when accessing waveformPeaks, or runtime type assertion needed everywhere.

### Pitfall 5: Peak Extraction Failing on Vercel Serverless
**What goes wrong:** The `audio-decode` WASM module fails to load in Vercel's serverless function environment.
**Why it happens:** Some WASM loaders have issues with serverless bundlers (edge runtime especially).
**How to avoid:** Run peak extraction in Node.js runtime (not Edge). In the admin upload Server Action or API route, ensure the route uses `export const runtime = "nodejs"`. Test peak extraction in the seed script first (runs locally) before wiring to the admin upload flow.
**Warning signs:** "WASM module not found" or "Cannot resolve module" errors in Vercel function logs.

### Pitfall 6: Drizzle Migration Not Applied
**What goes wrong:** The `waveform_peaks` column does not exist in the database after deploying.
**Why it happens:** Developer generates migration but forgets to run `drizzle-kit push` or `drizzle-kit migrate`.
**How to avoid:** Use `drizzle-kit push` for development (applies schema changes directly). Use `drizzle-kit generate` + `drizzle-kit migrate` for production. Verify column exists before seeding.
**Warning signs:** Runtime SQL error "column waveform_peaks does not exist".

## Code Examples

### Canvas Waveform Rendering (core draw function)
```typescript
function drawWaveform(
  ctx: CanvasRenderingContext2D,
  peaks: number[],
  width: number,
  height: number,
  progress: number,
  barWidth: number,
  barGap: number,
  waveColor: string,
  progressColor: string,
) {
  ctx.clearRect(0, 0, width, height)
  const step = barWidth + barGap
  const barCount = Math.floor(width / step)
  const progressX = progress * width

  // Downsample peaks
  for (let i = 0; i < barCount; i++) {
    const peakIndex = Math.floor((i / barCount) * peaks.length)
    const amplitude = peaks[peakIndex] ?? 0
    const barHeight = Math.max(2, amplitude * height)
    const x = i * step
    const y = height - barHeight

    ctx.fillStyle = x + barWidth <= progressX ? progressColor : waveColor
    ctx.fillRect(x, y, barWidth, barHeight)
  }
}
```

## Open Questions

1. **audio-decode WASM loading in Vercel serverless**
   - What we know: audio-decode claims to work in Node.js without native bindings. It uses WASM for MP3 decoding.
   - What's unclear: Whether Vercel's Node.js function bundler correctly includes the WASM files. Some bundlers strip .wasm files.
   - Recommendation: Test in the seed script (local Node.js) first. If it works locally, it will very likely work in Vercel Node.js runtime. If Vercel is problematic, peak extraction could run as a local script triggered manually rather than in a Server Action.

2. **Peak extraction for existing BREAK 5 beat**
   - What we know: BREAK 5 audio is stored in R2 at `beats/audio/break-5.mp3`. The seed script needs to generate peaks.
   - What's unclear: Whether R2 credentials are currently configured in the local `.env`.
   - Recommendation: Try downloading the audio in the seed script. If R2 is not configured, generate synthetic but realistic-looking peaks for BREAK 5 as a temporary measure, with a TODO to regenerate from real audio later.
