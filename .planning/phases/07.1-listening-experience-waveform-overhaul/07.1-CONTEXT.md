# Phase 07.1: Listening Experience & Waveform Overhaul - Context

**Gathered:** 2026-03-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace all decorative/fake waveforms with real audio-driven waveforms across every beat touchpoint (cards, list rows, player bar, sidebar widget). Make waveforms interactive with scrubbing. Polish the global player bar (desktop + mobile) and upgrade the sidebar now-playing widget to a rich mini-player with cover art and waveform.

</domain>

<decisions>
## Implementation Decisions

### Waveform Data Strategy
- **D-01:** Pre-compute waveform peaks server-side when a beat is uploaded (not lazy/on-first-play).
- **D-02:** Store peaks as a JSON column (`waveform_peaks`) in the `beats` table. Small data (~2-4KB per beat), comes with beat data in a single query — no extra fetch.
- **D-03:** Peak extraction should use a server-side audio processing library (e.g., audiowaveform, ffmpeg, or a Node audio decoder). Research the best approach for the Next.js/Vercel stack.

### Card Waveform Design
- **D-04:** Beat cards must show the real waveform shape generated from the stored peaks data, not decorative Math.sin() bars.
- **D-05:** Card waveforms must be **interactive with scrubbing** — click/drag to seek. Progress fills as the beat plays.
- **D-06:** Clicking a card waveform starts playback in the global audio player context if not already playing.

### List Row Waveform Design
- **D-07:** List rows must show real waveform from stored peaks, replacing the current decorative bars.
- **D-08:** Claude's Discretion — whether list row waveforms are fully interactive (scrubbing) or visual-only with progress. Decide based on row space constraints and UX.

### Sidebar Now-Playing Widget
- **D-09:** Upgrade widget to include: cover art thumbnail, mini waveform with progress, play/pause, track info (title, artist, time).
- **D-10:** Sidebar waveform must be **interactive — click to scrub** to a position.
- **D-11:** The tile becomes a rich mini-player, consistent with the interactive waveform pattern across the app.

### Player Bar Improvements
- **D-12:** Add **time display** (elapsed / total) to the desktop player bar alongside the waveform.
- **D-13:** Add a **real waveform with scrubbing on mobile** — replace the current 2px progress bar. Scale appropriately for smaller screens.
- **D-14:** Polish overall interaction quality — easy to scrub, good tap targets on mobile, responsive across breakpoints.
- **D-15:** The waveform in the player bar should render from the same pre-computed peaks data (not re-decode audio client-side), for consistency and performance.

### Claude's Discretion
- List row waveform interactivity level (D-08)
- Exact number of peaks/bars to render at each size (card, row, sidebar, player bar)
- WaveSurfer.js vs custom canvas renderer for peak-based waveforms (WaveSurfer may be overkill for pre-computed peaks — a lighter canvas approach may perform better)
- Mobile waveform height and tap target sizing
- How to handle the seed script / existing beat (BREAK 5) — generate its peaks as part of this phase

</decisions>
