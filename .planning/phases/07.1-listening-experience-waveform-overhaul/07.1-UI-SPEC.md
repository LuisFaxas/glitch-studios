## Component Inventory

### 1. Waveform Renderer (NEW)

A reusable canvas-based waveform component that renders pre-computed peaks data. Used in all four waveform contexts below.

**Props contract:**
- `peaks: number[]` -- normalized 0-1 amplitude values from `waveform_peaks` column
- `progress: number` -- 0 to 1 representing elapsed playback position
- `barWidth: number` -- width of each bar in pixels (default: 2)
- `barGap: number` -- gap between bars in pixels (default: 1)
- `height: number` -- canvas height in pixels
- `waveColor: string` -- color for unplayed bars (default: `#555555`)
- `progressColor: string` -- color for played bars (default: `#f5f5f0`)
- `interactive: boolean` -- whether click/drag scrubbing is enabled
- `onSeek?: (progress: number) => void` -- callback when user scrubs (0 to 1)

**Rendering rules:**
- Use HTML5 `<canvas>` element, not individual DOM divs
- Downsample peaks array to fit available pixel width: `Math.floor(canvasWidth / (barWidth + barGap))`
- Bars render bottom-aligned, centered vertically within the canvas height
- Canvas must resize on container width change (ResizeObserver)
- When `interactive` is true: cursor changes to `pointer`, mouse/touch events enable scrubbing

**Interaction behavior (when interactive):**
- Click: seek to clicked position instantly
- Drag: continuous scrub while mouse/touch is held
- Touch: same behavior with `touchstart`/`touchmove`/`touchend`
- Call `onSeek(clickX / canvasWidth)` on interaction

**Fallback (no peaks data):**
- Render a flat line at 30% height in `waveColor` -- never render the old Math.sin() decorative bars

## Interaction Contracts

### Waveform Scrubbing

| Context | Interactive | Scrub method | Min touch target |
|---------|------------|--------------|------------------|
| Beat card | Yes | Click + drag | 28px height (card context, fine for desktop; on mobile the card play overlay is the primary action) |
| List row | No | N/A | N/A |
| Sidebar widget | Yes | Click + drag | 20px height + 10px padding = 40px zone |
| Player bar desktop | Yes (WaveSurfer) | Click + drag | 40px (WaveSurfer height) |
| Player bar mobile | Yes | Tap + drag | 44px zone (24px canvas + 20px padding) |

### Playback State Sync

All waveform components read from `AudioPlayerProvider`:
- `currentBeat` -- determines which beat's peaks to highlight
- `isPlaying` -- determines animation state
- `currentTime` / `duration` -- determines progress fill (progress = currentTime / duration)
- `seek(time)` -- called on scrub interaction with `time = progress * duration`

A beat card waveform click when no beat is playing must:
1. Call `play({ id, title, artist, previewAudioUrl, coverArtUrl })`
2. Then track `currentTime`/`duration` for progress fill

### Peak Data Flow

1. Server: peaks pre-computed and stored in `beats.waveform_peaks` (JSON column, array of 0-1 floats)
2. API: `getPublishedBeats` includes `waveformPeaks` in response
3. Client: `BeatSummary` type extended with `waveformPeaks: number[] | null`
4. Render: Waveform component downsamples peaks to fit canvas width
5. Fallback: When `waveformPeaks` is null, render flat line at 30% height

### Peak Count Recommendations

| Context | Target bar count | Rationale |
|---------|-----------------|-----------|
| Beat card | ~48-80 | Card is ~200-300px wide, 2px+1px = ~3px per bar |
| List row | ~80-160 | Row waveform is wider, flex-1 in desktop layout |
| Sidebar widget | ~60-80 | Widget is ~248px internal width |
| Player bar desktop | ~200-400 | Flex-1, largest waveform surface |
| Player bar mobile | ~100-200 | Full phone width minus padding |

Store 256 peaks per beat in the database. The Waveform component downsamples to fit.
