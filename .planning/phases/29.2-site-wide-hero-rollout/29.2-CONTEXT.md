# Phase 29.2 — Site-Wide Hero Rollout + Categories Visual Upgrade

## Why this phase exists

After Phase 29.1 shipped (rankings polish), user UAT identified a consistent visual gap:
the `<TechHero>` component built in 29.1-05 was scoped to only three surfaces
(rankings hub, canonical leaderboard, category detail). Every other tech route still has
a bare `<h1>` — bland, missing CTAs, no editorial rhythm. User flagged this as
site-wide pattern debt:

> "It's missing hero section. It's missing rich elements. It's missing a reason to keep
> people reading. It's just a little bland. As a matter of fact, a lot of our pages are
> missing the hero section. And the hero section should be like a standard right now."

User also flagged the category tile imagery problem:

> "When it comes to categories and when it comes to images, etc., are we considering the
> categories, images, and their thumbnails at all? Because as of now, there's no images.
> There's just like a big box, you know? icon but it's too small either you make the
> icons large or we add images to this."

### Methodology page (`/tech/about`) editorial upgrade

Beyond the hero, the page is "a lot of text" — accurate but bland. Rich elements that
would help retention without compromising the methodology's rigor:

- **Stat cards row** above each major section — e.g. "43 tests", "13 disciplines",
  "7 BPR-eligible" rendered as 3 large mono-font cards with terminal-glitch styling.
- **Discipline cards** — replace the discipline table with a grid of hover-able
  cards. Each card: discipline name (large), description (small), BPR-eligible
  badge, animated underline on hover.
- **Medal threshold visual** — replace the table with a vertical timeline / ladder
  showing Platinum → Gold → Silver → Bronze with the score range as a visible bar.
- **Live formula block** — the BPR formula already exists as a `<pre>`. Style it
  with terminal-prompt aesthetic (`$` prefix, blinking cursor on hover).
- **Reference-baseline table glitch styling** — current table is functional but flat.
  Add row hover that shifts the discipline name 2px right with an RGB split.

### Category tile imagery

Currently `<CategoryTile>` and the rankings-hub inline tile both render a 40px
lucide icon centered in `aspect-square`. Two acceptable directions:

- **Direction A (preferred):** tile has a real category thumbnail (CDN-hosted glitch-styled
  product photo or representative image) as the bg with the lucide icon as a small
  badge in the corner. Falls back to a hero-sized icon if no thumbnail exists.
- **Direction B (acceptable for placeholder phase):** scale the lucide icon to 80–96px
  (50–60% of tile area) with a category-descriptor caption underneath
  ("12 reviewed laptops · 7 ranked").

Decision deferred to plan-phase, but A is the launch-quality target.

## Cross-cutting requirements

- TechHero today has a fixed `h-[280px]` and `bg-style` driven by `tone`. A few of
  these surfaces (about, blog) may want **larger or differently-shaped heroes**.
  Consider promoting `TechHero` to accept a `size?: "compact" | "default" | "tall"`
  variant prop so /tech/about can use `tall` (h-[400px] or 100svh) without
  inventing a new component.
- Every hero h1 wraps in `<GlitchHeading>` (memory: `feedback_glitch_headers`).
- No auto-running animation on h1 — hover-only RGB split (memory).
- Brand spelling: GlitchTech everywhere (memory).
- Sidebar must not scroll — if hero changes affect layout density, don't add new
  fixed-height widgets that push the sidebar past one screen (memory:
  `feedback_sidebar_no_scroll`).

## Suggested plan breakdown (sequential, no parallel waves)

Each plan is small, independently verifiable, and committed atomically. Same shape
as Phase 29.1 (Playwright spec per plan against `pnpm dev`).

**Mobile coverage requirement** (load-bearing, do not skip):

- Plan 01 runs both `--project=desktop --project=mobile` since the hero-height
  assertion is viewport-agnostic (Tailwind utility heights apply at every breakpoint).
- Plans 02–09 may run `--project=desktop` only when their per-route layout
  assertions are desktop-specific (e.g. side-by-side stat-card grids that wrap on
  mobile). When a plan's spec contains nothing desktop-specific, default to running
  both projects — saves churn at the phase smoke.
- Plan 10's phase smoke runs `--project=desktop --project=mobile` and is the
  authoritative mobile gate for every surface mounted in 02–09. If the smoke
  catches a mobile regression, the responsible plan reopens for a fix.

This pattern keeps individual plans cheap to verify while still satisfying the
phase-level requirement: "All Playwright specs green on desktop + mobile projects."

1. **29.2-01 — TechHero size variants** — promote `TechHero` to accept
   `size?: "compact" | "default" | "tall"` (default = current h-[280px]).
   Spec: assert each size renders at the right height.
2. **29.2-02 — /tech/about hero + stat cards** — mount tall TechHero;
   add 3-card stat row (43 tests / 13 disciplines / 7 BPR-eligible) above
   `<MethodologyFormula>`. Spec: hero visible, stats render with correct numbers.

4. **29.2-04 — /tech/about medal threshold ladder** — replace
   `<MethodologyMedalTable>` with vertical ladder visualization. Spec: 4 tiers in

5. **29.2-05 — /tech/reviews TechHero** — cyan, "Browse latest" CTA. Spec.
6. **29.2-06 — /tech/categories hub TechHero** — amber, "View rankings" CTA. Spec.
7. **29.2-07 — Category tile imagery (Direction A or B)** — decision in plan-phase.
   Spec: tile has visual bulk beyond a 40px icon.
8. **29.2-08 — /tech/compare TechHero** — cyan, "Pick devices" CTA. Spec.
9. **29.2-09 — /tech/benchmarks TechHero** — cyan, "Read methodology" CTA. Spec.

    or "Coming soon" treatment if content isn't ready. Spec.
