# Phase 29.3 Plan 01: Reduce Filter-Path GPU Baseline Summary

**Five surgical GPU/render-baseline fixes across four files: hover-gate Footer LogoTile, memoize AudioPlayerProvider context, drop idle WidgetNowPlaying canvas, clean stale columns useMemo deps, and reduce leaderboard table min-width — eliminates the persistent baseline cost suspects identified by the 4-agent code audit.**

## Accomplishments

- **Footer GPU layer eliminated** — `glitchLayer1`/`glitchLayer2` divs (mix-blend-mode + filter) no longer mount on idle pages. Permanent baseline GPU compositing layer is gone from every page that mounts the footer.
- **PlayerBar + WidgetNowPlaying re-render cascade stopped** — context value identity is stable across parent re-renders (e.g., usePathname change in `(tech)/layout.tsx`, sidebar collapse). Eliminates one of the suspects for chip-click crash compounding.
- **No-track-state canvas removed** — sidebar widget no longer allocates a `<canvas>`-backed Waveform when there's nothing to play. Replaced with a `h-8 w-full` div placeholder.
- **~14-column rebuild on every sort/filter click eliminated** — columns useMemo deps reduced to `[benchmarkColumns]`. SortHeader was already a no-op visual after the atomic fix; the deps were stale.
- **GPU rasterization surface shrunk** — table `min-width: 1600px → 1280px`. Page wrapper is `max-w-[1600px]`, so the previous value forced viewport-wide rasterization on a maxed Mac display and slightly overflowed on anything narrower.

## Decisions Made

- **Footer "use client"** chosen over extracting a `<FooterLogo>` client island. Rationale: NewsletterForm child is already a client component (verified by reading `src/components/forms/newsletter-form.tsx` line 1: `"use client"`), so marking the entire Footer as client costs zero SSR. Extracting an island would require a new file + exports for the same effect — bigger blast radius.
- **Table min-width: 1280px** over the plan's first-choice `min-width: 100%`. Rationale: the Codebox VM dev environment cannot easily verify whether columns squish past usability on real macOS browsers at narrow viewports. 1280px is the laptop-standard fallback explicitly authorized by the plan and is safer than risking a layout regression that Plan 29.3-02 would have to debug. The 1600px → 1280px reduction still removes ~320px of GPU rasterization surface on common viewports while preserving column legibility.
- **Columns useMemo eslint-disable retained.** Initially attempted to remove it (since `[benchmarkColumns]` is now exhaustive for variables read inside the memo). ESLint then flagged `filters.sort/filters.dir/onSort` as missing because they ARE referenced in the JSX body (passed to SortHeader as props). Since SortHeader destructures only `label` and `methodologyAnchor`, those props are dead — but ESLint can't see that. Restored the disable with an updated comment explaining why.

## Choice: Table Min-Width

**Plan 29.3-01 instructed: "Try `min-width: 100%` first ... If layout breaks on narrow viewports (columns squish past usability), use `min-width: 1280px`."**

Selected: **`min-width: 1280px`**

**Rationale:**
1. Codebox VM cannot run real macOS Safari + Firefox to validate `100%` layout at 1280×800, 1440×900, 1920×1080.
2. 1280px is the laptop-standard breakpoint and is the plan's explicit fallback.
3. Plan 29.3-02 is a real-browser human-gate verification — it can A/B 1280px vs 100% interactively if useful.
4. The 1600px → 1280px reduction still removes ~20% of GPU rasterization surface on a 1280-wide viewport (which was overflowing) and ~30% on a max-w-[1600px] desktop.
5. The product column already enforces `min-w-[200px] lg:min-w-[280px]` (line 632); other columns can size to content above 1280px.

If Plan 29.3-02 verification reveals columns are crowded at 1280px, switching to `100%` is a 1-character edit follow-up.
