## ⚠ FAILURE — chip click still freezes macOS Safari

**User test result (2026-04-27):** "I picked, clicked on the year, drop down came out, clicked on a specific year and everything freezes. The thing just freezes and stops working."

**Critical data point:** The DROPDOWN OPEN action works fine. The freeze happens on the chip click itself, which is the `setFilters → table re-render` path. This means:

- The dropdown architecture (Suspect #6 from audit) is NOT the dominant cost — opening the dropdown does not setFilters and does not crash
- The cost is in the table re-render that fires when filter state changes
- The Plan 29.3-01 baseline fixes (Footer hover-gate, AudioPlayer memo, WidgetNowPlaying canvas, columns memo deps, table min-width) were necessary but NOT SUFFICIENT
- There is at least one more cost in the chip-click → table-re-render path that we have not yet identified

**Yellow flag from Plan 04 in retrospect:** Headless Linux Playwright "passed" only after switching from natural `.click()` and `keyboard.press("Escape")` to `dispatchEvent("click")` everywhere. Natural events were hanging on chromium and throwing "Page closed" on webkit after the rapid batch. We rationalized this as "headless quirks" but it is now clear the rapid-chip-click cycle DOES degrade the page event loop even on Linux — just not fatally there. On macOS Metal, it IS fatal.

**Defensive action taken:** Plan 03 (filter re-mount) was reverted on master via `git revert 5368fcd` (revert commit `f60ab7d`). The filter UI is hidden again, so the live page does not crash for any visitor.

**Next:** New investigation phase needed. Suspect surface narrowed significantly — the cost is in the table re-render cycle on filter change, NOT the dropdown layer.
