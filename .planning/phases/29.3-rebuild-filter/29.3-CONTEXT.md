<domain>
## Phase Boundary

The 4-agent code audit on 2026-04-26 revealed that the original Phase 29.3 plan (rebuild the dropdown layer with a hand-rolled `<FilterDropdown>` primitive) was targeting the wrong layer. The dropdown architecture is NOT the dominant cost. Three agents independently converged on findings showing the cost is in the **persistent baseline** — code that runs on every page render whether the filter is mounted or not. The "sort header click also crashes" symptom in the original debug artifact is the smoking gun: sort uses the same `setFilters` write path as filter chips would, doesn't need the filter UI mounted, and triggers the same churn cycle.

This phase fixes the baseline first (Plan 01), validates stability on real macOS browsers via sort-header click WHILE FILTER STILL HIDDEN (Plan 02 — human gate), then re-mounts the existing `LeaderboardFilters` chip-bar as-is (Plan 03), gates with Playwright Webkit + Firefox crash-repro (Plan 04), and final-verifies on real macOS browsers (Plan 05 — human gate).

**In scope:**
- Footer hover-gate: mount `LogoTile` glitchLayer1/2 only on hover (mirror `logo-tile.tsx:24-55` pattern)
- AudioPlayerProvider value memoization (wrap context value in `useMemo`)
- WidgetNowPlaying canvas conditional render (skip `<Waveform>` when `currentBeat === null`)
- LeaderboardTable `columns` useMemo dep cleanup (remove stale `filters.sort, filters.dir`)
- LeaderboardTable `min-width: 1600px` reduction
- Re-mount `<LeaderboardFilters>` and `<LeaderboardFilterSheet>` (delete atomic-fix comment block; no other changes)
- Playwright crash-repro test on Webkit + Firefox engines (adds projects to `playwright.config.ts` if missing)
- Vercel preview + macOS Safari/Firefox real-browser verification (TWO checkpoints — baseline-only and final)

**Out of scope:**
- Rebuilding the dropdown layer / `<FilterDropdown>` primitive / rewriting `LeaderboardFilters` chip-bar (deferred — only resurrected if Plan 5 finds chip click still crashes after baseline fixes are confirmed working in Plan 2)
- Rewriting `applyFilters` / `deriveBounds` / `AllFilters` state machine
- Reintroducing URL state for filters (forbidden — debug artifact constraint)
- Replacing the placeholder product cell (already fixed, CSS-color tile stays)
- Reintroducing sticky cells, drop-shadow on shell elements, or 1200×800 placeholder PNG
- Removing the `<Toaster>`, `<CartDrawer>`, `<FloatingCartButton>`, `<TooltipProvider>` (out of scope for this pass; address only if Plan 5 still crashes after the targeted fixes)
- Replacing Base UI `Slider` in `PriceFilterPopover` (defer; minor compared to HIGH suspects)
- Stripping TechHero scanline gradients (Suspect #5 — defer; A/B test only if Plan 5 fails)

## Audit Findings (2026-04-26 — 4 parallel agents)

The audit ran 4 parallel agents across the leaderboard render path, the always-mounted shell, and cross-cutting concerns. Suspects ranked by likelihood + cost-to-fix.

### 🔴 HIGH suspects (load-bearing — MUST fix this phase)

**Suspect #1 — Footer mounts LogoTile glitchLayer1/2 UNGUARDED**
- File: `src/components/layout/footer.tsx:25-29`
- Pattern: `<div className={styles.glitchLayer1} />` and `<div className={styles.glitchLayer2} />` are rendered unconditionally inside the footer's logo wrapper. The CSS-module classes (`logo-tile.module.css:34, 40`) carry `mix-blend-mode: screen` + `filter: hue-rotate(...) saturate(3) brightness(1.2)` on `position: absolute inset-0` divs.
- Impact: On macOS Safari + Firefox (Metal/WebRender), `mix-blend-mode + filter` on a positioned element forces a permanent GPU compositing layer to be allocated for the wrapper, EVEN WHEN the layers are `opacity:0` and visually invisible. Footer is mounted on every page. Permanent baseline cost on /tech/rankings/laptops AND every other page.
- Reference fix: `src/components/tiles/logo-tile.tsx:24-55` already uses the correct pattern — `useState` + `onMouseEnter/onMouseLeave` + conditional render `{hovered && (<><div .glitchLayer1 /><div .glitchLayer2 /></>)}`. Footer needs the same treatment. Two audit agents found this independently — it is the strongest single finding.
- Note: Footer is a server component as written. To use `useState`, mark it `"use client"` OR extract a small client-only `<FooterLogo>` component that owns the hover state. Either is fine; pick the one that minimizes blast radius (extracting a client island is cleaner if you don't want to "client-ify" the whole footer).

**Suspect #2 — `columns` useMemo has stale `filters.sort, filters.dir` deps**
- File: `src/components/tech/leaderboard-table.tsx:541-542`
- Pattern: `const columns = useMemo(() => [ ... ], [benchmarkColumns, filters.sort, filters.dir])`
- Impact: Every sort click rebuilds all ~14 column defs → every `<th>` reconciles → the full 1600px-wide `<thead>` re-paints. `SortHeader` (lines ~142-171) is currently a no-op visual after the atomic fix — it does NOT consume `filters.sort` or `filters.dir`. The deps are stale. Sort and filter share the same write path through `setFilters`, so this churn fires on filter changes too once the filter is re-mounted.
- Fix: Remove `filters.sort, filters.dir` from the deps array — keep only `[benchmarkColumns]`. The eslint-disable comment that's already there acknowledges manual dep management, so removing the unused deps is consistent. Verify SortHeader doesn't actually use these values before removing.

**Suspect #3 — `<table style={{ minWidth: "1600px" }}>` inside `overflow-x-auto`**
- File: `src/components/tech/leaderboard-table.tsx:624` (and the mobile clone at `:708`)
- Pattern: Forces a ~1600px-wide GPU rasterization layer that gets fully invalidated end-to-end on every state change. Page wrapper is already `max-w-[1600px]` so the table is exactly viewport-wide on a maxed Mac display and slightly overflowing on anything narrower. The min-width is mostly redundant — Tailwind's grid + flex sizing already supports the column count.
- Fix: Try `min-width: 100%` first (most permissive). If layout breaks on narrow viewports (columns squish past usability), fall back to `min-width: 1280px` (laptop standard). Inspection in dev should confirm the right value before commit.

**Suspect #4 — `AudioPlayerProvider` context value is fresh object every render**
- File: `src/components/player/audio-player-provider.tsx:143-155`
- Pattern: `const value: AudioPlayerContextValue = { currentBeat, isPlaying, ... }` rebuilt every render, then `<AudioPlayerContext.Provider value={value}>` on line 158. Object identity changes every render → every context consumer re-renders.
- Impact: Consumers include `PlayerBar` (always mounted) and `WidgetNowPlaying` (always mounted in sidebar via `TileNav`). When audio plays, `currentTime` updates ~4× per second via `timeupdate`, but even when idle, ANY parent re-render (e.g., `usePathname` change in `(tech)/layout.tsx`, sidebar collapse, anything) cascades a fresh value object → re-renders `PlayerBar` and `WidgetNowPlaying` (which has its own `<canvas>` waveform per Suspect #8). Compounds with chip-click churn.
- Fix: Wrap `value` in `useMemo` with deps `[currentBeat, isPlaying, currentTime, duration, isMinimized, play, pause, toggle, seek, setMinimizedState]`. The audioRef is a stable ref; play/pause/toggle/seek/setMinimizedState are useCallback-wrapped already (verify) — if any are inline, wrap them in useCallback too.

### 🟡 MED suspect (worth fixing in this phase — small cost)

**Suspect #8 — `WidgetNowPlaying` always renders `<canvas>`**
- File: `src/components/tiles/widget-now-playing.tsx:46-54` (the `if (!currentBeat)` branch — the no-track-playing state STILL renders `<Waveform peaks={null} ... />`)
- Pattern: `<Waveform>` always mounts even when there's nothing to play. Canvas elements are GPU-backed surfaces on macOS even when "empty."

## Implementation Decisions (LOCKED)

### Methodology — baseline-first, then re-mount

The plan order is intentional and diagnostic:

1. **Plan 01** ships only the GPU/render-baseline fixes. Filter stays hidden. The sort header click in the table is the diagnostic interaction — sort uses the same `setFilters` write path that filter chips would, so if sort survives chip-click-equivalent churn, the baseline fixes worked.
2. **Plan 02** is a HUMAN GATE. User deploys to Vercel preview, opens `/tech/rankings/laptops` on real macOS Safari + macOS Firefox, clicks sort headers (BPR, GlitchMark, Year, Price, etc.) repeatedly. If the tab survives, baseline was the issue → proceed to re-mount filter. If the tab dies, baseline fixes weren't sufficient → STOP and investigate further (don't blindly continue to dropdown rebuild).
3. **Plan 03** re-mounts the existing `LeaderboardFilters` chip-bar as-is. No code changes to the chip-bar component itself. Just delete the atomic-fix comment block in `leaderboard-table.tsx` and let the existing components mount.
4. **Plan 04** is automated insurance — the Playwright crash-repro test on Webkit + Firefox catches DOM-leak / listener-leak / page-exception regression patterns. Cannot reproduce the actual macOS Metal crash (headless Linux clean), but catches the regression class.
5. **Plan 05** is the final HUMAN GATE. User clicks chips on real Safari + Firefox after Vercel preview. Phase complete only when user posts pass.
