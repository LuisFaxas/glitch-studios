<decisions>
## Implementation Decisions

### Architecture Constraints (LOCKED — debug artifact + roadmap)

- **NO URL state for filters.** No `nuqs`, no `router.replace`, no `router.push` from filter changes, no `usePathname`/`useSearchParams` reads in the filter render path. Filter state stays in local React state (the existing `useState<AllFilters>` pattern in `leaderboard-table.tsx`). Eliminated hypothesis from debug artifact: nuqs URL sync triggering router-sensitive re-renders; the user's tab churned because `usePathname` cascaded re-renders through the BottomTabBar shell on every chip click.
- **NO Base UI Popover** or other third-party popover/dropdown state machine (Floating UI / Headless UI / Radix Popover) for chip dropdowns. Eliminated hypothesis from debug artifact: Base UI Popover / Floating UI portals leaking listeners; replaced with hand-rolled CustomDropdown previously, bug persisted, but the constraint stands as architectural insurance.
- **NO framer-motion** in the filter subtree (no `<motion.*>`, no `<AnimatePresence>`). Eliminated hypothesis from debug artifact: framer-motion template wrapper causing route churn — replaced with plain div, bug persisted, constraint stands.
- **Bounded DOM mutations per chip click.** A single chip toggle must not re-mount the dropdown panel, the chip row, or the table header. Re-render is allowed (React reconciliation), but DOM node count for the filter subtree must stay constant (±2) across a chip toggle. This is the new architectural rule that catches the regression class the debug artifact was chasing — if we add any node-creating animation or portal-on-every-render, the Playwright leak guard fails.

### Hand-rolled Dropdown Primitive

- New file: `src/components/tech/filter-dropdown.tsx`
- Public API:
  ```tsx
  <FilterDropdown
    label={string}              // Trigger button label, e.g. "Year"
    activeCount={number}        // Selected-count badge on trigger; 0 hides badge
    onClose={() => void}        // Optional callback when panel closes
  >
    {/* panel content — chip rows, range inputs, etc. */}
  </FilterDropdown>
  ```
- State machinery (all local to `<FilterDropdown>`):
  - `const [open, setOpen] = useState(false)` — single boolean
  - Click-outside: `document.addEventListener('mousedown', handler)` attached only while `open === true`, removed on close
  - Escape: `document.addEventListener('keydown', handler)` attached only while `open === true`, removed on close
  - Focus return on close: cache the trigger element ref; on close, call `triggerRef.current?.focus()` if last interaction was keyboard
- Rendering:
  - Trigger: `<button type="button" aria-expanded={open} aria-haspopup="listbox">` — visual matches Phase 29.1 chip-bar trigger style
  - Panel: rendered as a sibling DOM node within the trigger's parent (no React portal). The parent already has `position: relative`; the panel uses `position: absolute; top: 100%; left: 0;` (or `right: 0` for right-aligned facets).
  - Mounting strategy: panel is **conditionally rendered** (`{open && <div>…</div>}`), NOT toggled with `display: none`. Bounded DOM mutations rule: when closed, the panel is gone; when open, it appears. Single mount/unmount per open cycle. The chip rows inside the panel toggle `aria-pressed` via class swap, NOT mount/unmount of the chip nodes.
- No portal, no transform animations, no transition (the panel appears instantly on open). If a fade is desired later, it must be CSS opacity-only with `transition: opacity 100ms` — never a node-mounting animation.

### Crash-Repro Test (Playwright)

- New test file: `tests/29.3-filter-chip-crash.spec.ts`
- Project coverage: must run on BOTH `webkit` and `firefox` Playwright projects (the engines that crashed on macOS). Chrome optional.
- Test flow:
  1. Navigate to `/tech/rankings/laptops` (assume dev server already running, mirror the harness pattern in `tests/forensics-overlay-leak.spec.ts` and `scripts/crash-repro.mjs`)
  2. Wait for the leaderboard table to render (assert at least one row)
  3. Open one filter dropdown (e.g., Year)
  4. Snapshot DOM node count of the filter subtree (`document.querySelector('[data-testid="leaderboard-filters"]').getElementsByTagName('*').length`)
  5. Click a chip 20× in rapid succession (no `await page.waitForTimeout` between clicks beyond what's needed for the click to register)
  6. Assertions:
     - No console errors thrown during the click batch
     - No unhandled exceptions on the page
     - Document still interactive (`document.readyState === 'complete'` and a follow-up assertion can interact with another element)
     - `aria-pressed` on the clicked chip ends in the expected toggled state (20 toggles → ends false if started false)
     - Row count in the table matches the expected filter outcome (proves the filter actually filtered)
     - DOM node count of the filter subtree after the batch is within ±2 of the snapshot (proves no leak)
- Add a `data-testid="leaderboard-filters"` attribute to the `<LeaderboardFilters>` root in the rewrite, so the test has a stable selector
