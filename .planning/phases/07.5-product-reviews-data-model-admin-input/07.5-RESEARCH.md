# Phase 07.5: Product reviews data model & admin input — Research

**Researched:** 2026-04-17
**Domain:** Drizzle schema design + admin form/editor surfaces
**Confidence:** HIGH

## Summary

Phase 07.5 delivers ~11 new tables (`tech_categories`, spec templates/fields, products with typed spec values, reviews with structured rating/pros/cons/media, benchmark templates/tests/runs) plus the admin UI to CRUD them and the Studios↔Tech context switcher in the sidebar. The UI-SPEC and CONTEXT have already resolved almost every architectural decision, so this research primarily confirms that the existing codebase supplies everything needed, identifies two small dependency gaps, and documents the exact existing patterns to mirror.

**Key discoveries driving the plan:**

1. **Tiptap — not markdown.** The existing blog editor at `src/components/admin/blog-post-form.tsx` uses **Tiptap outputting HTML**, not markdown. UI-SPEC confirms we reuse Tiptap for the review body. CONTEXT D-08/D-22 say "markdown" but UI-SPEC has authoritatively reinterpreted this as "rich text via Tiptap, rendered with live preview." Store review body as `text` containing HTML, matching `blogPosts.content`.
2. **dnd-kit is already installed and used.** `@dnd-kit/core`, `@dnd-kit/sortable`, `@dnd-kit/utilities` are in `package.json` and there's a working precedent in `src/components/admin/homepage-editor.tsx` (vertical sort with keyboard sensor). No new dependency needed. UI-SPEC Registry Safety table listed dnd-kit as "new dependency" — **outdated, it's already in.**
3. **Two shadcn components must be added via CLI:** `Command` (for the searchable category picker) and a server-rendered read-only Tiptap render helper (or we just use a second `useEditor` with `editable: false`). `vaul` is already installed so `Drawer` is usable.
4. **Postgres-js driver, not Neon serverless.** `src/lib/db.ts` uses `drizzle-orm/postgres-js`. Any new-table boilerplate should follow this pattern — do NOT import from `drizzle-orm/neon-serverless` even though STACK.md mentions Neon.
5. **Self-referencing FK requires `AnyPgColumn` annotation.** Drizzle 0.45's TypeScript inference breaks on self-references and returns `any`. Use `import { type AnyPgColumn } from "drizzle-orm/pg-core"` and annotate the references callback `(): AnyPgColumn => techCategories.id`.
6. **Permissions:** CONTEXT "Claude's Discretion" defaults any admin to `/admin/tech/*`. The middleware at `src/middleware.ts` already gates `/admin/:path*` for session presence. No middleware changes needed. Gate individual server actions via `requirePermission("manage_content")` (reuse existing permission — do NOT introduce a new permission enum in this phase; deferred).

**Primary recommendation:** Build in the order UI-SPEC's Implementation Notes dictate — schema → sidebar context switcher → categories tree → spec/benchmark template drawers → product form → review editor. Review editor is last because it's the heaviest build and depends on everything else.

## User Constraints (from CONTEXT.md)

### Locked Decisions

Copied verbatim from `07.5-CONTEXT.md <decisions>` block:

**Categories (3-level tree, admin-editable)**

- **D-01:** Categories form a **3-level tree**: Parent → Category → Subcategory. Example: `Computers > Laptops > Gaming Laptops`, `Audio > Headphones > Open-back`.
- **D-02:** Categories are **admin-editable** — you add/rename/reorder parents, categories, and subcategories from the admin dashboard. No hard-coded list in source.
- **D-03:** A dedicated **"Misc" parent** handles non-tech reviews. Example: `Misc > Furniture > Office Chairs`. Misc lives at the tree root just like Computers/Audio/etc. Tech homepage browsing can filter out Misc if desired (implementation detail for 07.6).

**Products & Specs (admin-defined templates per category)**

- **D-04:** Each category (typically at the leaf/subcategory level, but any level may define one) has an **admin-defined spec template**. A template is a named set of spec fields. Products within that category fill in those fields.
- **D-05:** Spec fields are **typed with units** — four field kinds: `text`, `number` with `unit`, `enum`/dropdown with admin-defined options, `boolean`.
- **D-05a:** A product has always-present base fields: `name`, `slug`, `hero_image_id`, `summary`, `price_usd` (nullable), `affiliate_url` (nullable), `manufacturer`, `release_date` (nullable), `category_id`, `created_at`, `updated_at`. Category-specific data comes from the spec template filled via join table.

**Reviews (structured editorial, multi-dimensional rating)**

- **D-06:** Multi-dimensional rating, fixed sitewide (not per-category). Four dimensions, each 1–10: Performance, Build quality, Value, Design/Aesthetics.
- **D-07:** Overall score auto-computed as average of the four dimensions. Admin can override with a manual overall score; auto-value is the suggested default.
- **D-08:** Every review has a verdict paragraph (2–4 sentence TL;DR, always visible at top) and a long-form body (markdown per CONTEXT, **reinterpreted as Tiptap HTML by UI-SPEC** — see §Design System, which is the authoritative contract).
- **D-09:** Pros & Cons lists — two bullet lists, rendered side-by-side. Admin inputs as structured arrays (not free-text parsed).
- **D-10:** Image gallery — one hero image (required) + optional secondary images. Images come from `mediaAssets` — no duplicate storage.
- **D-11:** Video embed (optional) — supports YouTube URL or a direct video media asset. Renders inline.
- **D-12:** "Who it's for" and "Who it's not for" audience callouts — two short paragraphs/bullet lists.
- **D-13:** Draft → Published workflow. Reviews have a `status` enum (`draft | published`). Publishing sets `published_at`; unpublishing reverts to `draft`.
- **D-14:** Reviewer = admin user. Byline attribution uses the existing `user` table. No separate `reviewers` table.

**Benchmarks (template-per-category structured scores)**

- **D-15:** Benchmark templates per category — similar to spec templates. A benchmark template defines a set of named tests with units. Categories without benchmarks simply have no template or an empty one.
- **D-15a:** A benchmark run is `{product_id, test_id, score, unit, recorded_at, notes}`. Multiple runs per test per product allowed.

**Admin Structure (hybrid context switcher)**

- **D-16:** `/admin` gets a Studios ↔ Tech context switcher. Two tabs, route-aware: `/admin/tech/*` auto-selects Tech, else Studios.
- **D-17:** Always-visible (shared): `Media`, `Clients/Users`, `Roles`, `Settings`.
- **D-18:** Studios-mode sidebar items (from current `/admin/*`): `Beats`, `Bundles`, `Services`, `Packages`, `Rooms`, `Availability`, `Bookings`, `Blog`, `Testimonials`, `Team`, `Newsletter`, `Inbox`.
- **D-19:** Tech-mode sidebar items: `Products`, `Reviews`, `Categories`, `Benchmarks`.
- **D-20:** Tech admin routes live at `/admin/tech/*`. Shared layout chrome, auth middleware, theme reused as-is.

**Admin UX Flows**

- **D-21:** Product creation uses a flat form with dynamic specs. Single page, no wizard. Saves as draft by default.
- **D-22:** Reviews use a dedicated full-page editor (Tiptap body + live preview per UI-SPEC, verdict textarea, 4 rating sliders, pros/cons rows, image picker, video URL, audience textareas, reviewer dropdown, draft/publish toggle, autosave ~10s).
- **D-23:** Categories managed via a tree view (collapsible, drag-to-reorder, inline rename, add-child). Deleting a category with products requires confirmation OR blocks if `onDelete=restrict`.
- **D-24:** Spec & benchmark templates managed per-category via modal/drawer. UI-SPEC §Placement resolves this to **drawer from the right**.

### Pattern 1: Server Actions with `requirePermission`

**What:** All admin mutations are Server Actions (not API routes). Each checks `requirePermission` at the top.
**When to use:** Every create/update/delete in `src/actions/admin-tech-*.ts`.
**Example:**

```typescript

import { requirePermission } from "@/lib/permissions"
import { revalidatePath } from "next/cache"

export async function createTechProduct(data: ProductFormData) {
  const session = await requirePermission("manage_settings")  // Any admin can manage Tech per CONTEXT discretion
  // ... validate with zod, insert, returning id
  revalidatePath("/admin/tech/products")
  return inserted.id
}
```

**Permission choice:** Use `manage_settings` for catalog schema ops (categories, templates) — they shape the public data model. Use `manage_content` for reviews/products — content ops. Do **not** introduce a new permission enum value in 07.5; deferred per CONTEXT "Claude's Discretion."

### Pattern 3: Drizzle self-reference via `AnyPgColumn`

**What:** `tech_categories.parent_id` references `tech_categories.id`. Drizzle TS inference returns `any` without an explicit annotation.
**When to use:** Category tree schema.
**Example:**

```typescript
// Source: https://gebna.gg/blog/self-referencing-foreign-key-typescript-drizzle-orm (verified Drizzle Team pattern)
import { pgTable, uuid, text, integer, timestamp } from "drizzle-orm/pg-core"
import type { AnyPgColumn } from "drizzle-orm/pg-core"

export const techCategories = pgTable("tech_categories", {
  id: uuid("id").defaultRandom().primaryKey(),
  name: text("name").notNull(),
  slug: text("slug").notNull().unique(),
  parentId: uuid("parent_id").references((): AnyPgColumn => techCategories.id, {
    onDelete: "restrict",  // D-23: block delete if children exist
  }),
  level: integer("level").notNull(),  // 0=parent, 1=category, 2=subcategory — convenience for CHECK constraints + sidebar filtering
  sortOrder: integer("sort_order").notNull().default(0),  // position within siblings
  createdAt: timestamp("created_at").defaultNow().notNull(),
  updatedAt: timestamp("updated_at").defaultNow().notNull(),
})
```

**Sort-order strategy:** Single `sort_order` column scoped to siblings (within same `parent_id`). When reordering, update all affected siblings' `sort_order` values. Planner may use gap numbering (100, 200, 300) to make most inserts avoid full re-numbering; periodic compaction is trivial.

### Pattern 5: Autosave with debounce + periodic timer

**What:** Review editor saves on keystroke (2s debounce) + every 30s while focused + on blur of title/verdict.
**When to use:** Only the review editor (D-22). Product form + templates use manual Save only (simpler, less risk).
**Example skeleton:**

```typescript
"use client"
import { useEffect, useRef, useState, useTransition } from "react"

export function useAutosave<T>(
  data: T,
  save: (value: T) => Promise<void>,
  opts = { debounceMs: 2000, periodMs: 30000 }
) {
  const [state, setState] = useState<"idle" | "saving" | "saved" | "error">("idle")
  const [lastSavedAt, setLastSavedAt] = useState<Date | null>(null)
  const [isPending, startTransition] = useTransition()
  const debounceRef = useRef<ReturnType<typeof setTimeout>>()
  const periodicRef = useRef<ReturnType<typeof setInterval>>()
  const latestData = useRef(data)
  latestData.current = data

  const flush = () => {
    setState("saving")
    startTransition(async () => {
      try {
        await save(latestData.current)
        setState("saved")
        setLastSavedAt(new Date())
      } catch {
        setState("error")
      }
    })
  }

  // Debounced save on change
  useEffect(() => {
    setState("idle")  // "Unsaved changes"
    if (debounceRef.current) clearTimeout(debounceRef.current)
    debounceRef.current = setTimeout(flush, opts.debounceMs)
    return () => clearTimeout(debounceRef.current!)
  }, [data])

  // Periodic 30s flush
  useEffect(() => {
    periodicRef.current = setInterval(flush, opts.periodMs)
    return () => clearInterval(periodicRef.current!)
  }, [])

  return { state, lastSavedAt, flushNow: flush, isPending }
}
```

**Status transitions are NOT autosaved.** Publishing/unpublishing is an explicit button click that calls a separate server action (`publishReview(id)` / `unpublishReview(id)`).

### Anti-Patterns to Avoid

- **Storing spec values as JSON blob on product row.** Breaks SQL sort/filter for 07.7 compare page. Use a normalized `tech_product_specs` join table with `(product_id, field_id, value_text, value_number, value_boolean, value_enum_option_id)` — one typed value column per field kind. Product spec values live in the kind-matching column.
- **Storing Tiptap content as markdown string.** Do not fight the existing pattern. Tiptap outputs HTML; mirror `blogPosts.content: text("content").notNull()`.
- **Putting review autosave in a separate long-running API endpoint.** Use server actions per project precedent. Race-condition handling = last-write-wins with a single-admin assumption (explicitly stated in UI-SPEC §Autosave behavior).
- **Introducing `react-hook-form`.** Not in the project. Existing forms use `useState` + manual validation with zod. Keep consistent.
- **Introducing `@tanstack/react-table`.** STACK.md lists it but it's NOT installed. Existing admin tables are hand-rolled. Follow the existing pattern for `products`, `reviews`, `benchmarks` list tables.
- **Adding a new permission enum value.** Out of scope for 07.5. Reuse `manage_content` and `manage_settings`.
- **`onDelete: "cascade"` on category → products.** Use `restrict` per D-23 — admin must explicitly reassign or delete products before deleting a category.

## Common Pitfalls

### Pitfall 1: Storing Tiptap body as "markdown" per CONTEXT D-08

**What goes wrong:** Planner sees "markdown body" in CONTEXT, installs `react-markdown`, writes new renderer, ends up with two content formats in the codebase.
**Why it happens:** CONTEXT D-08 uses the word "markdown" but UI-SPEC overrides to Tiptap HTML (see UI-SPEC §Known Constraints: "Tiptap content is stored as HTML (consistent with existing blog posts); NOT markdown string. 'Markdown editor with preview' in CONTEXT D-22 is interpreted as 'rich-text editor (Tiptap) with live preview'"). UI-SPEC is the visual/interaction contract — it wins.
**How to avoid:** Store `review.body_html: text()` and treat it exactly like `blogPosts.content`.
**Warning signs:** Any code importing `react-markdown`, `remark`, `rehype`, or `marked`.

### Pitfall 2: Drizzle self-reference inferring `any`

**What goes wrong:** `techCategories.parentId` references `techCategories.id`; the generated type for `techCategories` is `any`, losing all type safety on queries.
**Why it happens:** TypeScript cannot resolve circular references at inference time — Drizzle's `references()` callback runs during type resolution.
**How to avoid:** Annotate the callback return type: `.references((): AnyPgColumn => techCategories.id)`. Import `AnyPgColumn` from `drizzle-orm/pg-core`.
**Warning signs:** `tsc` reports "Type 'any' is not assignable to..." or IDE autocomplete fails on `techCategories.*`.

### Pitfall 3: Attempting to enforce 3-level depth via FK constraint alone

**What goes wrong:** SQL FK cannot express "parent_id must point to a node at exactly level-1." Rogue inserts create 4+ level trees.
**Why it happens:** Relational constraints are flat.
**How to avoid:** Add a `level` column (0|1|2) plus (a) DB `CHECK (level BETWEEN 0 AND 2)` constraint and (b) server action validation that sets `level = parent.level + 1` on insert and rejects level > 2. UI tree view also prevents 4th-level adds by hiding the "+ Add subcategory" affordance on level-2 nodes.
**Warning signs:** A category appears under a subcategory during testing.

### Pitfall 4: Benchmark "highest score" mis-ordering on non-higher-is-better metrics

**What goes wrong:** "Battery life (hours)" higher = better ✓. But "Weight (kg)" lower = better, and UI-SPEC says "product spotlight displays the highest/most-recent by default" (D-15a).
**Why it happens:** Not all benchmark tests have "higher = better" semantics.
**How to avoid:** Add `direction: pgEnum("higher_is_better", "lower_is_better")` column on `tech_benchmark_tests`. Sort spotlight by the winning direction. Keep "most-recent" as tiebreaker.
**Warning signs:** Product with a heavy weight shown as "top" benchmark.

### Pitfall 5: Uploaded orphan media when admin abandons a draft review

**What goes wrong:** Admin uploads hero image + 5 gallery images, closes the tab before any save → images live in R2 forever, review never existed.
**Why it happens:** `MediaPickerDialog` selects from `mediaAssets` which are uploaded before being attached.
**How to avoid:** Accept this — it's already the existing blog editor behavior. Don't build orphan-cleanup logic in 07.5; defer to a future "media library maintenance" task. Media library already shows all assets; admin can delete manually.
**Warning signs:** Storage costs grow linearly with review drafts.

### Pitfall 6: Context switcher nav flickers on path change

**What goes wrong:** User navigates from `/admin/tech/products` to `/admin/beats`; sidebar shows Tech sections for a beat before React re-renders with Studios sections.
**Why it happens:** RSC navigation is fast but sections computed client-side from `pathname` may trail by one tick.
**How to avoid:** `AdminContextSwitcher` is client component. Sidebar section rendering also reads `pathname` and recomputes per render — no extra memoization. Framer Motion `AnimatePresence` with 150ms `initial={{ opacity: 0 }}` → `animate={{ opacity: 1 }}` smooths any visible lag per UI-SPEC.
**Warning signs:** Visible nav-item mismatch during route transitions.

### Pitfall 7: Autosave fires during product category change with filled specs

**What goes wrong:** Admin fills spec values, changes category mid-draft, autosave fires with mismatched specs. D-21 warning "Changing category will clear spec values" only shows on manual Save.
**Why it happens:** Autosave runs on every debounced change.
**How to avoid:** **Product form does NOT autosave.** Only the review editor autosaves (per D-22). Product form uses explicit Save Draft / Save buttons. This is consistent with UI-SPEC which specifies autosave only for ReviewEditor.
**Warning signs:** Product spec column in DB shows values belonging to a prior category.

### Pitfall 8: zod v3 syntax in generated code (when actual installed is v4)

**What goes wrong:** Planner references STACK.md's "zod 3.x" and writes `z.string().optional().default("")` which works in both, but also writes `z.record(z.string())` which differs subtly.
**Why it happens:** STACK.md is stale; project uses zod 4.3.6 (verified in `package.json`).
**How to avoid:** Match existing patterns in `src/components/admin/blog-post-form.tsx` line 12. When in doubt, check zod v4 docs.
**Warning signs:** Type errors at Server Action boundary.

}) {
  await requirePermission("manage_content")

  const [product] = await db.insert(techProducts).values(data.base).returning({ id: techProducts.id })

  // Load field definitions to route values into the typed column
  const fields = await db.query.techSpecFields.findMany({
    where: (f, { inArray }) => inArray(f.id, data.specs.map(s => s.fieldId)),
  })
  const fieldById = new Map(fields.map(f => [f.id, f]))

  for (const spec of data.specs) {
    const f = fieldById.get(spec.fieldId)
    if (!f) continue
    await db.insert(techProductSpecs).values({
      productId: product.id,
      fieldId: spec.fieldId,
      valueText: f.type === "text" || f.type === "enum" ? String(spec.value) : null,
      valueNumber: f.type === "number" ? String(spec.value) : null,
      valueBoolean: f.type === "boolean" ? Boolean(spec.value) : null,
    })
  }
  revalidatePath("/admin/tech/products")
  return product.id
}

## Open Questions

1. **Should tech review drafts be visible to the assigned reviewer when they're NOT the creator?**
   - What we know: D-14 says the creator is the default reviewer and can re-assign.
   - What's unclear: If admin A drafts a review and assigns admin B as reviewer, does B see it in their own draft list?
   - Recommendation: All admins see all drafts (simpler, matches existing `manage_content` model). Defer per-user draft scoping until multi-editor workflow is a real need.

2. **Review URL slug uniqueness — globally unique or per-product unique?**
   - What we know: `blogPosts.slug` is globally unique. Reviews attach to products.
   - What's unclear: If two reviews exist for the same product over time (product v1 + product v2 refreshed), do they conflict?
   - Recommendation: Globally unique — matches existing pattern. Slug format `{product-slug}-review` or `{product-slug}-review-v2`. Planner adds slug-collision handling in server action (append suffix).

3. **Benchmark template auto-seed on category create?**
   - UI-SPEC Benchmark Template Editor lists starter values for Laptops. Should creating a new Laptop-descendant category auto-populate these?
   - Recommendation: **Do not auto-seed.** Admin clicks "Add test" — explicit is better. Deferred seed behavior = less magic, less to undo.

4. **Hero image aspect ratio constraint for products/reviews?**
   - UI-SPEC does not specify.
   - Recommendation: No enforcement in schema or server action — store whatever `mediaAssets` resolves. 07.6 public display renders via Next.js Image with `aspect-ratio` CSS.

5. **Stub seeding for 07.4 placeholder products (MacBook Pro M4 Max, ROG G14, Framework 16)?**
   - CONTEXT `<specifics>` lists them.
   - Recommendation: **Do NOT seed in 07.5.** Seed data is code and competes with real admin input. The admin UI should feel functional from an empty state (category tree empty → add parent → add subcategory → add spec template → add product → write review). Planner may add a separate "demo content" Wave at the end if user requests. Default answer: empty DB, admin creates real content through the UI.
