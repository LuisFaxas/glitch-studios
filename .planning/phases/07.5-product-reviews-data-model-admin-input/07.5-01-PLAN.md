Notes on the design decisions baked in (all from RESEARCH.md):
- `parentId.references((): AnyPgColumn => techCategories.id, ...)` — preserves Drizzle TS inference (Pitfall 2)
- `onDelete: "restrict"` on category parent and product category — prevents accidental cascade delete (D-23, Pitfall 7)
- `onDelete: "cascade"` on spec template → fields, review → pros/cons/gallery — children are meaningless without parent
- `tech_product_specs` is a normalized join table with one typed value column per field kind (Research §Don't Hand-Roll) — enables 07.7 compare-page SQL sort
- `bodyHtml` text column — stores Tiptap HTML, mirrors `blogPosts.content` (Pitfall 1)
- `reviewerId` is `text` (not uuid) — Better Auth `user.id` is text
- `direction` on benchmark tests — Pitfall 4 (some tests are lower-is-better like weight)
- `overallOverride` is `numeric(3,1)` nullable — null means "use auto-computed average"
- `tech_review_gallery` is a separate table (not an array column) — allows sort_order and media FK
