# Phase 07.5: Product reviews data model & admin input - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers the **data model** and **admin input surfaces** for Glitch Tech's product review vertical:

- DB schema for categories (3-level tree), products (with per-category spec templates), reviews (structured editorial with multi-dimensional ratings), benchmarks (template-per-category structured tests), and supporting tables
- Admin pages to CRUD all of the above (`/admin/*` — see D-16 for exact structure)
- The admin's **hybrid context switcher** (Studios ↔ Tech tabs) that gates which sidebar items appear per mode

**Explicitly out of scope (deferred to 07.6 / 07.7):**
- Public display pages (`/tech/reviews/[slug]`, `/tech/categories/*`, category listing pages) — 07.6
- Comparison tool UI (`/tech/compare` side-by-side tables) — 07.7
- Real editorial content (reviewers can draft after this phase; publishing real reviews is ongoing work, not a phase milestone)

</domain>

<decisions>
## Implementation Decisions

### Categories (3-level tree, admin-editable)

- **D-01:** Categories form a **3-level tree**: Parent → Category → Subcategory. Example: `Computers > Laptops > Gaming Laptops`, `Audio > Headphones > Open-back`.
- **D-02:** Categories are **admin-editable** — you add/rename/reorder parents, categories, and subcategories from the admin dashboard. No hard-coded list in source.
- **D-03:** A dedicated **"Misc" parent** handles non-tech reviews. Example: `Misc > Furniture > Office Chairs`. Misc lives at the tree root just like Computers/Audio/etc. Tech homepage browsing can filter out Misc if desired (implementation detail for 07.6).

### Products & Specs (admin-defined templates per category)

- **D-04:** Each category (typically at the leaf/subcategory level, but any level may define one) has an **admin-defined spec template**. A template is a named set of spec fields. Products within that category fill in those fields.
- **D-05:** Spec fields are **typed with units** — four field kinds:
  - `text` — free-form string (e.g., "CPU name: Intel i9-13900H")
  - `number` with `unit` — numeric with a display unit (e.g., "RAM: 16 GB", "Weight: 1.5 kg")
  - `enum` / dropdown — admin-defined option set (e.g., "Display type: OLED | IPS | TN")
  - `boolean` — yes/no (e.g., "Mechanical keyboard: true")
  This lets the compare page (07.7) sort numerically and group by enum value correctly.
- **D-05a:** A **product** has always-present base fields independent of category: `name`, `slug`, `hero_image_id`, `summary`, `price_usd` (nullable), `affiliate_url` (nullable), `manufacturer`, `release_date` (nullable), `category_id`, `created_at`, `updated_at`. Category-specific data comes from the spec template filled via join table.

### Reviews (structured editorial, multi-dimensional rating)

- **D-06:** Reviews use a **multi-dimensional rating**, fixed sitewide (not per-category). Four dimensions, each 1–10:
  1. **Performance**
  2. **Build quality**
  3. **Value**
  4. **Design / Aesthetics**
- **D-07:** **Overall score** is auto-computed as the average of the four dimensions. Admin can override with a manual overall score if desired (the auto-value is shown as a suggested default).
- **D-08:** Every review has a **verdict paragraph** (2–4 sentence TL;DR, always visible at top of the review page) and a **long-form body in markdown** (main review content).
- **D-09:** **Pros & Cons lists** — two bullet lists, rendered side-by-side. Admin inputs as structured arrays (not free-text parsed).
- **D-10:** **Image gallery** — one hero image (required) + optional secondary images. Images come from the existing media library (`mediaAssets` table — no duplicate storage).
- **D-11:** **Video embed** (optional) — supports YouTube URL or a direct video media asset. Renders inline in the review page.
- **D-12:** **"Who it's for" and "Who it's not for"** audience callouts — two short paragraphs (or bullet lists) admin-inputs. Rendered as callout boxes on the review page.
- **D-13:** **Draft → Published workflow.** Reviews have a `status` enum (`draft | published`). Drafts are invisible to the public but previewable by the admin at a `?preview=1` or admin-only route. Publishing sets `published_at` timestamp; unpublishing reverts to `draft`.
- **D-14:** **Reviewer = admin user.** Byline attribution uses the existing `user` table (Better Auth). No separate `reviewers` table. The admin who creates the review is the default author; admin can re-assign to another admin user via a dropdown.

### Benchmarks (template-per-category structured scores)

- **D-15:** **Benchmark templates per category** — similar to spec templates. A benchmark template defines a set of named tests with units (e.g., Laptop benchmark template: `Geekbench 6 Multi (score)`, `Cinebench R23 (score)`, `3DMark Wild Life (score)`, `Battery life (hours)`). Products in that category fill in those test scores. Categories without benchmarks (chairs, coffee makers) simply have no template or an empty one — the benchmark section on the review/product page is skipped.
- **D-15a:** A benchmark run is `{product_id, test_id, score, unit, recorded_at, notes}`. Multiple runs per test per product are allowed (e.g., "Geekbench 6 Multi, Balanced profile: 14,200" vs "Performance profile: 16,800"). The product spotlight displays the highest/most-recent by default.

### Admin Structure (C — Hybrid context switcher)

- **D-16:** `/admin` gets a **Studios ↔ Tech context switcher** at the top of the sidebar. Two tabs (pills/chips) that switch which sidebar sections are visible. The active tab is **route-aware**: URL prefix `/admin/tech/*` auto-selects the Tech tab, anything else selects Studios.
- **D-17:** **Always-visible** (shared, below the context switcher, rendered in both modes): `Media`, `Clients/Users`, `Roles`, `Settings`. These are cross-domain concerns.
- **D-18:** **Studios-mode sidebar items** (from current `/admin/*`): `Beats`, `Bundles`, `Services`, `Packages`, `Rooms`, `Availability`, `Bookings`, `Blog`, `Testimonials`, `Team`, `Newsletter`, `Inbox`.
- **D-19:** **Tech-mode sidebar items:** `Products`, `Reviews`, `Categories`, `Benchmarks`. (Future additions like `Comparisons`, `Affiliate links` slot here in later phases.)
- **D-20:** Tech admin routes live at `/admin/tech/*` (not a separate `/tech-admin`). Shared layout chrome (page header, auth middleware, theme) reused as-is. Only the sidebar's section list changes per tab.

### Admin UX Flows

- **D-21:** **Product creation** uses a **flat form with dynamic specs**. Single page: name, category (dropdown triggers spec fields), hero image, summary, price, affiliate URL, then the spec template fields appear below based on selected category. Saves as draft. No wizard.
- **D-22:** **Reviews** use a **dedicated full-page editor** with:
  - Markdown body with live preview pane
  - Verdict paragraph input (textarea)
  - Rating dimension sliders (4 dimensions, 1–10 each, overall auto-computed and displayed)
  - Pros/Cons structured inputs (add/remove rows)
  - Image picker (selects from existing media library)
  - Video URL field (optional)
  - "Who it's for" / "Who it's not for" textareas
  - Reviewer assignment dropdown
  - Draft/Publish toggle
  - **Autosave** every ~10s to prevent data loss
- **D-23:** **Categories** managed via a **tree view** — collapsible nodes, drag-to-reorder children under a parent, inline rename, add-child button. Deleting a category with products requires confirmation (or blocks if `onDelete=restrict`).
- **D-24:** **Spec & benchmark templates** managed per-category — when editing a category, admin sees "Edit spec template" and "Edit benchmark template" buttons that open modal/drawer with field list (add/remove/reorder fields, set type + unit + enum options).

### Claude's Discretion

The planner/executor can make calls on these without re-asking:
- Exact table names and column naming conventions (follow existing schema.ts patterns: snake_case, camelCase in Drizzle exports, `id uuid primary key default random_uuid()`, `created_at`/`updated_at` timestamps on all tables)
- Markdown editor library choice (existing blog editor pattern if one exists; otherwise a lightweight MDX editor or `react-markdown` + textarea preview)
- Image picker UI (reuse whatever the admin beat/blog editors use for media selection)
- How autosave is implemented (debounced server action, WebSocket, or periodic PUT)
- Tab switcher visual (chip group, segmented control, or underlined tabs — should feel native to shadcn/ui)
- Route middleware for role gating (can Tech editors who aren't full admins access `/admin/tech/*` only? Decide based on existing role infra — default: any admin can manage Tech unless we get feedback otherwise)

</decisions>
