## Summary

Phase 07.4 ships the architectural shell for Glitch Tech as a sibling route group `(tech)/` alongside the existing `(public)/` and `(auth)/` groups under the **existing single root layout** at `src/app/layout.tsx`. Because the root layout already hosts `AudioPlayerProvider`, `CartProvider`, `TooltipProvider`, and `CartDrawer` — and Next.js explicitly preserves shared layouts and their React state during client-side `<Link>` navigation — no provider migration is needed. Cross-entity navigation between `/` and `/tech` will be seamless out of the box, and audio/cart/auth state will survive the crossover for free.

The primary open design questions are: (1) how to achieve a "glitch flicker" page-transition animation without the FrozenRouter hack, (2) how to scope CSS variables per route group with Tailwind v4, and (3) how to reuse `tile-nav.tsx` with a different config without refactoring every call site. The planner should treat items 1–3 as architectural decisions that must be made up front — they touch multiple tasks.

**Primary recommendation:** Add `(tech)/` as a sibling route group under the existing root layout. Do NOT introduce a second root layout (that would force a full page reload at the `/` ↔ `/tech` boundary and break the D-13 seamless-navigation requirement). Use `template.tsx` (root) for the zone-shift transition animation instead of the FrozenRouter pattern. Parameterize `tile-nav.tsx` to accept a `navItems` prop + optional widget slots instead of hard-coding `publicNavItems`.

## Route Group Architecture (HIGH confidence — verified from Next.js 16 docs)

### Core rules (from Next.js 16.2.4 official docs)

1. **Single root layout, multiple sibling groups:** The existing `src/app/layout.tsx` IS the root layout. All existing groups (`(public)/`, `(auth)/`) already share it. Adding `(tech)/` as a sibling means they ALL share one root — this is the normal pattern.
2. **Layouts do not re-render on navigation.** Next.js doc: "Layouts are cached in the client during navigation to avoid unnecessary server requests... they can be cached and reused to avoid unnecessary computation when navigating between pages." Applies to root AND route-group layouts (`(public)/layout.tsx` is NOT re-instantiated if you navigate `/beats` → `/services` because both are in the same group).
3. **Crossing route groups under a shared root layout keeps the root layout mounted.** This is why `AudioPlayerProvider` at root will keep audio state — the provider never unmounts.
4. **Multiple ROOT layouts trigger full page reload.** Next.js doc (route-groups page, "Caveats"): *"If you navigate between routes that use different root layouts, it'll trigger a full page reload."* **This is only an issue if `src/app/layout.tsx` is deleted and each route group defines its own root.** Our plan does NOT do that — we keep the single root. ✓
5. **Conflicting paths are disallowed.** `(public)/about/page.tsx` and `(tech)/about/page.tsx` both resolve to `/about` — Next.js will error at build. We must namespace Tech routes under `/tech/*`. ✓ (all Tech routes specified in D-02 live under `/tech`).

## Cross-Group Navigation State Preservation (HIGH confidence)

**The question:** When a user clicks the "GLITCH TECH" cross-link tile on `/` and navigates to `/tech`, what persists and what doesn't?

| Item | Persists across `/` → `/tech`? | Why |
|------|-------------------------------|-----|
| Root layout (`html`, `body`, fonts) | YES | Next.js: layouts cached during navigation. |
| `AudioPlayerProvider` state (current track, play head, queue) | YES | Provider mounted at root — never unmounts during client-side nav. |
| `CartProvider` state (cart items) | YES | Same reason. |
| `TooltipProvider` | YES | Same. |
| Better Auth session (`useSession`) | YES | Session is fetched; client cache per request. Doesn't reset on client navigation. |
| `(public)/layout.tsx` state (e.g., `SidebarProvider`) | NO — unmounts | Different route group; layout boundary crossed. Expected — Tech has its own sidebar. |
| `template.tsx` (if present) | **REMOUNTS** on every navigation (any route change) | Template is remounted by design — this is where page-transition animations go. |

**Source:** Next.js 16 docs, "Linking and Navigating" page: *"Client-side transitions … Keeping any shared layouts and UI. Replacing the current page with the prefetched loading state or a new page if available."*

**Implication for D-13 seamless nav:** This is automatic. No provider migration, no hack. The only work is to move the PlayerBar UI component up to root (see Existing Root Layout Analysis). ✓

## Page Transition Animation (MEDIUM-HIGH confidence — pattern verified, need to adapt existing code)

### Existing state (from `src/app/template.tsx`)

```tsx
"use client"
import { motion } from "motion/react"
import { usePathname } from "next/navigation"

export default function Template({ children }: { children: React.ReactNode }) {
  const pathname = usePathname()
  return (
    <motion.div
      key={pathname}
      initial={{ opacity: 1 }}
      animate={{ opacity: 1 }}
      transition={{ duration: 0.15, delay: 0.05 }}
      className="relative w-full"
    >
      <motion.div
        className="pointer-events-none absolute inset-0 z-50 animate-glitch-hover"
        initial={{ opacity: 0.6 }}
        animate={{ opacity: 0 }}
        transition={{ duration: 0.2 }}
        aria-hidden="true"
      />
      {children}
    </motion.div>
  )
}
```

The project already has a glitch-flash template at root. It runs on **every** navigation (any pathname change). **D-06 wants a MORE pronounced effect specifically on cross-group transitions** (`/` ↔ `/tech`), while keeping the subtle existing effect for within-group nav.

### Recommended approach for D-06

**Use the existing `template.tsx` with a pathname-derived `isCrossBrand` flag.**

```tsx
"use client"
import { motion } from "motion/react"
import { usePathname } from "next/navigation"
import { useRef } from "react"

function useIsCrossBrandTransition() {
  const pathname = usePathname()
  const prevRef = useRef(pathname)
  const prev = prevRef.current
  prevRef.current = pathname

  const prevIsTech = prev.startsWith("/tech")
  const currIsTech = pathname.startsWith("/tech")
  return prevIsTech !== currIsTech
}

export default function Template({ children }: { children: React.ReactNode }) {
  const pathname = usePathname()
  const isCross = useIsCrossBrandTransition()
  const flickerDuration = isCross ? 0.35 : 0.2
  const flickerOpacity = isCross ? 0.85 : 0.6

  return (
    <motion.div
      key={pathname}
      initial={{ opacity: isCross ? 0 : 1 }}
      animate={{ opacity: 1 }}
      transition={{ duration: flickerDuration, ease: [0.215, 0.61, 0.355, 1] }}
      className="relative w-full"
    >
      <motion.div
        className="pointer-events-none absolute inset-0 z-50 animate-glitch-hover"
        initial={{ opacity: flickerOpacity }}
        animate={{ opacity: 0 }}
        transition={{ duration: flickerDuration, ease: [0.215, 0.61, 0.355, 1] }}
        aria-hidden="true"
      />
      {children}
    </motion.div>
  )
}
```

**Why this is better than the FrozenRouter pattern:**
1. `template.tsx` already remounts on every navigation — Next.js gives it a unique key per navigation target. No hacky `LayoutRouterContext` access needed.
2. The UI-SPEC proposes `AnimatePresence mode="wait"` with `key={pathname}`, but **AnimatePresence inside a template.tsx would require exit animations** — which the FrozenRouter pattern is specifically designed to enable (and is fragile). By using only `initial` → `animate` on a template that already remounts, we get the entrance animation cleanly.
3. No dependency on Next.js internals (`next/dist/shared/lib/app-router-context.shared-runtime` is undocumented and may change between versions).

### Reduced motion

`prefers-reduced-motion` must zero-out the glitch flicker and collapse to a short fade. The existing `animate-glitch-hover` CSS class already respects `prefers-reduced-motion` (see `logo-tile.module.css` line 166). Verify new code paths in template also honor `useReducedMotion()` or equivalent.

### FrozenRouter alternative — reject unless needed

The FrozenRouter pattern ([reference](https://www.imcorfitz.com/posts/adding-framer-motion-page-transitions-to-next-js-app-router)) is needed when you want **overlapping exit + entrance animations** (exit page fades out while new page fades in). It works by freezing the previous layout's router context during the exit animation.

**We don't need this.** D-06 describes a fade + flicker = *entrance-only* animation. The outgoing page instantly disappears, the flicker flashes, the incoming page appears. That's what `template.tsx` already does. Avoid the FrozenRouter.

**Only reconsider FrozenRouter if** the UI-SPEC is upgraded to require the outgoing page to "glitch out" visually before the incoming page arrives. For Phase 07.4 locked scope, it's not needed.

## Tailwind v4 Theme Scoping for Tech Accent (MEDIUM confidence — well-supported pattern, but team's Tech accent is deferred)

### What the UI-SPEC says

*"Phase 7.4 ships Tech with identical monochrome palette to Studios… A visual accent for Tech (D-05 mentions 'subtle accent variant') should NOT launch in 7.4."*

**So for 07.4 itself, no accent is needed.** But D-05 locks "subtle accent variant" as the eventual direction. The planner should **lay the groundwork in 7.4** so 7.5/7.6 can add the accent without refactoring.

### Recommended scaffolding (Tailwind v4 + CSS custom properties)

Tailwind v4 supports `@theme` at file top-level, but DOES NOT allow nesting `@theme` inside selectors. For per-route-group theming, the supported pattern is:

1. **Base theme vars at `:root`** (existing in `globals.css`).
2. **Scoped overrides via a data attribute selector** using plain CSS (not `@theme`):

```css
/* src/styles/globals.css, after existing @theme blocks */

/* Tech variant accent — scoped to elements inside a data-brand="tech" container */
[data-brand="tech"] {
  --color-accent: #00aa00;         /* terminal green (UI-SPEC suggestion) */
  --color-accent-soft: #003311;
}
```

3. **Mount the attribute on the Tech route-group's layout root:**

```tsx
// src/app/(tech)/layout.tsx
export default function TechLayout({ children }: { children: React.ReactNode }) {
  return <div data-brand="tech" className="min-h-screen bg-black">{/* ... */}</div>
}
```

4. **Consume via arbitrary values in Tailwind:**

```tsx
<div className="border-[var(--color-accent)]">...</div>
```

### Why NOT `@custom-variant`

`@custom-variant tech (&:where([data-brand="tech"] *))` creates a Tailwind variant prefix (e.g., `tech:text-green-500`). This is useful when you want the SAME utility with a conditional override — but since we want the accent tokens themselves to change (the monochrome palette stays the same; only accent usages need the green), the CSS-variable-override approach is simpler and more maintainable.

**Reserve `@custom-variant` for cases where the same utility class must behave differently in Tech context** (e.g., if we wanted `text-primary` to be white in Studios but green in Tech, and we specifically need a class-based switch). For Phase 07.4 the variable override pattern is the right default.

### Phase 07.4 action

- Ship the `data-brand="tech"` wrapper on `(tech)/layout.tsx` — empty effect, future-proofs 7.5/7.6 accent introduction.
- Do NOT define any accent CSS variables yet (7.4 is monochrome-only per UI-SPEC).
- Add a 3-line comment in `globals.css` explaining the `[data-brand="tech"]` hook for future devs.

## Common Pitfalls

### Pitfall 1: Accidentally creating a second root layout
**What goes wrong:** Deleting `src/app/layout.tsx` or forgetting that `(tech)/layout.tsx` is a segment layout (not a root layout) — leading to full-page reloads at `/` ↔ `/tech` boundary.
**Why it happens:** The mental model "each entity gets its own everything" incorrectly suggests each group defines its own `<html>`/`<body>`. Next.js docs say multiple root layouts force full page reloads.
**How to avoid:** Keep the existing `src/app/layout.tsx` as the single root. `(tech)/layout.tsx` must NOT render `<html>` or `<body>` — it's a segment layout. Only one `html` element in the entire app directory tree.
**Warning signs:** Full page flash on cross-link click; audio cuts out; cart resets; hydration warnings about duplicate `<body>`.

### Pitfall 2: `SidebarProvider` scoping mistake
**What goes wrong:** Both Studios and Tech share the same `SidebarProvider` instance at root, and Tech's "collapsed" state bleeds into Studios (or vice versa).
**Why it happens:** Hoisting provider too high without thinking through per-group state.
**How to avoid:** Keep `SidebarProvider` per route-group layout (as it already is via `HomepageSidebarController`). Each `(public)/layout.tsx` and `(tech)/layout.tsx` gets its own. The provider should live BELOW the root.
**Warning signs:** Sidebar collapse state changes when crossing brands; sidebar width "remembers" previous entity's state.

### Pitfall 3: Player bar not visible on Tech pages
**What goes wrong:** User starts a beat on `/beats`, navigates to `/tech`, beat keeps playing (because `AudioPlayerProvider` is at root) but the player bar UI disappears (because `<PlayerBar />` component is rendered only in `(public)/layout.tsx`).
**Why it happens:** State preservation ≠ UI rendering. These are independent.
**How to avoid:** Move `<PlayerBar />` rendering to root layout so it shows regardless of route group.
**Warning signs:** Audio audible but no visible controls on Tech pages; cart count not visible on Tech.

### Pitfall 4: `<Link>` with `target="_blank"` used for cross-link tiles
**What goes wrong:** Tile opens in new tab, breaking state continuity, breaking the user's seamless-nav intent (D-13 explicitly rejected new-tab).
**Why it happens:** Copy-pasting from older sister-site patterns (Verge → Polygon use separate tabs historically).
**How to avoid:** Cross-link `<Link>` MUST omit `target` and `rel="noopener"`. Use default client-side navigation.
**Warning signs:** `target="_blank"` or `rel="noopener"` in `TechLogoTile` or `StudiosLogoTile`.

### Pitfall 5: `use client` directives in the wrong files
**What goes wrong:** Accidentally marking `(tech)/layout.tsx` as `"use client"` turns the entire Tech subtree into client-rendered code, losing server rendering of pages, metadata, and data fetching.
**Why it happens:** Copy-pasting from interactive components.
**How to avoid:** Keep layouts as server components by default. Only mark as client if you're using hooks (usePathname etc.) directly — and even then, extract the client parts to a sub-component. Reference: current `(public)/layout.tsx` is a server component; `TileNav` is the client piece.
**Warning signs:** `"use client"` at the top of a `layout.tsx`; loss of metadata/SEO on Tech pages.

### Pitfall 6: Conflicting route at `/`
**What goes wrong:** Accidentally putting `(tech)/page.tsx` directly in the group (not in `(tech)/tech/page.tsx`) — resolves to `/` and conflicts with `(public)/page.tsx`. Next.js build errors.
**Why it happens:** Confusing route-group syntax (parens are stripped from URL, but folder name inside the group IS part of the URL).
**How to avoid:** All Tech pages live under `(tech)/tech/*`. Verify at build time that `/tech` resolves uniquely.
**Warning signs:** `Error: You cannot have two parallel pages that resolve to the same path`.

### Pitfall 7: Widget component coupling to Studios state
**What goes wrong:** Reusing `WidgetNowPlaying` in Tech sidebar (violates D-17 — no Now Playing widget in Tech sidebar) OR creating Tech widgets that reach into Studios-specific state (e.g., Tech's "Latest Review" accidentally imports Studios' blog query).
**Why it happens:** Copy-paste momentum from building Studios sidebar.
**How to avoid:** Tech sidebar widgets are NEW components (`WidgetLatestReview`, `WidgetFeaturedProduct`, `WidgetSocialTech`) even if they visually mimic Studios widgets. Reserve imports.
**Warning signs:** Tech layout imports `WidgetNowPlaying` or Studios-specific queries.

### Pitfall 8: Forgetting GlitchTechPromoSection on Studios homepage
**What goes wrong:** Tech launches at `/tech` but nobody can find it because there's no entry point on the Studios homepage (D-14 violated).
**Why it happens:** D-14 is an addition to `(public)/page.tsx`, not a Tech-group task — easy to miss if tasks are Tech-only.
**How to avoid:** Planner must include a task that edits `src/app/(public)/page.tsx` to insert `<GlitchTechPromoSection />` per UI-SPEC "Option A" placement (between Portfolio and Testimonials).
**Warning signs:** Studios homepage renders unchanged after 07.4; no "Glitch Tech" promo visible between portfolio and testimonials.

### Pitfall 9: Tailwind v4 `@theme` nested in selector
**What goes wrong:** Developer tries to write `[data-brand="tech"] { @theme { --color-accent: green; } }` — Tailwind v4 throws because `@theme` must be top-level.
**Why it happens:** Tailwind v3 `.tech { @apply ... }` muscle memory.
**How to avoid:** For route-scoped tokens, use plain CSS variable overrides inside attribute selectors. `@theme` blocks only at file top-level.
**Warning signs:** Build error referencing `@theme` must be top-level, or Tech accent fails silently to apply.

### Pitfall 10: Audio autoplay across brand boundaries
**What goes wrong:** User navigates to `/tech`, mobile browser autoplay-pause logic kicks in (because route changed) and audio silently pauses despite our intent for it to keep playing.
**Why it happens:** Some mobile browsers pause `<audio>` on any significant DOM change.
**How to avoid:** Verify with Playwright mobile emulation that audio persists across cross-brand nav. If pausing occurs, the AudioPlayerProvider must detect and resume (check for existing resume logic). Test on real iOS Safari.
**Warning signs:** Audio pauses when crossing brands on mobile; play button shows but audio doesn't resume.
