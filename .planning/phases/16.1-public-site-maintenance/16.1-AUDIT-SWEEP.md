## Findings

### F-1 — `/tech/reviews/[slug]` returns 404 for the only linked slug

**Severity:** blocker (user-visible: clicking "Latest Review" in the tech sidebar used to 404)
**Where:** `src/components/tiles/widget-latest-review.tsx` hardcodes a link to `/tech/reviews/macbook-pro-m4`, but no seeded review exists with that slug. Per memory `project_placeholder_first_build` the plan is to seed ONE laptop review (MBP 16 M5 Max) as the first real content — that seed has not landed yet.

**Triage:** Fix-in-phase (partial) + Promote-to-backlog
- **Fixed in-phase (commit log attached):**
  - Pointed `widget-latest-review.tsx` href at the reviews index (`/tech/reviews`) so the click target always resolves. Widget still shows placeholder copy/rating.
  - Made `getPublishedReviewBySlug` resilient to partial DB state by switching `innerJoin` to `leftJoin` for `techCategories` + `user` (reviewer). A future review with a missing category/reviewer will now render instead of 404.
  - Updated `PublicReviewDetail` type to allow nullable category fields + updated the detail page to skip `getCategoryBreadcrumb` and `getRelatedReviews` when category is null.
- **Promoted to backlog:** seeding the MacBook Pro M5 Max placeholder review itself is NOT a code change — it's a data task that lives in `src/db/seed-*.ts`. Captured as `999.x` backlog: "Seed one representative tech review + product + gallery so `/tech/reviews/[slug]` renders against live data during dev."

The sweep still records this route as 404 since no seeded slug exists yet — that's honest. The user-visible "dead click" path has been closed.

### F-2 — `/tech/categories/laptops` crashed with HTTP 500 on `next/image` host not configured

**Severity:** blocker (original); resolved
**Where:** `next.config.ts` `images.remotePatterns` didn't include `placehold.co`. The tech catalog seed references placeholder URLs from this host (memory `project_placeholder_first_build`).

**Triage:** Fix-in-phase. Added `placehold.co` to `remotePatterns` with an inline comment tying it to the placeholder-first memory rule. The `_next/image` proxy still returns HTTP 400 for the URL-encoded query string `?text=MBP+16+M5` (visible in console errors at info severity, 3 findings), but the page itself renders cleanly — `next/image` falls back to the unoptimized URL. Low priority; fix when placeholder imagery gets replaced with real uploads.

### F-3 — Lone-nav-tile detector (regression guard, false positives)

**Severity:** info (tuning)
**Where:** The sweep spec runs a heuristic `grep for col-span-1 in rendered nav tile classes — if odd, flag`. This was a proactive addition from the Blog-tile bug user feedback.

**Findings during this run:** 0 (after restarting pm2 mid-sweep to flush a stale HMR layer that had left Blog/About mismatched). A prior sweep pass flagged 12 tech routes with "odd col-span-1 count" purely due to dev-server state drift.

**Triage:** Accepted-as-is. The detector works as intended — during the session it correctly flagged a stale render. For downstream CI use, run the sweep AFTER a fresh dev-server start or against a production build so HMR state can't interfere.

### F-4 — Hydration-mismatch warnings on 8 tech routes (first sweep only)

**Severity:** info (hidden by "warning: this won't be patched up" console log from React)
**Where:** Multiple tech pages logged hydration mismatches in the prior (stale-HMR) sweep. After pm2 restart, these disappeared.

**Triage:** Accepted-as-is for now. If they reappear in CI, Promote-to-backlog for proper investigation (common causes: `typeof window !== 'undefined'` in a Server Component branch, or `Date.now()` / `Math.random()` in SSR-returned JSX).

### F-5 — `_next/image` 400 Bad Request on placeholder URLs (info, 6 findings)

**Severity:** info
**Where:** `placehold.co/1200x800?text=MBP+16+M5` gets a 400 from Next.js image optimizer when the query string is URL-encoded in certain contexts.

**Triage:** Accepted-as-is. Visible in dev console only; page renders. Clears when placeholder images are replaced with real Uploadthing/R2 URLs.

## Method notes for downstream runs

- Dev server must be freshly started (pm2 restart glitch_studios) before a sweep — stale Turbopack HMR state has caused false-positive "lone nav tile" findings mid-session. Prod builds or CI are not affected.
- The spec uses soft-fail logging instead of `expect(...).toBeLessThan(400)` for HTTP status — a single 404 in a serial run would otherwise abort all downstream assertions and leave the findings report incomplete.
- Dynamic routes in `ROUTES` use one representative seeded slug. If the underlying seed changes, update the inventory file — both Plans 04 and 05 consume it.
