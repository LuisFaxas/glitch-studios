# Phase 26 — UI Design Contract

> Visual and interaction contract for the brand-aware auth redesign (login, register, forgot-password, reset-password, verify-email). Generated by gsd-ui-researcher, consumed by gsd-planner + gsd-executor, verified by gsd-ui-checker.

Scope covered: five auth surfaces themed by host (`glitchstudios.io` vs `glitchtech.io`), split register flow (customer wizard vs artist request), social login (Google + Meta + GitHub), enumeration-safe branded error copy.

## Spacing Scale

All values are multiples of 4. This phase uses a restricted subset of the project's existing scale:

| Token | Value | Usage in auth surfaces |
|-------|-------|------------------------|
| xs | 4px | Inline icon gaps (e.g., eye-toggle to input edge), field-error spacing |
| sm | 8px | Label-to-input gap, intra-field vertical rhythm |
| md | 16px | Between-field gap, card internal padding on mobile, social button internal padding |
| lg | 24px | Between form sections, `--pad-page` (project token), brand-side panel internal padding |
| xl | 32px | Between header block and form, between form and footer link row, wizard step gap |
| 2xl | 48px | Top/bottom padding of AuthShell on desktop, gap between form column and brand-side panel at ≥lg |
| 3xl | 64px | Brand-side panel outer padding at ≥xl viewports only |

**Layout grid (desktop ≥1024px):** 50/50 split. Form column: min-width 420px, max-width 480px, centered within its column. Brand-side panel: fluid, max-height 100vh, sticky inside viewport.

**Layout grid (mobile <1024px):** Single column. Brand-side panel is hidden; a compact brand header (logo + one-line tagline) renders above the form at 16px top padding. Form column expands to full width with 24px horizontal padding.

**Touch target exception:** OAuth buttons and the primary CTA are `size="lg"` (44px min-height) — exception to the 40px default Button size because auth is a high-risk, thumb-driven interaction on mobile.

**No exceptions beyond the above.** No hard-coded pixel values outside the scale in any auth-phase component.

## Typography

Declared roles (4 sizes, 2 weights — at project cap):

| Role | Size | Weight | Line Height | Font | Notes |
|------|------|--------|-------------|------|-------|
| Display (brand-side panel tagline) | 28px | 600 | 1.2 | JetBrains Mono | Uppercase, letter-spacing 0.05em. Used ONLY on desktop brand-side panel. Mobile brand header uses Heading size instead. |
| Heading (H1 page title) | 20px | 600 | 1.2 | JetBrains Mono | Uppercase, letter-spacing 0.05em. Wrapped in `<HoverGlitchHeading>` — glitch triggers on hover only, never auto-runs. |
| Body (form copy, footer links, descriptions) | 16px | 400 | 1.5 | Inter | Default paragraph + input text + button labels. |
| Label (field labels, micro-labels, badge text) | 12px | 600 | 1.2 | JetBrains Mono | Uppercase, letter-spacing 0.08em. Used for field labels, step indicators ("STEP 2 OF 3"), helper-text headers. |

**Weights in use:** exactly 2 — 400 (Inter body) and 600 (Mono labels/headings). No 500, 700, 800 anywhere in this phase.

**Sizes in use:** exactly 4 — 12, 16, 20, 28. No 14, 18, 24 etc.

**Error/helper text:** 12px Inter 400 at line-height 1.5. Red-on-dark uses `--destructive` token.

**Button label:** 16px Inter 400. Uppercase transformation is NOT applied to buttons in auth (buttons elsewhere on site follow mono uppercase; here, natural-case labels feel more product-grade and less admin-panel).

## Color

Site is monochrome-dark. Phase 26 does NOT introduce new chromatic tokens — it introduces brand-side panel ART that is already monochrome + uses the existing tokens. The `[data-brand="tech"]` hook remains empty of accent CSS (locked to Phase 07.5/07.6 activation, deferred in v3.0).

| Role | Value (token) | Hex fallback | Usage in auth surfaces |
|------|--------------|--------------|-----------------------|
| Dominant (60%) | `var(--background)` / `--color-bg-dominant` | `#000000` | Full-page background, form column background |
| Secondary (30%) | `var(--card)` / `--color-bg-secondary` | `#111111` | Brand-side panel background (subtle lift), AuthFormCard surface, Input field background |
| Accent (10%) | `var(--foreground)` / `--color-text-fg` | `#f5f5f0` (off-white) | Reserved-for list below |
| Destructive | `var(--destructive)` | `#dc2626` | Form validation errors, expired-token Alert, destructive confirmation |
| Border | `var(--border)` / `--color-border` | `#222222` | Input borders, AuthFormCard border, Separator |
| Muted foreground | `var(--muted-foreground)` / `--color-text-muted` | `#888888` | Helper text, "don't have an account?" prose, placeholder copy |

**Accent (off-white `#f5f5f0`) is reserved for — explicit list, never "all interactive elements":**
1. Primary submit CTA button background (`Sign In`, `Create Account`, `Send Reset Link`, `Reset Password`, `Submit Request`)
2. Text color on the primary CTA (inverted — black on off-white)
3. Hovered/focused input border (replaces the dim `#222` border on focus)
4. Focus ring on all interactive elements (`--ring` token, 2px offset)
5. Selected wizard step indicator in `<WizardProgress>` (completed + current steps use off-white; future steps use `#555555`)
6. Link underline + link text hover (matches existing footer-link pattern)
7. Brand-side panel tagline text (display size)
8. The GlitchLogo wordmark (already off-white)

**Accent is NEVER used for:** social login button backgrounds (those are `#111111` surface with `#222` border — same visual weight for all 3 providers to avoid brand favoritism); secondary CTAs (which render as ghost variant with border only); decorative dividers (those use `--border`).

**Studios vs Tech brand differentiation — since both share the monochrome palette today, the differentiation carries via:**
- **Brand-side panel artwork** (Studios: vinyl/waveform motif; Tech: circuit/benchmark-grid motif — both monochrome SVG, static, no auto-animation)
- **Wordmark** (Studios: "GLITCH STUDIOS" / Tech: "GLITCHTECH")
- **Tagline copy** (see Copywriting Contract)
- **Accent activation hook** (`[data-brand="tech"]` may gain its own accent later; for now both resolve to `#f5f5f0` — this is a known limitation, NOT a bug)

**Name spelling is load-bearing:** the tech sub-brand is **GlitchTech** (matches `glitchtech.io`). Planning docs that say "GlitchTek" are stale — do NOT propagate that spelling into any auth copy, button label, logo, tagline, or page title. All auth surfaces MUST render "GlitchTech" when brand is tech.

## Interaction & State Contract

Non-copy interaction rules the executor must honor:

| Rule | Behavior |
|------|---------|
| All H1s | Wrapped in `<HoverGlitchHeading>`. Glitch only on hover; never auto-animates. Reduced-motion honored via existing `@media (prefers-reduced-motion)` block. |
| Social login row position | ABOVE the email/password fields on `/login` and `/register?role=customer`. Separator "or continue with email" follows. This matches modern consumer auth convention. |
| Password field | Show/hide eye toggle on every password input (login, register step 1, reset-password). Toggle is an icon-button INSIDE the input group, right-aligned, using existing `@/components/ui/input-group`. |
| Password strength meter | Register-only (not login, not reset). Updates live as user types. Zero network calls. Strength tiers: Weak / Fair / Strong (three-stop meter, not five). |
| Wizard navigation | Step 1 → Step 2 → Step 3. "Back" button on steps 2-3 (ghost variant, left-aligned). Step indicator `<WizardProgress>` sticky at top of form column. URL query param `?step=1\|2\|3` for deep-linking + browser back button support. |
| Register role selection | `/register` without a role param shows two CTA tiles side-by-side on desktop, stacked on mobile. Clicking a tile navigates to `/register?role=customer` or `/register?role=artist`. |
| Artist request form | Single page, not a wizard. On submit, form replaces itself with the on-page confirmation copy (no redirect). |
| Forgot-password submit | Always shows the enumeration-safe success message regardless of whether the email matched an account. No distinct error state for "email not found." |
| Reset-password expired-token landing | Route renders an Alert with expired copy + a secondary CTA "Send a new link" that routes back to `/forgot-password`. |
| Verify-email states | One route, three states resolved server-side by token validity: pending (no token in URL), success (valid token consumed), expired (invalid/expired token). All three states share the same AuthShell — only the heading + body + CTA swap. |
| Social login buttons loading state | Clicking one disables all three + the email/password form. Button shows inline spinner at its icon position. Cancel at provider → restore enabled state + show the "Social sign-in didn't complete" toast via sonner. |
| Inline field validation | Blur-validation only for email format; submit-validation for everything else. No real-time password-match checking (the register wizard step 1 omits a "confirm password" field entirely — use the show/hide toggle to let the user visually verify). |
| Keyboard affordance | Enter in any form field submits that form. Tab order: social buttons → email → password → show/hide → submit → secondary link. Skip-to-content not required (forms are the main content). |
| Focus management | On route change, focus moves to the H1 so screen readers announce the page title. On wizard step change, focus moves to the step heading. |

### Breakpoints

| Breakpoint | Behavior |
|-----------|---------|
| <640px (mobile) | Single column. Brand header = GlitchLogo (size sm) + one-line tagline, 16px top, 24px horizontal. Form fills width. Social buttons stack vertically (3 rows). |
| 640–1023px (tablet) | Single column, slightly wider max (520px). Brand header same as mobile. Social buttons render in a 3-up row if viewport ≥480px, stacked otherwise. |
| ≥1024px (desktop) | Two-column 50/50 split. Brand-side panel (left) + form column (right). Brand-side panel: full-height, sticky, 48px padding, tagline + wordmark + decorative monochrome art. Form column: centered at max-width 480px with 48px vertical padding. |

No desktop viewport override larger than 1024px — the layout is stable from 1024 up. Form column max-width caps at 480px so text measure stays readable on ultrawide displays.
