# Pitfalls Research

**Domain:** Music/video production studio website with beat e-commerce, booking, and media showcase
**Researched:** 2026-03-25
**Confidence:** HIGH

## Critical Pitfalls

### Pitfall 1: Serving Audio Files Through Vercel Serverless Functions

**What goes wrong:**
Beat files (WAV/MP3) get routed through Vercel serverless functions which have a 4.5 MB request body limit and are not designed to serve media. Audio previews fail, downloads break, and bandwidth costs explode because Vercel charges $0.15/GB overage and $20 per 100 GB on the Hobby tier.

**Why it happens:**
Developers treat Next.js API routes as a general-purpose file server. It works in local dev but breaks in production on Vercel because serverless functions are meant to be a lightweight API layer, not a media server.

**How to avoid:**
Store all audio files (previews, tagged/watermarked versions, purchased downloads) in external object storage (Vercel Blob, AWS S3, or Cloudflare R2). Serve previews via CDN URLs. Deliver purchased files via time-limited pre-signed URLs generated by an API route -- the API route returns the URL, not the file itself.

**Warning signs:**
- API routes that read files from disk and pipe them to the response
- Audio files stored in the `public/` directory (works but doesn't scale and can't be access-controlled)
- Download endpoints returning binary data instead of redirect URLs
- Vercel bandwidth bills climbing unexpectedly

**Phase to address:**
Phase 1 (Foundation) -- storage architecture must be decided before any audio upload or playback features are built.

### Pitfall 2: No Audio Watermarking on Beat Previews

**What goes wrong:**
Untagged/unwatermarked beat previews are available in the browser. Users inspect the network tab, grab the direct MP3 URL, and download the full-quality preview for free. The entire beat catalog becomes free to steal.

**Why it happens:**
Developers build the audio player first and plan to "add watermarks later." Later never comes, or it gets deprioritized. Meanwhile every preview URL is a direct link to a clean audio file.

**How to avoid:**
Never serve unwatermarked audio to the browser. Pre-generate tagged versions of every beat (with a repeating voice tag or audio watermark baked into the file). The browser only ever receives the tagged version. Clean files are delivered only after purchase via pre-signed download URLs. Process watermarking at upload time, not at playback time -- runtime watermarking is computationally expensive and fragile.

**Warning signs:**
- Audio player sources pointing to clean/untagged files
- No separate "preview" and "download" file variants in the database schema
- Ability to right-click and save a playable audio file from the page

**Phase to address:**
Phase 1 (Foundation) -- the data model must distinguish between tagged preview files and clean purchase files from the start. The upload pipeline must generate both variants.

### Pitfall 3: Beat Licensing Model Not Enforced in Code

**What goes wrong:**
The site sells beats but the licensing terms (non-exclusive lease vs. exclusive rights) are just text on a page. There is no enforcement: exclusive beats remain available for purchase after being sold exclusively, stream/download limits are never tracked, and multiple "exclusive" licenses get sold for the same beat.

**Why it happens:**
Licensing is treated as a legal/copy problem rather than a data model problem. The developer builds a generic e-commerce checkout and bolts licensing terms on as descriptive text.

**How to avoid:**
Model license types as first-class entities in the database. Each beat has a licensing status (available, leased, exclusively sold). When an exclusive license is purchased, the beat status changes and it is automatically removed from the catalog or marked as sold. Non-exclusive leases track the number of active leases. The checkout flow must validate license availability before processing payment -- not after.

**Warning signs:**
- License types stored as plain strings rather than structured data with rules
- No status field on beats that changes after exclusive purchase
- No pre-checkout validation of beat availability
- Exclusive beats still showing "Buy" buttons after sale

**Phase to address:**
Phase 2 (E-commerce) -- must be built into the beat catalog and checkout system from the start, not retrofitted.

### Pitfall 4: iOS Safari Audio Playback Broken on Launch

**What goes wrong:**
The audio player works perfectly in Chrome desktop during development. On iOS Safari (the primary mobile browser for music consumers), audio fails to play, throws silent errors, or requires awkward double-taps. The beat catalog -- the core revenue feature -- is broken for a large segment of users.

**Why it happens:**
iOS Safari enforces strict autoplay restrictions: audio cannot play without a direct user interaction event (tap/click). Many audio player libraries abstract this away in ways that work on desktop but fail on mobile. Additionally, iOS Safari has known bugs with changing audio sources on the same `<audio>` element (documented in iOS 17+).

**How to avoid:**
- Never attempt autoplay. Every play action must originate from a user tap/click event handler.
- Test on a real iOS device from day one, not just browser DevTools mobile simulation.
- When switching tracks, create a new Audio context or properly reset the source before loading.
- Use the `playsinline` attribute on any media elements.
- Implement a "tap to enable audio" interaction pattern if building a persistent player across page navigation.

**Warning signs:**
- Audio player tested only in Chrome/Firefox desktop
- No real iOS device in the testing workflow
- Using `audio.play()` outside of a direct user gesture callback
- Console errors about "user gesture" or "NotAllowedError" being swallowed silently

**Phase to address:**
Phase 2 (Beat Catalog / Audio Player) -- must be validated on iOS Safari immediately when the player is built, not deferred to QA.

### Pitfall 5: Dual Stripe + PayPal Checkout Creates Reconciliation Nightmare

**What goes wrong:**
Two separate payment systems means two dashboards, two refund workflows, two chargeback processes, and no unified view of revenue. Order records in the database may reference a Stripe payment ID or a PayPal transaction ID with no consistent interface. Refunds issued on one platform don't reflect in the other. Financial reporting requires manual reconciliation.

**Why it happens:**
Stripe and PayPal are integrated as two completely independent payment flows with separate API integrations. The developer builds "Stripe checkout" and "PayPal checkout" as parallel code paths with no abstraction layer.

**How to avoid:**
Use Stripe as the primary payment orchestrator. Stripe supports PayPal as a payment method through their Payment Element -- this routes PayPal payments through Stripe's system, giving you a single dashboard, single refund API, and unified webhook handling. If direct PayPal integration is required instead, build a payment abstraction layer that normalizes both into a common order/transaction model in your database. Every order record stores: amount, status, provider, provider_transaction_id, and refund_status regardless of which provider processed it.

**Warning signs:**
- Two completely separate checkout code paths with no shared interface
- Order records that only store Stripe-specific or PayPal-specific fields
- No unified webhook handler -- separate endpoints for each provider
- Manual work needed to answer "how much revenue did we make this month?"

**Phase to address:**
Phase 2 (E-commerce / Payments) -- payment architecture must be unified from the first transaction.

### Pitfall 6: Calendar Booking Without Timezone or Double-Booking Protection

**What goes wrong:**
Clients book sessions and receive confirmation emails showing the wrong time because timezone conversion was not handled. Or two clients book the same studio slot because availability was checked at read time but not locked at write time, creating a race condition.

**Why it happens:**
Developers store booking times as local time strings or naive Date objects instead of UTC timestamps with timezone metadata. Availability checks use a simple "is this slot taken?" query without database-level locking, so two simultaneous requests can both see the slot as available.

**How to avoid:**
- Store all times in UTC in the database. Store the client's timezone separately. Convert to local time only for display.
- Use database-level constraints (unique constraint on resource + timeslot) or optimistic locking (version column) to prevent double bookings at the database level, not the application level.
- Display timezone clearly on the booking page and let clients confirm their timezone before booking.
- Handle Daylight Saving Time transitions explicitly -- a 2:30 AM booking during a DST change can be ambiguous.

**Warning signs:**
- Dates stored as strings like "2026-03-25 2:00 PM" without timezone info
- No unique constraint or lock on the booking timeslot in the database
- Booking confirmation emails showing different times than what the client selected
- No timezone selector or auto-detection on the booking page

**Phase to address:**
Phase 3 (Booking System) -- timezone handling and concurrency protection must be in the initial implementation, not patched later.

### Pitfall 7: Admin Dashboard Too Complex for Non-Technical Owner

**What goes wrong:**
The admin dashboard is built with developer ergonomics (raw data tables, JSON editors, technical jargon) instead of being designed for the actual user -- a music producer who wants to update a price, check today's bookings, and see how many beats sold this week. The owner stops using it and asks for manual changes, defeating the purpose.

**Why it happens:**
Developers build admin panels for themselves. CRUD interfaces for every database table feel "complete" but are unusable for non-technical users. The admin becomes a database browser instead of a business tool.

**How to avoid:**
- Design the admin around tasks, not tables: "Add a beat," "View today's bookings," "Check this week's revenue" -- not "Beats table," "Bookings table," "Transactions table."
- Limit options to what the owner actually needs. Hide advanced settings behind progressive disclosure.
- Use familiar patterns: drag-and-drop for ordering, inline editing for text, visual calendars for bookings.
- Test with the actual owner early -- show mockups before building.

**Warning signs:**
- Admin pages that look like phpMyAdmin or a Supabase dashboard
- More than 3 clicks to do any common task
- Technical labels (IDs, foreign keys, JSON blobs) visible in the interface
- Owner asking "how do I..." for basic operations

**Phase to address:**
Phase 4 (Admin Dashboard) -- but wireframe/mockup validation should happen in Phase 1 to ensure the design direction is right.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Audio files in `public/` folder | Simple, works immediately | No access control, no CDN optimization, bloats deployment, files are publicly accessible to anyone | Never for purchasable content. OK for site sound effects only. |
| Single email provider for transactional + marketing | One integration to maintain | Marketing emails tank transactional deliverability (booking confirmations land in spam) | Never -- use separate sending domains or providers for transactional vs. marketing from day one |
| Storing beat metadata in filesystem/JSON | No database needed initially | Cannot query, filter, search, or update efficiently. Migration pain when database is added later | Never -- use a database from Phase 1 |
| Skipping audio waveform pre-generation | Faster upload pipeline | Waveform visualization calculated client-side on every page load, causing jank and high CPU usage on mobile | MVP only -- generate waveforms at upload time before public launch |
| No rate limiting on booking/contact forms | Simpler implementation | Spam bots fill booking calendar with fake reservations or flood contact inbox | Never -- add basic rate limiting from the start |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Stripe | Trusting client-side price. Sending the price from the frontend and charging it server-side. | Always look up the price server-side from the database. The client sends a beat ID and license type; the server determines the price. |
| Stripe | Not handling webhook failures. Assuming the checkout session redirect means payment succeeded. | Use Stripe webhooks as the source of truth for payment status. Implement idempotent webhook handlers with retry tolerance. |
| PayPal (via Stripe) | Assuming PayPal through Stripe works identically to card payments. | PayPal via Stripe has different settlement timing, different dispute flows, and may not support all Stripe features. Test the full lifecycle including refunds. |
| Vercel | Expecting persistent file storage on the server. | Vercel deployments are immutable. Files written at runtime disappear on next deploy. Use external storage for all user-uploaded content. |
| Email (Resend/SendGrid) | No retry logic for transactional emails. If the API call fails, the booking confirmation or receipt is lost forever. | Wrap email sends in try/catch with exponential backoff retry. Log failures. Consider a queue for critical emails. |
| Video Embeds (YouTube/Vimeo) | Embedding 10+ video iframes on a portfolio page. | Lazy-load video embeds. Show thumbnail placeholders and load the iframe only on click. Each iframe is a full browser context that destroys scroll performance. |
| Calendar/Booking | Not accounting for the studio's own schedule (holidays, maintenance, personal time). | Build a "blocked time" or "availability override" system in admin, not just regular hours. |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Loading full audio files for preview | Long initial load, high bandwidth, mobile users on cellular data bail | Stream audio with range requests. Serve compressed MP3 previews (128kbps), not WAV files. Keep previews under 30 seconds. | Immediately on mobile / slow connections |
| No image optimization for portfolio | Portfolio page loads 20+ full-resolution images, 50MB+ page weight | Use Next.js `<Image>` component with automatic WebP conversion and responsive sizing. Lazy-load below-fold images. | At 5+ portfolio items |
| Client-side beat catalog filtering | JavaScript filters the entire beat array on every keystroke | Server-side filtering with database queries. Paginate results (20-30 per page). Debounce search input. | At 100+ beats |
| Unoptimized video thumbnails | Custom video thumbnails served as full-size PNGs | Generate multiple thumbnail sizes at upload. Serve appropriate size per viewport. Use WebP/AVIF. | At 10+ videos on a page |
| No CDN caching for static assets | Every beat preview request hits origin storage | Set proper Cache-Control headers. Use CDN (Vercel Edge or Cloudflare) for all static media. Bust cache on content update only. | At 50+ concurrent users |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Download URLs that never expire | Anyone who gets a download link (shared on forums, social media) can download the beat file forever for free | Use pre-signed URLs with short expiration (15-60 minutes). Log download attempts. Tie downloads to authenticated sessions. |
| No authentication on admin API routes | Attackers can modify prices, delete beats, access client data, send emails from the admin system | Protect every admin API route with authentication middleware. Use role-based access (admin vs. client). Never rely on "hiding" the admin URL. |
| Client-side license validation only | A modified frontend request can bypass license checks, allowing purchase of sold-out exclusive beats or manipulation of pricing | All license availability checks and price lookups happen server-side. Client-side validation is UX convenience only. |
| Storing PayPal/Stripe secrets in frontend code | API keys exposed in browser JavaScript, enabling unauthorized charges or refunds | All payment API calls go through server-side API routes. Only publishable/client keys go to the frontend. Use environment variables on Vercel. |
| Contact form without CAPTCHA or honeypot | Spam bots flood inbox, booking calendar filled with fake entries | Implement at minimum a honeypot field. For production, add Turnstile (Cloudflare) or reCAPTCHA. Rate limit submissions per IP. |
| Email addresses in client-side newsletter signup without double opt-in | Fake signups, spam complaints, domain reputation destroyed, email provider account suspended | Always use double opt-in (confirmation email) for newsletter subscriptions. Required by GDPR and CAN-SPAM. |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Audio player stops when navigating pages | User clicks a beat, starts listening, clicks to another page to browse -- music stops. They leave. | Implement a persistent audio player that lives outside the page router (layout-level component). Player state survives navigation. This is the #1 UX expectation from BeatStars/Airbit users. |
| No visual feedback during checkout | User clicks "Buy," nothing happens for 3 seconds while Stripe session creates. They click again, get duplicate charges. | Show loading state immediately on click. Disable the button. Implement idempotent checkout session creation. |
| Beat catalog with no filtering or sorting | User sees 200 beats in a flat list. Cannot find beats by genre, BPM, key, mood. They leave. | Provide filters for genre, BPM range, key, mood/tag. Allow sorting by newest, popular, price. These are table-stakes features for beat platforms. |
| Booking form asks too many questions upfront | Client wants to book a studio session but faces a 15-field form before seeing available times. They abandon. | Show available times first. Collect minimum info (name, email, service type). Gather details after booking confirmation. |
| License terms buried or confusing | Artist buys a beat without understanding usage limits. Later discovers they cannot monetize on Spotify with a basic lease. Disputes and refund requests follow. | Show license comparison table prominently on every beat page. Use plain language (not legal jargon). Highlight key limits: streams, sales, video rights, exclusivity. |
| No mobile-optimized audio player | Tiny play buttons, no swipe gestures, waveform too small to scrub on touch screens | Design the player mobile-first. Large touch targets (44px minimum). Simple controls. Test on actual phones. |

## "Looks Done But Isn't" Checklist

- [ ] **Beat purchase flow:** Often missing post-purchase delivery -- verify the buyer actually receives a download link via email AND in their account dashboard
- [ ] **Audio player:** Often missing error states -- verify behavior when audio file fails to load (404, network error, unsupported format)
- [ ] **Booking system:** Often missing cancellation/rescheduling flow -- verify clients can cancel or reschedule, and the slot reopens in the calendar
- [ ] **Email system:** Often missing bounce/failure handling -- verify what happens when a transactional email fails to send (is it retried? logged? lost?)
- [ ] **Admin dashboard:** Often missing bulk operations -- verify admin can manage 100+ beats without clicking into each one individually
- [ ] **Client accounts:** Often missing password reset flow -- verify the full reset cycle works end-to-end including the email link
- [ ] **Video portfolio:** Often missing embed consent/privacy -- verify embedded YouTube/Vimeo players respect cookie consent and don't load tracking scripts before consent
- [ ] **Search/filtering:** Often missing empty states -- verify what the user sees when no beats match their filters (helpful message vs. blank page)
- [ ] **Checkout:** Often missing failed payment recovery -- verify the user can retry after a declined card without starting over
- [ ] **Licensing:** Often missing license PDF generation -- verify that a purchased license generates a downloadable contract/receipt with the buyer's details and terms

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Audio files served through serverless functions | MEDIUM | Migrate files to object storage (S3/R2/Vercel Blob). Update all file references. Add pre-signed URL generation. 1-2 days of work. |
| No audio watermarking on previews | HIGH | Must regenerate tagged versions of every beat. Update upload pipeline. Replace all preview URLs. If beats were already stolen, damage is done. |
| Licensing not enforced in code | HIGH | Requires data model migration. Must audit all past sales for licensing conflicts. Potential legal exposure if exclusive beats were sold multiple times. |
| iOS audio player broken | LOW | Fix event handling to require user gesture. Test matrix across iOS versions. Usually 1-2 day fix once identified. |
| Dual payment reconciliation mess | MEDIUM | Build abstraction layer retroactively. Migrate historical transaction records to unified schema. 2-3 days. |
| Timezone booking errors | MEDIUM | Migrate stored times to UTC. Recalculate all existing bookings. Notify affected clients. 1-2 days plus client communication. |
| Admin dashboard unusable | HIGH | Essentially a redesign. Task-oriented UX requires rethinking information architecture, not just restyling. 1-2 weeks. |
